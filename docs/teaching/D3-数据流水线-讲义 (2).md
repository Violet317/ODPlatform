# 阶段 0: 准备工作

本阶段不写业务代码, 只立两条**项目级公约**, 它们贯穿后面所有阶段:

1. **数据集目录约定**——`data/raw/<dataset_name>/{images, annotations}/`
2. **paths.py 是路径的唯一真相来源**——任何模块要路径都从这里 import
3. 新增 **`common/constants.py`**——跨模块共享的非路径常量(图像扩展名、格式名、任务名、浮点容差...)

## 0.1 数据集目录约定

**问题**: 一个新同学拿到 ODPlatform 仓库, 想训练一个安全帽检测模型——他要怎么放数据集?

可能的回答:
- `data/raw/safety_helmet.zip`(整个压缩包)
- `data/raw/images/` + `data/raw/annotations/`(平铺)
- `data/raw/safety_helmet/images/` + `data/raw/safety_helmet/annotations/`(按数据集名分目录)

**我们选第三种**——每个数据集占一个文件夹, 内部固定 `images/` + `annotations/`:

```
data/raw/
├── safety_helmet/
│   ├── images/         <- 原始图像 (jpg/png/...)
│   └── annotations/    <- 原始标注 (xml/json/txt, 按格式而定)
├── rsod/
│   ├── images/
│   └── annotations/
└── coco_demo/
    ├── images/
    └── annotations/
```

**三个理由**:

1. **多数据集天然隔离**——今天跑 safety_helmet, 明天跑 rsod, 两个数据集不会互相污染
2. **数据集名 = 文件夹名 = 唯一标识**——CLI 只需要 `--dataset safety_helmet` 一个参数就能定位
3. **生成的 yaml 可以叫 `safety_helmet.yaml`**——多数据集互不覆盖

> **金句**: **"约定的本质是【消除歧义】。一个项目里能用'数据集名'定位一切, 胜过靠多个参数拼凑路径。"**

## 0.2 `data/raw/` 的洁癖原则(铁律)

> **`data/raw/` 下只允许出现"数据集文件夹"这一种东西。**

具体含义:

- ❌ `data/raw/yolo_staged_labels/`(转换的中间产物)不允许
- ❌ `data/raw/temp_xxx/`(临时文件)不允许
- ❌ `data/raw/converted_helmet/`(转换的输出副本)不允许
- ✅ 只能有 `data/raw/<数据集名>/{images, annotations}/`

这条铁律会**逼出后面 `tempfile` 中转**的设计——你会看到。

## 0.3 paths.py 是路径的【唯一真相来源】

D2 已经在 `apps/platform/src/odp_platform/common/paths.py` 里定义了一堆路径常量:

```python
# paths.py (D2 留下的, 截取关键部分)
ROOT_DIR: Path = ...                                  # 仓库根
DATA_DIR: Path = ROOT_DIR / "data"
RAW_DATA_DIR: Path = DATA_DIR / "raw"                 # 数据集存放根
LOGGING_DIR: Path = ...                               # 日志根
CONFIGS_DIR: Path = APP_DIR / "configs"               # 端私有配置

TRAIN_DIR: Path = DATA_DIR / "train"
VAL_DIR:   Path = DATA_DIR / "val"
TEST_DIR:  Path = DATA_DIR / "test"

TRAIN_IMAGES_DIR: Path = TRAIN_DIR / "images"
TRAIN_LABELS_DIR: Path = TRAIN_DIR / "labels"
VAL_IMAGES_DIR:   Path = VAL_DIR   / "images"
VAL_LABELS_DIR:   Path = VAL_DIR   / "labels"
TEST_IMAGES_DIR:  Path = TEST_DIR  / "images"
TEST_LABELS_DIR:  Path = TEST_DIR  / "labels"
```

> 💡 **常量名 vs 目录名**:
> `TRAIN_LABELS_DIR` 这个**变量名**里有 "LABELS"(代码语义角色), 它的**值**指向 `train/labels`(磁盘目录名)——跟 ultralytics 训练时寻找标签的硬编码约定一致。

**铁律**: 任何模块需要路径, 只能 import paths.py 里的常量或函数, 不允许自己拼接:

```python

self.train_images = DATA_DIR / "train" / "images"

# ❌ 反模式 (拼半句)
self.raw_dataset = DATA_DIR / "raw" / dataset_name

# ✅ 正确
from odp_platform.common.paths import TRAIN_IMAGES_DIR, raw_dataset_root
self.train_images = TRAIN_IMAGES_DIR
self.raw_dataset = raw_dataset_root(dataset_name)
```

> **金句**: **"paths.py 是项目唯一的【路径真相来源】(Single Source of Truth)——绕开它就等于在项目里立了第二个版本的真相, 早晚会跟第一个版本打架。"**

## 0.4 paths.py 增量(本节新增 3 项)

> 💡 **为什么把 3 个 paths 增量放在阶段 0 一次性给齐, 而不是阶段 6 / 9 用到时再加?**
>
> `paths.py` 是项目的"基础设施", 它的修改应该集中在**"项目启动"和"加新子系统"** 这两个时刻——而不是散落在阶段 6 加一条、阶段 8 又加一条、阶段 9 再加一条。集中放在这里, 后续阶段都只"使用"不再"修改", commit 历史也更干净(D3 全部 11 次 commit, 没有一次叫"补一个 paths 常量")。
>
> **代价是**: 你现在看到 `raw_dataset_root()` 还不知道它用在哪——这没关系, 阶段 9 的 `orchestrator` 会把它接起来。先把它当作"orchestrator 即将要用的工具"放进工具箱。
>
> **这条惯例值得记下**: **改 paths.py 等于改基础设施, 把它的修改集中在少数几个时刻, 比"哪里用哪里加"更可控。**

D3 需要新增 3 个路径概念, 一次性加到 `paths.py` 末尾:

```python
# ============================================================
# D3 增量: 数据集配置目录 + 路径辅助函数
# ============================================================
DATASET_CONFIGS_DIR: Path = CONFIGS_DIR / "datasets"


def raw_dataset_root(dataset_name: str) -> Path:
    """返回某个数据集的 raw 根目录: data/raw/<name>/

    Args:
        dataset_name: 数据集名 (= data/raw/ 下的子目录名)
    """
    return RAW_DATA_DIR / dataset_name


def dataset_yaml_path(dataset_name: str) -> Path:
    """返回某数据集的 ultralytics yaml 输出路径:
    <DATASET_CONFIGS_DIR>/<name>.yaml
    """
    return DATASET_CONFIGS_DIR / f"{dataset_name}.yaml"
```

并把 `DATASET_CONFIGS_DIR` 加到 `get_dirs_to_initialize()` 的返回列表里, 让 `odp-init` 顺手把它建好:

```python
def get_dirs_to_initialize() -> List[Path]:
    return [
        # ... D2 已有的所有目录 ...
        DATASET_CONFIGS_DIR,   # ★ D3 新增
    ]
```

> ⚠️ **陷阱预警**: 不要把"数据集名"硬编码进 `paths.py`。`raw_dataset_root` 接受参数返回 Path, 而不是 `SAFETY_HELMET_DIR = RAW_DATA_DIR / "safety_helmet"`——后者每加一个数据集就要改 paths.py, 立刻违反开闭原则。**SSoT 不等于"全部写死"**, 而是"所有路径来源都从这里出"。

## 0.5 引入 common/constants.py (跨模块共享词汇表)

paths.py 解决"路径"的真相来源。但还有一些**跨模块共享的非路径常量**——比如"哪些扩展名算图像"、"任务类型有哪些"、"浮点容差是多少"——它们既不是路径, 也不是逻辑, 而是**词汇**。

立一个新文件 `apps/platform/src/odp_platform/common/constants.py`:

```python
# apps/platform/src/odp_platform/common/constants.py
"""项目级共享常量——所有模块的'共同词汇表'。

放在这里的标准:
    - 多模块共享 (>= 2 个模块用到)
    - 与具体业务无关 (纯定义, 不含逻辑)
    - 修改频率极低
"""
from typing import Tuple


# ============================================================
# 图像扩展名 (converter / splitter / validator 等模块共享)
# ============================================================
IMAGE_EXTENSIONS: Tuple[str, ...] = (
    ".jpg", ".jpeg", ".png", ".bmp", ".tiff", ".webp",
)


# ============================================================
# 标注格式名
# ============================================================
class AnnotationFormat:
    """支持的标注格式 (字符串常量, 不用 Enum 是为了 @register 装饰器能直接吃)"""
    PASCAL_VOC = "pascal_voc"
    COCO       = "coco"
    YOLO       = "yolo"

    @classmethod
    def all(cls) -> Tuple[str, ...]:
        return (cls.PASCAL_VOC, cls.COCO, cls.YOLO)


# ============================================================
# 任务类型
# ============================================================
class Task:
    """模型任务类型"""
    DETECT  = "detect"
    SEGMENT = "segment"

    @classmethod
    def all(cls) -> Tuple[str, ...]:
        return (cls.DETECT, cls.SEGMENT)


# ============================================================
# 浮点 / 划分相关
# ============================================================
DEFAULT_RANDOM_STATE: int = 42
"""默认随机种子。所有需要可复现的随机操作的默认值。"""

RATE_EPSILON: float = 1e-6
"""划分比例的浮点容差。

用途: 判断 train_rate / val_rate / test_rate 是否"实质等于 0 或 1"。
原因: Python 浮点运算的精度问题——比如 `1.0 - 0.7 - 0.3` 不等于 0,
      而是 5.55e-17, 必须用容差兜底。
"""


# ============================================================
# 数据集覆盖率阈值 (阶段 9 使用)
# ============================================================
COVERAGE_HARD_THRESHOLD: float = 0.5
"""图像-标注覆盖率硬阈值: 低于此值直接 fail-fast。"""

COVERAGE_SOFT_THRESHOLD: float = 0.9
"""图像-标注覆盖率软阈值: 低于此值仅警告。"""
```

> 💡 **为什么 `AnnotationFormat` 是类不是 Enum?**
> 因为后面注册表的 `@register("pascal_voc")` 装饰器吃**字符串**——Enum 写起来要 `AnnotationFormat.PASCAL_VOC.value`, 太啰嗦。用类常量, `AnnotationFormat.PASCAL_VOC == "pascal_voc"`, 一气呵成。

## 1.1 第一版: 单文件直接写

需求很具体: 把 VOC 格式的 XML 转成 YOLO 格式的 txt。**最朴素的写法**——一个文件一个函数搞定:

```python
# apps/platform/src/odp_platform/data_pipeline/converter.py
"""第一版: 朴素实现, 一个函数处理一种格式。"""
from __future__ import annotations

import logging
import xml.etree.ElementTree as ET
from pathlib import Path
from typing import List, Tuple

logger = logging.getLogger(__name__)


def convert_voc_to_yolo(
    input_dir: Path,
    output_labels_dir: Path,
) -> List[str]:
    """把 input_dir 下所有 VOC XML 转成 YOLO txt, 写到 output_labels_dir。

    Returns:
        类别名列表 (按首次出现顺序)
    """
    xml_files = sorted(input_dir.glob("*.xml"))
    if not xml_files:
        raise FileNotFoundError(f"在 {input_dir} 下未找到任何 XML")

    output_labels_dir.mkdir(parents=True, exist_ok=True)
    classes: List[str] = []

    for xml_path in xml_files:
        tree = ET.parse(xml_path)
        root = tree.getroot()

        # 取图像尺寸
        size = root.find("size")
        if size is None:
            logger.warning(f"{xml_path.name} 缺少 <size>, 跳过")
            continue
        W = float(size.findtext("width", "0"))
        H = float(size.findtext("height", "0"))
        if W <= 0 or H <= 0:
            logger.warning(f"{xml_path.name} 尺寸非法 (W={W}, H={H}), 跳过")
            continue

        lines = []
        for obj in root.findall("object"):
            name = obj.findtext("name")
            if name is None:
                continue
            if name not in classes:
                classes.append(name)
            cls_id = classes.index(name)

            bbox = obj.find("bndbox")
            xmin = float(bbox.findtext("xmin"))
            ymin = float(bbox.findtext("ymin"))
            xmax = float(bbox.findtext("xmax"))
            ymax = float(bbox.findtext("ymax"))

            cx = (xmin + xmax) / 2 / W
            cy = (ymin + ymax) / 2 / H
            w  = (xmax - xmin) / W
            h  = (ymax - ymin) / H
            lines.append(f"{cls_id} {cx:.6f} {cy:.6f} {w:.6f} {h:.6f}")

        out_txt = output_labels_dir / (xml_path.stem + ".txt")
        out_txt.write_text("\n".join(lines), encoding="utf-8")

    return classes
```

现在想想:

- 加第 3 种格式(yolo 直通) → 调用方加一行 elif
- 加第 4 种格式(LabelMe) → 调用方加一行 elif
- 加第 5 种格式(CreateML) → 调用方加一行 elif

更糟的是: **不只一处需要这个 if/elif**——CLI 一处、pipeline 一处、测试 fixture 一处、说不定 GUI 还有一处。

这就是 if/elif 的**漂移**: **它不是一个分发点, 是 N 个**, 每加一种格式要改 N 个文件。

> **金句**: **"if/elif 的本质问题不是'丑', 是'分支信息会跨文件复制'——同一个'我支持哪些格式'的真相, 散在 N 个文件里, 改一处忘一处, 必然漂移。"**

## 1.4 第一次抢救: service.py 中间层(还是不够)

直觉反应: 把 if/elif 集中到一个**中间层**, 调用方只跟中间层说话:

```python
# data_pipeline/service.py(初版,仍有问题)
def convert(format: str, input_dir, output_labels_dir):
    if format == "pascal_voc":
        return convert_voc_to_yolo(input_dir, output_labels_dir)
    elif format == "coco":
        return convert_coco_to_yolo(input_dir, output_labels_dir)
    else:
        raise ValueError(f"不支持的格式: {format}")
```

调用方变成:

```python
classes = service.convert(format, input_dir, output_labels_dir)
```

**漂移问题修了**——只剩一个 if/elif。但**新问题来了**: **加第 N+1 种格式还是要改 `service.py`** —— 加 createml → 改 service.py 加一行 elif → service.py 体积线性增长。

**这不开闭**。开闭原则:

> **对扩展开放, 对修改封闭——加新格式应该【新增文件】, 而不是【修改老文件】。**

## 1.5 真正的解药: 注册表模式

让每个 converter **自己声明"我能处理什么格式"**, 中间层只查表分发——这就是**注册表(Registry)模式**。

设计:

```
data_pipeline/
├── __init__.py      (对外公共 API)
├── registry.py      (框架: 注册表 + @register 装饰器 + ConvertOptions)
├── service.py       (框架: 几行的调度层, 永不增长)
└── core/            (实现: 各格式 converter, 想加新格式就加新文件)
    ├── __init__.py
    ├── pascal_voc.py
    └── coco.py
```

**关键观察**:

- `registry.py` + `service.py` 是**框架**, 永远不动
- `core/*.py` 是**实现**, 加新格式 = 加新文件
- **物理目录反映了"开闭原则"**——框架在外, 实现在内, 内层增长不污染外层

## 1.6 横向对比: 工厂 vs ABC vs 注册表 (选读)

学到这里你可能会问: 多态分发的常见模式有好几个, 为什么我们选**注册表**而不是**工厂**或**抽象基类**? 简短对比一下, 帮你把这三个模式归位——下次遇到类似问题时, 能立刻选对工具。

### 工厂模式 (Factory)

```python
def create_converter(format_name):
    if format_name == "voc":
        return VOCConverter()
    elif format_name == "coco":
        return COCOConverter()
    else:
        raise ValueError(...)
```

**问题**: 工厂内部还是 if/elif, 加新格式还是要改工厂——**等于把撞墙①的"分发漂移"集中到一个文件,但没解决"加新格式要改老文件"的根本问题**。

### 抽象基类 (ABC)

```python
from abc import ABC, abstractmethod

class Converter(ABC):
    @abstractmethod
    def convert(self, input_dir, output_dir, options): ...

class VOCConverter(Converter):
    def convert(self, ...): ...
```

**问题**: 调用方还是要"知道有 VOCConverter 这个类"才能用。**没解决"按字符串名查找"的需求**——CLI 收到 `--format pascal_voc` 之后, 仍然要写 if/elif 才能挑到对应的类。ABC 解决的是"接口一致", 不是"按名字找对象"。

### 注册表 (Registry, 我们选这个)

```python
@register("pascal_voc", supported_tasks=(Task.DETECT,))
def convert_voc(input_dir, output_labels_dir, options): ...
```

**优点全占**:
- 加新格式: 写新文件 + 加装饰器, **registry / service / 其他 converter 都一行不动**(配合 pkgutil 自动发现)
- 按字符串名查找(CLI 友好: `--format pascal_voc` 直接走)
- 能力可声明(`supported_tasks`)
- Python 圈惯例——Flask 的 `@app.route`、pytest 的 `@pytest.fixture`、Django 的 admin 注册, 本质都是同一个模式

### 综合

| 维度 | 工厂 | ABC | 注册表 |
|---|---|---|---|
| 加新种类不改老代码 | ❌(改工厂) | ✅(加子类) | ✅(加新文件) |
| 按字符串名查找 | ✅ | ❌ | ✅ |
| 能力声明 (谁支持什么) | ❌ | ❌ | ✅ |
| 函数级实现 (不必造类) | 部分 | ❌ | ✅ |

**三个模式回答的是三个不同问题**:
- 工厂解决了"**怎么造对象**"
- ABC 解决了"**对象遵守同一接口**"
- 注册表解决了"**按名字找对象 + 它能干什么**"

> **金句**: **"工厂、ABC、注册表不是同一题的三个答案——是三道不同的题。下次设计模块时, 先想清楚你要解决哪一题。"**

# 阶段 2: registry.py + ConvertOptions + 能力声明

注册表听起来玄, 实现起来就是一个 dict + 一个装饰器。但要顺道解决另外两个真问题, 否则注册表也救不了:

- **问题 A**: 各格式 converter 参数不一致(VOC 不需要 task, COCO 需要; COCO 有 91→80 类映射, VOC 没有)——函数签名压不平
- **问题 B**: "支持检测还是分割"是 converter 的**能力**, 也得显式声明, 否则用户传 `task=segment` 给一个只会 detect 的 converter, 错误延迟到运行时

⚠️ **陷阱预警 (问题 A)**:

```python
# ❌ 反模式: 各 converter 函数签名各不一样
def convert_voc(input_dir, output_labels_dir): ...
def convert_coco(input_dir, output_labels_dir, task, cls91to80): ...
def convert_yolo(input_dir, output_labels_dir, classes): ...

# 调用方只能 if format == "coco": ... elif format == "yolo": ... 再走一遍
```

→ **解药**: 把所有可能的参数打包成一个 `ConvertOptions` dataclass, **所有 converter 函数签名统一**, 不需要的字段忽略。

⚠️ **陷阱预警 (问题 B)**:

```python
# ❌ 反模式: 把 task 藏在 coco 专属参数里
def convert_coco(..., coco_task="detect"):
    if coco_task == "segment": ...

# 加 yolo 也想支持 segment → 又得在 yolo 函数里加 yolo_task 参数, 重复
```

→ **解药**: `task` 是 `ConvertOptions` 的一等字段, 每个 converter 通过 `@register(supported_tasks=...)` 声明自己支持哪些 task。

## 2.1 registry.py 完整代码

`apps/platform/src/odp_platform/data_pipeline/registry.py`:

```python
"""data_pipeline 注册表 + 统一参数包 + 能力声明。

设计要点:
    - 一个 dict (_REGISTRY) 存"format -> ConverterEntry"映射
    - @register 装饰器在 converter 文件被 import 时自动登记
    - core/ 下的 converter 模块由 pkgutil 自动发现, 新增格式无需改注册表
    - ConvertOptions 是所有 converter 共用的参数包
    - ConverterEntry 同时携带"实现函数 + 能力声明"
"""
from __future__ import annotations

import importlib
import logging
import pkgutil
from dataclasses import dataclass, field
from pathlib import Path
from typing import Callable, Dict, List, Optional, Tuple

from odp_platform.common.constants import Task

logger = logging.getLogger(__name__)


# ============================================================
# 统一参数包: 所有 converter 函数签名一致
# ============================================================
@dataclass
class ConvertOptions:
    """所有 converter 共用的参数包。

    每个 converter 按需取用——不需要的字段忽略即可。
    新增字段时, 老 converter 自动兼容(因为不读它就行)。
    """
    task: str = Task.DETECT
    """任务类型: detect / segment / ..."""

    classes: Optional[List[str]] = None
    """类别白名单。
    - 对 pascal_voc / coco: 一般为 None, converter 自己探测
    - 对 yolo: 必须传, 因为 yolo 格式不含类别名信息
    """

    coco_cls91to80: bool = False
    """COCO 专属: 是否做 91 类 → 80 类映射"""


# ============================================================
# 注册表条目
# ============================================================
ConverterFunc = Callable[[Path, Path, ConvertOptions], List[str]]
"""converter 函数签名:
    (input_dir, output_labels_dir, options) -> List[str](类别名)
"""


@dataclass(frozen=True)
class ConverterEntry:
    """注册表里一条记录: 函数 + 它的能力声明。"""
    func: ConverterFunc
    supported_tasks: Tuple[str, ...]

    def supports(self, task: str) -> bool:
        return task in self.supported_tasks


# ============================================================
# 注册表本体 (模块级单例)
# ============================================================
_REGISTRY: Dict[str, ConverterEntry] = {}


def register(
    format_name: str,
    supported_tasks: Tuple[str, ...] = (Task.DETECT,),
) -> Callable[[ConverterFunc], ConverterFunc]:
    """装饰器: 把一个 converter 函数注册到 _REGISTRY。

    Usage:
        @register("pascal_voc", supported_tasks=(Task.DETECT,))
        def convert_voc(input_dir, output_labels_dir, options):
            ...
    """
    def decorator(func: ConverterFunc) -> ConverterFunc:
        if format_name in _REGISTRY:
            logger.warning(f"格式 {format_name} 被重复注册, 后者覆盖前者")
        _REGISTRY[format_name] = ConverterEntry(
            func=func,
            supported_tasks=tuple(supported_tasks),
        )
        logger.debug(
            f"注册 converter: format={format_name}, "
            f"tasks={supported_tasks}"
        )
        return func
    return decorator


# ============================================================
# 查询 API (供 service.py / CLI / 测试使用)
# ============================================================
def get_converter(format_name: str) -> ConverterEntry:
    """按 format 名取出 ConverterEntry。

    Raises:
        ValueError: 未注册的格式
    """
    _lazy_init()
    if format_name not in _REGISTRY:
        raise ValueError(
            f"未注册的格式: {format_name!r}。"
            f"已注册: {sorted(_REGISTRY.keys())}"
        )
    return _REGISTRY[format_name]


def list_capabilities() -> Dict[str, Tuple[str, ...]]:
    """返回当前所有已注册格式 → 支持的 task 列表。
    用于 CLI --help 显示能力矩阵。"""
    _lazy_init()
    return {fmt: entry.supported_tasks for fmt, entry in _REGISTRY.items()}


# ============================================================
# 延迟初始化: 自动发现 core/ 下所有 converter
# ============================================================
_LAZY_INITIALIZED = False


def _lazy_init() -> None:
    """扫描 core/ 包, 触发其中所有模块的 @register 装饰器执行。

    为什么需要"延迟":
        如果 registry.py 顶部直接 import core/* 模块,
        而那些模块又 `from data_pipeline.registry import register`,
        就形成循环 import。把 import 推迟到首次查询时, 循环就解开了。

    为什么"自动发现"而不是手写 import:
        手写 `from .core import pascal_voc, coco, yolo` 看起来直白,
        但每加一种格式都要回这里改一行——装饰器的"开闭原则"打了折扣。
        用 pkgutil.iter_modules 扫描包目录, 新增 core/<format>.py 后
        什么都不用改, 文件存在即自动注册。
        (Flask blueprints / Django apps / pytest 插件都是这个模式)

    为什么 _LAZY_INITIALIZED 要放在最后置 True:
        如果放在 import 之前置 True (你可能这么写过),
        import 中途抛异常时, 标志位已经是 True 但 _REGISTRY 是空的,
        后续所有 get_converter / list_capabilities 都会静默返回空,
        而且不会再重试。放最后的好处: 异常如实抛出, 修好环境后
        下一次调用会重新尝试, 一旦成功就 latch 住不再重复扫描。
    """
    global _LAZY_INITIALIZED
    if _LAZY_INITIALIZED:
        return

    from odp_platform.data_pipeline import core

    for module_info in pkgutil.iter_modules(core.__path__):
        # 跳过以 _ 开头的私有/工具模块 (如 _helpers.py)
        if module_info.name.startswith("_"):
            continue
        importlib.import_module(f"{core.__name__}.{module_info.name}")

    _LAZY_INITIALIZED = True   # ★ 必须放在 import 全部成功之后
```

> 💡 **`pkgutil.iter_modules` 的语义**: 只列 `core/` 的**直接子模块**, 不递归, 不包括 `__init__.py` 本身, 不包括 `__pycache__`——刚好就是我们想要的"`core/` 下的所有 converter 文件"。新增 `core/labelme.py` 后, 重启进程就自动接入, 注册表代码一行都不用改。这才是装饰器真正的解耦。

> 💡 **`_lazy_init` 模式**: 这是注册表 + 装饰器组合的**标准武器**。Flask 的 url_map、pytest 的 fixture 收集、Django 的 admin 注册——本质上都是这个模式。学会它你的工具箱里就有了**"插件化系统"**的通用解法。

## 2.1.1 ConvertOptions 三个设计选择逐项拆解

ConvertOptions 看起来就是个普通的 dataclass, 但里面藏了三个**值得讲清楚**的小选择——这三个点你迁移到自己的项目里时最容易踩坑, 单独拆开讲。

### 选择 ①: 为什么用 dataclass, 不用 dict / namedtuple / **kwargs?

| 选项 | 拼写错误检查 | IDE 自动补全 | 类型注解 | 默认值 | 适用 |
|---|---|---|---|---|---|
| `dict`         | ❌ | ❌ | ❌ | ✅ | 配置文件 / JSON 出入口 |
| `namedtuple`   | ✅ | ✅ | 部分 | ✅ (3.7+) | 短小不变的值对象 |
| `dataclass`    | ✅ | ✅ | ✅ | ✅ | **本场景** |
| `**kwargs`     | ❌ | ❌ | ❌ | ✅ | 真正不知道参数有哪些的元函数 |

**`dataclass` 和 `namedtuple` 在表格上几乎打平**——真正的差别在**可变性**: `namedtuple` 是天然 frozen, 字段不能改; `dataclass` 默认可变, 想要不可变加 `@dataclass(frozen=True)`。本场景里 converter 不会改 options 本身, 但 ConvertOptions 可能未来加字段、加方法 (例如 `validate()`), `dataclass` 的可扩展性更适合这种"还在演化的参数对象"; `namedtuple` 适合"已经定型的小值对象"。

**为什么不用 `**kwargs`?** 给 service.py 的函数签名是项目的**对外契约**, `**kwargs` 等于把契约写成"我也不知道接受什么"——调用方传错了参数会被静默吃掉, IDE 没法补全, 文档也写不清楚。

### 选择 ②: 为什么 task 默认 `Task.DETECT`, 不强制传?

```python
task: str = Task.DETECT   # ← 有默认值
```

**因为 80% 的实际场景都是 detect**。让最常见的用法零成本(`ConvertOptions()` 就能用), 少数高级用户传 `task=Task.SEGMENT`。

> ⚠️ **反面教训**: 强制传等于"把绝大多数人不在乎的选择, 摆在 80% 用户面前每次都要回答一遍"——这是糟糕的 API 设计。API 设计的"默认值"是给主流用户用的, "可选参数"是给小众场景留的口子。

### 选择 ③: 为什么 classes 用 `Optional[List[str]]` 不用 `[]`?

```python
classes: Optional[List[str]] = None   # ← 不是 = []
```

**因为 `None` 和 `[]` 是两种不同的语义**:

- `None` = "用户没指定, 让 converter 自己探测"
- `[]` = "用户说【没有任何类别】" (这种情况几乎从不发生, 但语义上不能跟"没指定"混)

混用会让 converter 没法区分"没传"和"传了空"——直接踩进**二义性 bug 黄金陷阱**。日后调试时你只能怀疑"是用户传错了还是我没传"。

> ⚠️ **陷阱预警 (Python 默认参数的可变陷阱)**: 即使你想用 `[]` 当默认值, 也**不能直接写 `classes: List[str] = []`**——Python 的默认参数在函数定义时**只算一次**, 同一个 `[]` 会被所有调用共享, 第一个调用 append 进去的元素会污染下一个调用。dataclass 的 `field(default_factory=list)` 才是安全写法。

### 综合: 一个 dataclass 三句话

```python
task: str = Task.DETECT              # ← ② 主流场景零成本
classes: Optional[List[str]] = None  # ← ③ None 是"没指定", 不是"空"
coco_cls91to80: bool = False         # ← 格式专属字段加前缀, 命名空间清晰
```

> **金句**: **"API 设计有三个一等公民——【主流场景的默认值】【小众场景的可选参数】【两种 None 语义的区分】。把这三个想清楚, 你写出来的 API 就已经比大多数库好用了。"**

## 2.2 service.py: 5 行调度层

`apps/platform/src/odp_platform/data_pipeline/service.py`:

```python
"""data_pipeline 调度层——5 行接住所有格式。"""
from __future__ import annotations

from pathlib import Path
from typing import List

from odp_platform.data_pipeline.registry import (
    ConvertOptions, get_converter,
)


def convert_data_to_yolo(
    input_dir: Path,
    output_labels_dir: Path,
    annotation_format: str,
    options: ConvertOptions,
) -> List[str]:
    """统一入口: 根据 format 分发到具体 converter。

    Args:
        input_dir:         原始标注目录
        output_labels_dir: YOLO txt 输出目录
        annotation_format: 格式名 (见 AnnotationFormat)
        options:           ConvertOptions

    Returns:
        类别名列表 (顺序 = yolo class_id 顺序)

    Raises:
        ValueError: 格式未注册 或 该 converter 不支持请求的 task
    """
    entry = get_converter(annotation_format)

    if not entry.supports(options.task):
        raise ValueError(
            f"格式 {annotation_format!r} 不支持 task={options.task!r}。"
            f"支持: {entry.supported_tasks}"
        )

    return entry.func(input_dir, output_labels_dir, options)
```

**注意 `service.py` 永远只有这几十行**——无论将来加多少种格式, 它一行都不需要改。**这就是开闭原则的物理体现**。

## 3.1 代码

```python
"""PASCAL VOC XML → YOLO txt converter。"""
from __future__ import annotations

import logging
import xml.etree.ElementTree as ET
from pathlib import Path
from typing import List

from odp_platform.common.constants import AnnotationFormat, Task
from odp_platform.data_pipeline.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(AnnotationFormat.PASCAL_VOC, supported_tasks=(Task.DETECT,))
def convert_voc(
    input_dir: Path,
    output_labels_dir: Path,
    options: ConvertOptions,
) -> List[str]:
    """读 input_dir 下所有 .xml, 写 YOLO txt 到 output_labels_dir。

    Returns:
        类别名列表 (按 XML 中首次出现顺序)
    """
    xml_files = sorted(input_dir.glob("*.xml"))
    if not xml_files:
        raise FileNotFoundError(f"在 {input_dir} 下未找到任何 XML")

    output_labels_dir.mkdir(parents=True, exist_ok=True)
    # ★ 注意: None vs [] 的语义区分 (见 2.1.1 节)
    #   - None  → 用户没指定, converter 自动发现类别
    #   - []    → 用户说"白名单为空", 所有 object 都不在白名单 → 写空 txt
    #   不能写成 `not options.classes`, 那会把 [] 也当成 None
    auto_discover = options.classes is None
    classes: List[str] = list(options.classes) if options.classes is not None else []

    n_ok = n_skip = 0
    for xml_path in xml_files:
        ok = _convert_one(
            xml_path, output_labels_dir, classes, auto_discover,
        )
        if ok:
            n_ok += 1
        else:
            n_skip += 1

    logger.info(
        f"VOC 转换完成: {n_ok} 成功, {n_skip} 跳过, 类别 {len(classes)} 种"
    )
    return classes


def _convert_one(
    xml_path: Path,
    output_labels_dir: Path,
    classes: List[str],
    auto_discover: bool,
) -> bool:
    """转一个 XML, 写一个 txt。返回 True 表示成功。"""
    try:
        tree = ET.parse(xml_path)
    except ET.ParseError as e:
        logger.warning(f"{xml_path.name} XML 损坏: {e}, 跳过")
        return False

    root = tree.getroot()

    size = root.find("size")
    if size is None:
        logger.warning(f"{xml_path.name} 缺少 <size>, 跳过")
        return False
    W = float(size.findtext("width", "0"))
    H = float(size.findtext("height", "0"))
    if W <= 0 or H <= 0:
        logger.warning(f"{xml_path.name} 尺寸非法 (W={W}, H={H}), 跳过")
        return False

    lines: List[str] = []
    for obj in root.findall("object"):
        name = obj.findtext("name")
        if name is None:
            continue

        if name not in classes:
            if auto_discover:
                classes.append(name)
            else:
                logger.debug(f"{xml_path.name}: 类别 {name!r} 不在白名单, 跳过该 object")
                continue
        cls_id = classes.index(name)

        bbox = obj.find("bndbox")
        if bbox is None:
            continue
        try:
            xmin = float(bbox.findtext("xmin"))
            ymin = float(bbox.findtext("ymin"))
            xmax = float(bbox.findtext("xmax"))
            ymax = float(bbox.findtext("ymax"))
        except (TypeError, ValueError):
            logger.debug(f"{xml_path.name}: bndbox 数值非法, 跳过该 object")
            continue

        cx = (xmin + xmax) / 2 / W
        cy = (ymin + ymax) / 2 / H
        w  = (xmax - xmin) / W
        h  = (ymax - ymin) / H

        # 边界裁剪
        cx = max(0.0, min(1.0, cx))
        cy = max(0.0, min(1.0, cy))
        w  = max(0.0, min(1.0, w))
        h  = max(0.0, min(1.0, h))

        lines.append(f"{cls_id} {cx:.6f} {cy:.6f} {w:.6f} {h:.6f}")

    out_txt = output_labels_dir / (xml_path.stem + ".txt")
    out_txt.write_text("\n".join(lines), encoding="utf-8")
    return True
```

> ⚠️ **陷阱预警 (auto-discover vs 白名单)**:
> 这里的策略——`options.classes` 为 None 时自动发现, 非 None 时严格按白名单过滤——是**有意为之**:
>
> - **自动发现**: 一次性把数据集所有类别都吃下来, 适合"我就有这些数据, 全要"
> - **白名单**: 适合"这数据集有 80 类, 但我只关心 'person' 和 'car'", 其他 object 被跳过, 类别 id 也按白名单顺序而不是 XML 顺序
>
> **不能两者都没有**, 不然 `classes.index(name)` 会失败。

# 阶段 4: coco converter (tempfile 中转)

COCO 比 VOC 复杂得多——一个 JSON 文件可能含上千张图的标注, 而且 ultralytics 已经提供了 `convert_coco()` 工具函数。**我们不重新造轮子, 直接调它**。

但这里有个**真实工程问题**: `ultralytics.data.converter.convert_coco` 把结果输出到一个指定目录, 而**它的输出结构跟我们要的不一样**——这个差异如何处理? 直接写到 `output_labels_dir` 行不行?

## 4.0.1 先停 5 分钟想清楚: 中转目录的本质属性是什么?

讲具体方案之前, 先停 5 分钟想想——

> 🤔 **思考题**: 这个"中转目录"的**本质属性**是什么? 它的**生命周期**有多长?
>
> - 转换开始时它出生
> - ultralytics 转完, 摊平到 `output_labels_dir` 后, 它就**没用了**
> - 异常退出时也应该被清理(不能留垃圾)
> - 多个进程并发跑同一格式不应该互相干扰

把这几条属性放一起看, 描述跟什么类似?

→ **临时变量 / 栈帧 / CPU 缓存**——同一类: **会话级临时状态, 不该被命名, 不该被持久化**。

**这就引出一个反直觉的结论**: 它**不应该有名字**, 也**不应该在项目目录里**。任何"想给它起个名字、找个项目内的位置"的尝试, 都是把它当成"持久资产"对待——身份错了。

→ 操作系统已经给了答案: **`tempfile`**(`/tmp/`、`%TEMP%`、`/var/folders/...`)——它就是为"会话级临时状态"专门设计的位置。

> **金句**: **"在项目里给每个'临时需求'都建个目录的人, 三个月后会回过头发现项目根下多了 12 个 `.cache_xxx/`、`temp_xxx/`、`_staging_xxx/`——没人知道哪个还在用, 没人敢删。"**

## 4.0.2 三种"放在项目内"的反模式为什么都不对

⚠️ **陷阱预警 (临时目录放哪)**:

```python
# ❌ 反模式 A: 临时目录放在 data/raw/
tmp_dir = RAW_DATA_DIR / "coco_staged"
convert_coco(..., save_dir=str(tmp_dir))
# → 违反 0.2 的"data/raw/ 洁癖"铁律

# ❌ 反模式 B: 临时目录放在项目根
tmp_dir = ROOT_DIR / "coco_tmp"
# → 污染仓库, .gitignore 还要专门加一条

# ❌ 反模式 C: 直接写 output_labels_dir
convert_coco(..., save_dir=str(output_labels_dir))
# → ultralytics 在 output_labels_dir 下还会建子目录 (labels/、images/),
#   我们要的是平铺 *.txt, 结构对不上, 还得二次搬运
```

→ **解药**: 用 Python 标准库 `tempfile.mkdtemp()` ——操作系统会把它放进 `/tmp/`(Linux) 或 `%TEMP%`(Windows), **完全不污染项目目录**, 进程结束后自然清理。

## 4.1 代码

`apps/platform/src/odp_platform/data_pipeline/core/coco.py`:

```python
"""COCO JSON → YOLO txt converter, 走 ultralytics.convert_coco。

设计要点:
    - 不重造轮子: ultralytics 已有成熟实现, 调它
    - tempfile 中转: ultralytics 的输出结构跟我们不一样,
      先让它写到系统临时目录, 然后挑出我们要的 *.txt 平铺到
      output_labels_dir, 不污染 data/raw/ 也不污染仓库
"""
from __future__ import annotations

import json
import logging
import shutil
import tempfile
from pathlib import Path
from typing import List

from odp_platform.common.constants import AnnotationFormat, Task
from odp_platform.data_pipeline.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(
    AnnotationFormat.COCO,
    supported_tasks=(Task.DETECT, Task.SEGMENT),
)
def convert_coco(
    input_dir: Path,
    output_labels_dir: Path,
    options: ConvertOptions,
) -> List[str]:
    """把 input_dir 下的 COCO JSON 转 YOLO txt, 写到 output_labels_dir。

    Returns:
        类别名列表 (顺序与 COCO categories 的 'id' 升序一致)
    """
    # 第三方依赖延迟 import, 让模块导入更轻
    from ultralytics.data.converter import convert_coco as _ul_convert_coco

    # 1. 找 JSON
    json_files = sorted(input_dir.glob("*.json"))
    if not json_files:
        raise FileNotFoundError(f"在 {input_dir} 下未找到 COCO JSON")
    if len(json_files) > 1:
        logger.warning(
            f"找到多个 JSON, 仅使用第一个: {json_files[0].name}"
        )
    coco_json = json_files[0]

    # 2. 读 categories (用于返回类别名列表)
    classes = _read_classes(coco_json)
    logger.info(f"COCO 类别 {len(classes)} 种: {classes[:5]}...")

    # 3. ★ 关键: 用 tempfile 做中转, 不污染任何项目目录
    output_labels_dir.mkdir(parents=True, exist_ok=True)
    tmp_root = Path(tempfile.mkdtemp(prefix="odp_coco_"))
    logger.debug(f"COCO 临时中转目录: {tmp_root}")

    try:
        # ultralytics 把 coco_json 转成 yolo txt, 输出到 tmp_root 下
        _ul_convert_coco(
            labels_dir=str(input_dir),
            save_dir=str(tmp_root),
            use_segments=(options.task == Task.SEGMENT),
            cls91to80=options.coco_cls91to80,
        )

        # 从 tmp_root 找出所有 *.txt, 平铺复制到 output_labels_dir
        n_copied = _flatten_txts(tmp_root, output_labels_dir)
        logger.info(
            f"COCO 转换完成: {n_copied} 个 yolo txt 已写入 {output_labels_dir}"
        )

    finally:
        # 清理临时目录
        shutil.rmtree(tmp_root, ignore_errors=True)

    return classes


def _read_classes(coco_json: Path) -> List[str]:
    """从 COCO JSON 的 categories 段读取类别名, 按 id 升序。"""
    data = json.loads(coco_json.read_text(encoding="utf-8"))
    cats = sorted(data.get("categories", []), key=lambda c: c["id"])
    return [c["name"] for c in cats]


def _flatten_txts(src_root: Path, dst_dir: Path) -> int:
    """递归找 src_root 下所有 *.txt, 平铺复制到 dst_dir。"""
    n = 0
    for txt in src_root.rglob("*.txt"):
        shutil.copy2(txt, dst_dir / txt.name)
        n += 1
    return n
```

> 💡 **为什么"延迟 import" ultralytics**?
> ultralytics 是重型依赖(几百 MB, 含 torch / opencv)。如果 `registry.py` 顶部 import 它, 用户只想用 VOC converter 也得等几秒。**把第三方重依赖放在函数内 import**, 是一个常用优化, 也避免了"用户没装 ultralytics 但只想用 voc"的报错。

# 阶段 5: yolo passthrough (接口等价性)

第三个 converter——但它**最特殊**: 原始数据已经是 YOLO 格式了, 还需要"转换"吗?

直觉答案是 "不需要, 直接用就行"。但这个直觉是**陷阱**:

## 5.0.1 先停 5 分钟想清楚: yolo "converter" 应该干什么?

> 🤔 **思考题**: 既然 YOLO 不需要"格式转换", yolo "converter" 应该:
>
> - **选项 A**: 只【验证】TXT 合法, 然后返回 classes(**不写任何文件**)
> - **选项 B**: 只【返回】input_dir 这个路径, 让调用方拿这个原始目录
> - **选项 C**: 把 TXT 【硬链接 / 复制】到 output_labels_dir, **让接口跟其他 converter 一致**
>
> 选 A/B 的话, 调用方(下游 orchestrator)拿到返回值后会怎么处理? 它的 `staged_dir` 在哪? 其他格式的 `staged_dir` 在哪? **这两者一致吗?**

想清楚再往下看。

### A 和 B 的代价

A 和 B 都让 yolo converter "什么都不写到 output_labels_dir"——调用方就得想:

```python
# orchestrator 里(选 A 或 B 的话):
classes = convert_data_to_yolo(input_dir, staged_dir, format=fmt, options=opts)
if fmt == "yolo":
    actual_staged_dir = input_dir   # ← yolo 不用 staged_dir, 直接用 input
else:
    actual_staged_dir = staged_dir
```

**问题层层放大**:

1. **逻辑分叉**——orchestrator 处处要写 `if fmt == "yolo"` 特殊处理
2. **状态修改**——如果 `staged_dir` 是 `self` 字段, 这就是**原地修改 self 状态**(二次调用 `run()` 时状态错乱)
3. **未来扩展爆炸**——加 LabelMe 时要不要也"直通"? 加 yolov5_legacy 呢? **每个特殊格式都要在 orchestrator 里加分支, 撞墙①重演**

### C 的设计哲学: 接口等价性原则

> **核心洞察**: **所有 converter 都应该有【完全相同的输入输出形态】, 无论内部做什么**。

```
所有 converter 的统一形态:
    input:        input_dir + output_labels_dir + options
    side effect:  在 output_labels_dir 中产出 txt 文件
    output:       classes 列表

VOC converter:   解析 XML → 生成 txt   → 写 output_labels_dir
COCO converter:  调 ultralytics → tempfile 中转 → 摊平到 output_labels_dir
YOLO converter:  验证已有 txt  → 硬链接/复制   → output_labels_dir  ★ 也写!
```

orchestrator 看到的所有格式都**完全等价**——内部干啥它不知道, 也不关心。**不需要任何 if 分支**。

> **金句**: **"接口等价性(Interface Equivalence)——让所有实现都遵守同一外部契约, 内部可以差异巨大, 但外面看不出来。这是消除 if/elif 的终极武器。"**

下面正式给陷阱预警 + 解药:

⚠️ **陷阱预警 (yolo 直通的两种错误处理)**:

```python
# ❌ 反模式 A: pipeline 里搞特殊
if format == "yolo":
    # yolo 不调 converter, 直接用 input_dir 当 labels 目录
    labels_dir = input_dir
else:
    classes = convert_data_to_yolo(...)
    labels_dir = output_labels_dir
# → pipeline 处处要 if format == "yolo", 又是 if/elif 漂移!

# ❌ 反模式 B: yolo "converter" 啥也不做, 调用方负责把文件复制到 output
@register("yolo")
def convert_yolo(input_dir, output_labels_dir, options):
    return options.classes   # 啥也不做, 让调用方自己处理位置
# → 调用方又要判断 "yolo 时 labels 在 input_dir, 其他在 output", 同样污染调用方
```

→ **解药 (接口等价性原则)**: yolo "converter" **也输出到 `output_labels_dir`**——通过**硬链接** (Linux/同盘) 或 **拷贝** (跨盘/Windows) 把 input 的 txt"落地"到 output。这样**所有 converter 行为完全一致**, 调用方一行 if/elif 都不需要。

> **金句**: **"接口等价性 = 让特殊 case 看起来跟普通 case 一样——不是消灭特殊性, 是消灭【特殊性对外可见】。"**

## 5.1 代码

`apps/platform/src/odp_platform/data_pipeline/core/yolo.py`:

```python
"""YOLO 直通 converter (接口等价性): 验证 + 硬链接/复制。

设计要点:
    - YOLO 已是目标格式, 无需"转换", 但仍要"输出"到 output_labels_dir
      ——让上层调用方看到所有 converter 行为一致 (接口等价)
    - 优先硬链接 (零空间, 零 IO), 失败降级到 copy
    - 顺道验证 txt 合法性: 列数、class_id 范围、坐标范围
"""
from __future__ import annotations

import logging
import os
import shutil
from pathlib import Path
from typing import List

from odp_platform.common.constants import AnnotationFormat, Task
from odp_platform.data_pipeline.registry import ConvertOptions, register

logger = logging.getLogger(__name__)


@register(AnnotationFormat.YOLO, supported_tasks=(Task.DETECT, Task.SEGMENT))
def passthrough_yolo(
    input_dir: Path,
    output_labels_dir: Path,
    options: ConvertOptions,
) -> List[str]:
    """直通: 验证 input_dir 是合法的 YOLO 目录, 硬链接/复制到 output。"""
    if options.classes is None or len(options.classes) == 0:
        raise ValueError(
            "YOLO 格式不包含类别名信息, 必须通过 options.classes 显式提供"
        )

    txt_files = sorted(input_dir.glob("*.txt"))
    if not txt_files:
        raise FileNotFoundError(
            f"YOLO 直通: 在 {input_dir} 下未找到任何 .txt"
        )

    logger.info(
        f"YOLO 直通: 在 {input_dir} 中找到 {len(txt_files)} 个标签"
    )

    # 验证 (task 相关: detect 是 5 列, segment 是 1 + 偶数列)
    n_classes = len(options.classes)
    invalid = 0
    for txt in txt_files:
        if not _validate_yolo_txt(txt, n_classes, task=options.task):
            invalid += 1
    if invalid:
        logger.warning(
            f"YOLO 直通: {invalid} 个 txt 含非法行 (debug 日志记录详情)"
        )

    # 物理落地: 硬链接 > copy
    output_labels_dir.mkdir(parents=True, exist_ok=True)
    use_hardlink = _supports_hardlink(input_dir, output_labels_dir)
    method = "hardlink" if use_hardlink else "copy"
    logger.debug(f"YOLO 直通: 使用 {method} → {output_labels_dir}")

    for txt in txt_files:
        dst = output_labels_dir / txt.name
        if dst.exists():
            dst.unlink()
        if use_hardlink:
            try:
                os.link(txt, dst)
                continue
            except OSError as e:
                # 首次失败就 latch 住, 后续文件直接走 copy, 不再重复 try
                logger.debug(f"hardlink 失败, 降级到 copy: {e}")
                use_hardlink = False
        shutil.copy2(txt, dst)

    return list(options.classes)


def _validate_yolo_txt(txt: Path, n_classes: int, task: str) -> bool:
    """验证 YOLO txt 合法性。

    格式按 task 分支:
        DETECT  : 每行 'cls_id cx cy w h'  (5 列, 4 个坐标 ∈ [0, 1])
        SEGMENT : 每行 'cls_id x1 y1 x2 y2 ... xn yn'
                  (1 + 2N 列, N ≥ 3 个顶点, 2N 个坐标 ∈ [0, 1])

    空行允许 (代表无 object 的负样本)。
    """
    try:
        text = txt.read_text(encoding="utf-8")
    except OSError as e:
        logger.debug(f"{txt.name} 读取失败: {e}")
        return False

    for ln_no, line in enumerate(text.splitlines(), 1):
        line = line.strip()
        if not line:
            continue
        parts = line.split()
        try:
            cls = int(parts[0])
            coords = [float(x) for x in parts[1:]]
        except (ValueError, IndexError):
            logger.debug(f"{txt.name}:{ln_no} 解析失败")
            return False

        # 1. 列数校验 (task 相关)
        if task == Task.DETECT:
            if len(coords) != 4:
                logger.debug(
                    f"{txt.name}:{ln_no} DETECT 期望 5 列, 实际 {len(parts)}"
                )
                return False
        elif task == Task.SEGMENT:
            # cls + N 个 (x, y) 点对, N ≥ 3 → 坐标个数为偶数且 ≥ 6
            if len(coords) < 6 or len(coords) % 2 != 0:
                logger.debug(
                    f"{txt.name}:{ln_no} SEGMENT 期望 cls + 至少 3 个 (x,y) "
                    f"点对, 实际 {len(parts)} 列"
                )
                return False
        else:
            logger.debug(f"{txt.name}:{ln_no} 未知 task={task!r}")
            return False

        # 2. class_id 范围
        if not (0 <= cls < n_classes):
            logger.debug(f"{txt.name}:{ln_no} class_id={cls} 越界")
            return False

        # 3. 坐标范围 [0, 1]
        if not all(0.0 <= v <= 1.0 for v in coords):
            logger.debug(f"{txt.name}:{ln_no} 坐标越界 [0, 1]")
            return False
    return True


def _supports_hardlink(src_dir: Path, dst_dir: Path) -> bool:
    """探测两个目录是否在同一文件系统 (硬链接的必要条件)。"""
    try:
        return src_dir.stat().st_dev == dst_dir.stat().st_dev
    except OSError:
        return False
```

## 6.0 写代码前先想清楚: 6 步流程的关注点归属

下面要写的不只是"写几个文件"——而是**一次架构决策**: **端到端 6 步该让谁做?** 写代码之前先停下来做一张归属表, 比"先写代码再回头重构"省事得多。

从用户视角, 他给一个数据集名(比如 `safety_helmet`), 期望:

```
输入:  data/raw/safety_helmet/{images, annotations}/
输出:  data/{train,val,test}/{images, labels}/
       apps/platform/configs/datasets/safety_helmet.yaml (ultralytics 用)
```

中间 6 步:

| 步骤 | 关注什么 | 跟"标注格式"相关吗 | 该住哪 |
|---|---|---|---|
| 1. 校验 raw | 业务规则(目录结构 / 覆盖率) | ✗ | orchestrator (阶段 9) |
| 2. 转换 | 格式 A → yolo txt | ✓ 强相关 | `core/` (阶段 3-5 已做) |
| 3. 找匹配 | (image, label) 配对 | ✗ | orchestrator (IO 重, 难独立测) |
| 4. 划分 | 列表 → 三组 | ✗ | **`split/` 新模块**(纯逻辑, 可独测) |
| 5. 复制 | 三组 → 落盘 | ✗ | **`split/` 的 materializer** |
| 6. 写 yaml | 类别+划分+元数据 → 文件 | ✗ | **`split/` 的 yaml_writer** |

### 两个关键观察

**观察 ①: 步骤 4/5/6 都跟标注格式无关**

它们看到的 `(image, label)` 只是两个 Path——跟原始格式是 VOC 还是 COCO 完全没关系。**这就是把它们抽出 `core/` 之外的依据**: `core/` 是"格式相关"的家, 跟格式无关的逻辑放进来会污染这层职责。

**观察 ②: 步骤 4/5/6 又强相关**

划分结果(manifest)→ 落盘(materializer)→ yaml(写文件) 是一条直线, `manifest` 是中间数据载体。**这就是把它们打包成一个 `split/` 子包的依据**——而不是各自再立独立顶层模块。

### 备选架构: 为什么不把 split/ 提到顶层?

另一种合理的选择是: 让 `split/` 和 `data_pipeline/` **平级**, 形成

```
data_pipeline/  (只管转换)
data_splitter/  (只管划分/落盘/yaml)
pipelines/      (orchestrator)
```

三个顶层目录。这种结构也能工作——**为什么我们不这么做?**

→ "划分"**不是独立业务**, 是"准备训练数据"这件大事的**内部步骤**。

| 维度 | 三顶层 (拆) | 单顶层 (合, 我们选) |
|---|---|---|
| 业务地图清晰 | 顶层目录数量随技术细分膨胀 | 顶层 = 业务阶段 (data_pipeline / data_validation / training) |
| 子包耦合 | 跨子系统(split 要 import data_pipeline 的 manifest?) | 同一子包内, import 自然 |
| 新人 1 分钟看懂 | 要先理解三者关系 | 看顶层就知道"这是数据准备子系统" |

**顶层目录回答"这个项目有哪几件大事", 子目录回答"这件大事内部分几步"——别让技术职能爬到顶层去**。

> **金句**: **"架构决策的第一题不是'怎么拆', 是'按什么轴拆'。按业务阶段拆出来的顶层目录, 比按技术职能拆出来的, 更经得起项目演化。"**

### 路线图

下面 3 个阶段(6/7/8)分别建 `split/` 内的 3 个文件:

- **6**: `manifest.py` (数据类) + `splitter.py` (纯函数划分)
- **7**: `materializer.py` (落盘)
- **8**: `yaml_writer.py` (生成 ultralytics yaml, 含 `odp_meta` 元数据)

```python
"""SplitManifest——划分结果的数据载体。

设计选择:
    - 用 dataclass, 不用 dict——字段有类型, IDE 能补全
    - 同时携带"三组样本"和"划分元数据"(rates, random_state),
      让 yaml_writer 不需要从别处拼凑这些信息
"""
from __future__ import annotations

from dataclasses import dataclass, field
from pathlib import Path
from typing import List, Tuple

# 类型别名: 一对样本 = (image_path, label_path)
Pair = Tuple[Path, Path]
PairList = List[Pair]


@dataclass
class SplitManifest:
    """划分结果 + 可复现元数据。

    Attributes:
        train / val / test: 三组样本列表
        train_rate / val_rate / test_rate: 实际使用的比例 (保留浮点原值,
            用于写入 yaml 的 odp_meta)
        random_state: 实际使用的随机种子
    """
    train: PairList = field(default_factory=list)
    val:   PairList = field(default_factory=list)
    test:  PairList = field(default_factory=list)

    train_rate:   float = 0.0
    val_rate:     float = 0.0
    test_rate:    float = 0.0
    random_state: int   = 0

    def summary(self) -> dict:
        return {
            "train": len(self.train),
            "val":   len(self.val),
            "test":  len(self.test),
            "total": len(self.train) + len(self.val) + len(self.test),
        }
```

## 6.2 splitter.py: 纯函数划分

`apps/platform/src/odp_platform/data_pipeline/split/splitter.py`:

```python
"""split_pairs——把图像-标签对划分成 train/val/test。

设计选择:
    - 纯函数 (不接触硬盘), 输入输出全是内存对象
    - 边界处理集中在这里, 不漏到调用方
"""
from __future__ import annotations

import logging

from sklearn.model_selection import train_test_split

from odp_platform.common.constants import (
    DEFAULT_RANDOM_STATE, RATE_EPSILON,
)
from odp_platform.data_pipeline.split.manifest import (
    PairList, SplitManifest,
)

logger = logging.getLogger(__name__)


def split_pairs(
    pairs: PairList,
    train_rate: float = 0.8,
    val_rate: float = 0.1,
    random_state: int = DEFAULT_RANDOM_STATE,
) -> SplitManifest:
    """把 pairs 划成 train/val/test, 返回 SplitManifest。

    Args:
        pairs:        [(image_path, label_path), ...]
        train_rate:   训练集占比 [0, 1]
        val_rate:     验证集占比 [0, 1]
                      (test_rate 自动 = 1 - train_rate - val_rate)
        random_state: 随机种子
    """
    test_rate = 1.0 - train_rate - val_rate
    if not (0 <= train_rate <= 1 and 0 <= val_rate <= 1 and 0 <= test_rate <= 1):
        raise ValueError(
            f"比例越界: train={train_rate}, val={val_rate}, test={test_rate:.4f}"
        )

    manifest = SplitManifest(
        train_rate=train_rate,
        val_rate=val_rate,
        test_rate=test_rate,
        random_state=random_state,
    )

    n = len(pairs)
    if n == 0:
        return manifest

    # ---------- 边界 1: 样本数 < 3, 全归 train ----------
    if n < 3:
        logger.warning(f"split_pairs: 样本数 {n} < 3, 全归 train")
        manifest.train = list(pairs)
        return manifest

    # ---------- 边界 2: train_rate 实质 = 1, 全归 train ----------
    if train_rate >= 1.0 - RATE_EPSILON:
        manifest.train = list(pairs)
        return manifest

    # ---------- 第一次切: train vs temp ----------
    images = [p[0] for p in pairs]
    labels = [p[1] for p in pairs]

    train_i, temp_i, train_l, temp_l = train_test_split(
        images, labels,
        train_size=train_rate,
        random_state=random_state,
    )
    manifest.train = list(zip(train_i, train_l))

    if not temp_i:
        return manifest

    # ---------- 边界 3: val+test 实质 = 0 ----------
    remaining = val_rate + test_rate
    if remaining < RATE_EPSILON or len(temp_i) < 2:
        manifest.val = list(zip(temp_i, temp_l))
        return manifest

    # ---------- 边界 4: test_rate 实质 = 0, temp 全归 val ----------
    # 必须在算 val_size 之前判, 否则:
    #   val_size = val_rate / (val_rate + ε) ≈ 0.9999...8 < 1.0
    # 会让 sklearn 把 1 个样本错挤进 test
    if test_rate < RATE_EPSILON:
        manifest.val = list(zip(temp_i, temp_l))
        return manifest

    # ---------- 边界 5: val_rate 实质 = 0, temp 全归 test (对称) ----------
    if val_rate < RATE_EPSILON:
        manifest.test = list(zip(temp_i, temp_l))
        return manifest

    # ---------- 第二次切: temp -> val vs test ----------
    val_size = val_rate / remaining
    val_i, test_i, val_l, test_l = train_test_split(
        temp_i, temp_l,
        train_size=val_size,
        random_state=random_state,
    )
    manifest.val  = list(zip(val_i, val_l))
    manifest.test = list(zip(test_i, test_l))

    return manifest
```

> ⚠️ **陷阱预警 (浮点容差为什么不可少)**:
> 你可能想问: "用户传 `train=0.7, val=0.3`, test 显然就是 0, 干嘛还要 RATE_EPSILON 兜底?"
>
> 试试这个:
> ```python
> >>> 1.0 - 0.7 - 0.3
> 5.551115123125783e-17
> ```
> Python 浮点不是 0! 这个 5.5e-17 顺着代码往下走会变成:
> ```
> remaining = 0.3 + 5.5e-17 = 0.30000000000000004
> val_size  = 0.3 / 0.30000000000000004 = 0.9999999999999998
> ```
> 然后 sklearn 用 `floor(0.9999...8 * 30) = 29`, 把 **1 个本该归 val 的样本错挤到 test**——用户预期 70/30/0, 实际拿到 70/29/1。
>
> 这就是为什么 `constants.py` 里有 `RATE_EPSILON = 1e-6`, 而这里有 5 个边界判断。**浮点不等于实数, 工程里必须用容差**——这是 D3 第一条贯穿的工程素养。

> ⚠️ **陷阱预警 (sklearn train_size 的"开区间"约束)**:
> 即使没有上面的浮点问题, 你直接把 `train_size=1.0` 传给 `train_test_split`, 在较新版 sklearn 里会报 `InvalidParameterError: train_size must be in (0.0, 1.0)` ——它是**开区间**, 1.0 也不行。这就是边界 2 / 边界 4 / 边界 5 必须在调 sklearn **之前**拦截的另一个原因。

# 阶段 7: split/materializer.py (落盘)

splitter 产出的 `SplitManifest` 还只是内存里的"分组方案"——三组 `(image_path, label_path)` 对而已, 没有任何文件被移动。

**materializer 负责把这个方案落地**: 把三组样本分别复制到 `data/train/{images, labels}/` / `data/val/...` / `data/test/...` 三个目录。

## 7.1 设计权衡: 为什么用依赖注入 (DI)?

最简单的写法是 materializer 直接 import `paths.py`:

```python
# materializer.py 简单版
from odp_platform.common.paths import (
    TRAIN_IMAGES_DIR, TRAIN_LABELS_DIR,
    VAL_IMAGES_DIR,   VAL_LABELS_DIR,
    TEST_IMAGES_DIR,  TEST_LABELS_DIR,
)

def materialize(manifest):
    for img, lbl in manifest.train:
        shutil.copy(img, TRAIN_IMAGES_DIR / img.name)
        shutil.copy(lbl, TRAIN_LABELS_DIR / lbl.name)
    # ... 重复 val / test
```

**能跑, 但有两个问题**:

1. **测不动**: 单元测试想验证 "材料化逻辑对不对", 必须真的往 `data/train/` 写东西。每跑一次测试就污染真实数据目录, 跑完还要清理。
2. **不灵活**: 想把同一份 manifest 材料化到别处(比如临时实验目录)? 不行——目标路径写死在 paths.py 里。

> ⚠️ **陷阱预警 (硬依赖 vs 依赖注入)**:
> "直接 import 全局常量"是新手最常见的模块设计反模式之一。它让模块**跟全局状态耦合**, 测试时无法隔离, 复用时无法替换。
>
> **依赖注入 (DI)**: 把模块需要的"东西"作为参数传进来, 而不是模块自己去全局找。这里"东西"就是三组目标目录。

## 7.1.2 三场景对比: 为什么 DI 不只是"好看"

如果你觉得"测不动、不灵活"两条还不够说服你, 下面用三个**真实会发生**的场景, 逐个对比"写法 A (直接 import paths)" 和 "写法 B (DI 注入 output_dirs)" 的代价。

### 场景 ①: 单元测试

我想验证 "材料化逻辑写得对不对"——比如"传 100 对样本, 三个目录里各有多少文件"。

- **写法 A**: 测试**只能**把文件复制到真实的 `data/train/labels/` 下——因为路径来源是 import 死的, 测试代码改不动。要么污染真实数据目录, 要么 monkey-patch `paths` 模块。后者每个测试都要写一遍, 还容易写错。
- **写法 B**: 测试构造一个临时目录字典传进去:

  ```python
  with tempfile.TemporaryDirectory() as tmp:
      out = Path(tmp)
      output_dirs = SplitOutputDirs(
          train_images=out/"tr_img", train_labels=out/"tr_lbl",
          val_images=out/"v_img",   val_labels=out/"v_lbl",
          test_images=out/"t_img",  test_labels=out/"t_lbl",
      )
      materialize(manifest, output_dirs)
      # 验证 out 下文件正确
  ```
  
  **零污染, 零 mock**。`with` 退出自动清理, 跨测试隔离, 并发安全。

### 场景 ②: 输出到挂载点 / 网络盘 / S3-fuse

V1.1 假设产品提了需求: 把训练数据落到一个独立挂载点 `/mnt/datasets/<dataset>/`, 而不是项目内的 `data/`。

- **写法 A**: materializer **跟 paths.py 物理粘死**——要支持另一个目标位置, 得改 materializer 自己, 或者改 paths.py 让它指向新位置。两个都是**侵入式改动**, 而且 paths.py 改了之后, 其他用 paths 的地方全跟着变(可能并不想这样)。
- **写法 B**: caller 决定要落哪——传一份指向 `/mnt/datasets/...` 的 `SplitOutputDirs` 即可。**materializer 一个字不用改**。

### 场景 ③: paths.py 自己改名 (反对 DI 的人最爱举的反例)

"如果 paths.py 把 `TRAIN_LABELS_DIR` 的值从 `train/labels/` 改成 `train/yolo_labels/`, A 写法是不是自动跟着改? B 写法是不是还要 caller 重新构造?"

- **写法 A**: materializer 通过 import 拿到的就是新值, **自动跟着改**——这点 A 反而占优。
- **写法 B**: caller (orchestrator) 同样通过 import 拿到新值, 传给 materializer 的还是新值, **也自动跟着改**。打平。

### 综合

| 场景 | 写法 A (直接 import) | 写法 B (DI) |
|---|---|---|
| 单元测试 | 痛 (要 mock 或污染) | 爽 (传临时目录) |
| 多目标位置 | 痛 (改 materializer 或 paths) | 爽 (caller 自己定) |
| paths.py 改值 | 平 | 平 |

**写法 B 完胜**——多一个参数的"啰嗦", 换来三种场景里的**根本灵活性**。这就是**依赖注入**(Dependency Injection, DI)——**不让被调用方主动去拿资源, 而是让调用方把资源传进来**。在 Python 里它的表达形式可以朴素到就是"多一个函数参数"。

> **金句**: **"DI 的本质是【翻转资源获取的方向】——从'我自己去找' 变成 '调用我的人告诉我'。一个参数换三种灵活性, 永远划算。"**

## 7.2 代码

`apps/platform/src/odp_platform/data_pipeline/split/materializer.py`:

```python
"""把 SplitManifest 落地到三个目标目录。

设计选择:
    - 用 DI 接收目标目录, 不 import paths.py——便于测试和复用
    - 复制 (而非链接), 因为目标目录可能跟 raw 在不同盘
      (例: raw 在本地 SSD, train 数据放在网络盘)
"""
from __future__ import annotations

import logging
import shutil
from dataclasses import dataclass
from pathlib import Path

from odp_platform.data_pipeline.split.manifest import (
    PairList, SplitManifest,
)

logger = logging.getLogger(__name__)


@dataclass(frozen=True)
class SplitOutputDirs:
    """三组目标目录——materializer 落盘的去处。

    使用方在调用前从 paths.py 取常量构造这个对象, materializer
    本身不知道 paths.py 的存在 (DI)。
    """
    train_images: Path
    train_labels: Path
    val_images:   Path
    val_labels:   Path
    test_images:  Path
    test_labels:  Path

    def mkdir_all(self) -> None:
        for p in (
            self.train_images, self.train_labels,
            self.val_images,   self.val_labels,
            self.test_images,  self.test_labels,
        ):
            p.mkdir(parents=True, exist_ok=True)


def materialize(
    manifest: SplitManifest,
    output_dirs: SplitOutputDirs,
) -> dict:
    """把 manifest 的三组样本复制到 output_dirs。

    Returns:
        {split: 实际复制成功的样本数} 的字典
    """
    output_dirs.mkdir_all()

    counts = {}
    counts["train"] = _copy_pairs(
        manifest.train, output_dirs.train_images, output_dirs.train_labels
    )
    counts["val"] = _copy_pairs(
        manifest.val, output_dirs.val_images, output_dirs.val_labels
    )
    counts["test"] = _copy_pairs(
        manifest.test, output_dirs.test_images, output_dirs.test_labels
    )

    logger.info(
        f"materialize 完成: train={counts['train']}, "
        f"val={counts['val']}, test={counts['test']}"
    )
    return counts


def _copy_pairs(
    pairs: PairList,
    images_dst: Path,
    labels_dst: Path,
) -> int:
    """复制一组样本到目标目录。返回成功复制的对数。"""
    n_ok = 0
    for img_src, lbl_src in pairs:
        try:
            shutil.copy2(img_src, images_dst / img_src.name)
            shutil.copy2(lbl_src, labels_dst / lbl_src.name)
            n_ok += 1
        except OSError as e:
            logger.warning(f"复制失败 {img_src.name}: {e}")
    return n_ok
```

> 💡 **关于 `shutil.copy2`**: 跟 `copy` 的区别是它会保留文件元数据(修改时间、权限)。训练数据复制场景一般无所谓, 但工程惯例里 `copy2` 更稳——文件归属和时间戳可能未来用于审计。

## 7.3 阶段 7 暂不单独 commit

materializer 单独 commit 意义不大——它跟 yaml_writer 一起在阶段 8 之后才能真正被 orchestrator 用上。我们把 7 + 8 + orchestrator 第一版合并成一次 commit, 保持 commit 粒度跟"业务里程碑"对齐。

# 阶段 8: split/yaml_writer.py (含 odp_meta)

最后一个 split/ 模块——生成 ultralytics 训练时读的 `<dataset>.yaml`。

## 8.1 ultralytics 期望的 yaml 长什么样

```yaml
# 标准 ultralytics 训练 yaml
path: /abs/path/to/dataset_root        # 训练数据根
train: train/images                    # 训练图像相对路径
val: val/images                        # 验证图像相对路径
test: test/images                      # 测试图像相对路径

names:                                 # 类别 id -> 名称
  0: cat
  1: dog
```

ultralytics 拿这个 yaml 就能开训。

## 8.2 但只写这些信息不够——加 odp_meta 块

⚠️ **陷阱预警 (元数据缺失的"半年后剧本")**:

```
背景: 半年前你跑过一次模型训练, 用的某个 toy_voc 数据集。
半年后老板来问: "那次实验你 train/val/test 怎么分的? 用的什么 seed?"

如果 yaml 里只有 ultralytics 期望的字段:
    你: "呃...我看下..."
    你: (翻 git log, 翻笔记, 翻 chat 记录) 
    你: (找不到)

如果 yaml 里有 odp_meta 块:
    你: (cat yaml) 0.8/0.1/0.1, seed=42, 2026-03-15 跑的
    用时: 10 秒
```

`split_pairs` 是**纯函数**——给定输入相同 + random_state 相同, 输出 100% 一样。也就是说**只要持久化好元数据, 任何人任何时候都能精确复现这次划分**。这个能力**不写进 yaml 就被浪费**。

所以 yaml_writer 在 ultralytics 标准字段之外, 加一个非标准的 `odp_meta` 块, 把所有"半年后会被问到"的信息记下来。

## 8.3 代码

`apps/platform/src/odp_platform/data_pipeline/split/yaml_writer.py`:

```python
"""生成 ultralytics 训练用 yaml, 加 odp_meta 块持久化划分元数据。"""
from __future__ import annotations

import logging
from datetime import datetime
from pathlib import Path
from typing import List

import yaml

from odp_platform.data_pipeline.split.manifest import SplitManifest

logger = logging.getLogger(__name__)


def write_dataset_yaml(
    yaml_path: Path,
    *,
    dataset_root: Path,
    classes: List[str],
    manifest: SplitManifest,
    dataset_name: str,
    source_format: str,
    task: str,
) -> None:
    """写一份 ultralytics 兼容的 yaml, 含 odp_meta 块。

    Args:
        yaml_path:     输出 yaml 路径
        dataset_root:  ultralytics 的 path 字段值 (训练数据根)
        classes:       类别名列表 (顺序 = yolo class_id)
        manifest:      划分结果 + 元数据
        dataset_name:  数据集名 (写入 odp_meta)
        source_format: 原始格式名 (写入 odp_meta)
        task:          任务类型 (写入 odp_meta)
    """
    doc = {
        # ============================================================
        # ultralytics 标准字段
        # ============================================================
        "path":  str(dataset_root.resolve()),
        "train": "train/images",
        "val":   "val/images",
        "test":  "test/images",
        "names": {i: name for i, name in enumerate(classes)},

        # ============================================================
        # odp_meta: 非标准, 仅 ODPlatform 自己读
        # 用途: 实验追溯 / 划分复现 / 数据沿袭
        # ============================================================
        "odp_meta": {
            "dataset":       dataset_name,
            "source_format": source_format,
            "task":          task,
            "created_at":    datetime.now().isoformat(timespec="seconds"),
            "split": {
                "train_rate":   round(manifest.train_rate, 6),
                "val_rate":     round(manifest.val_rate, 6),
                "test_rate":    round(manifest.test_rate, 6),
                "random_state": manifest.random_state,
                "counts":       manifest.summary(),
            },
            "schema_version": 1,
        },
    }

    yaml_path.parent.mkdir(parents=True, exist_ok=True)
    with yaml_path.open("w", encoding="utf-8") as f:
        yaml.safe_dump(doc, f, sort_keys=False, allow_unicode=True)

    logger.info(f"已写入 dataset yaml: {yaml_path}")
```

> 💡 **`schema_version: 1`** 是为未来兼容做的占位。如果以后 odp_meta 字段调整(比如加 `kfold_index` 字段、改名 `split.counts` → `split.distribution`), 老 yaml 还能识别 `schema_version=1` 走老的解析路径, 新 yaml 用 `schema_version=2` 走新路径。
>
> **所有非标准元数据块都该带版本号**, 这是工程惯例。**没有版本号的"未来兼容"是空话**——半年后你想改字段, 既不敢改(怕老 yaml 解析失败), 又不敢不改(新需求堵着)。带版本号就有了**升级路径**: 写一个 `_migrate_v1_to_v2(doc)` 函数即可。
>
> 反例: 历史上 docker-compose 的 yaml 没版本号时, 多个不兼容版本的 yaml 共存导致用户体验灾难, 后来 docker-compose 自己加了 `version: '3.8'` 字段才缓解。**前人踩过的坑, 我们直接绕开**。

> 💡 **ultralytics 会不会因为 yaml 里多了 `odp_meta` 就报错?**
> 不会。yaml 是开放结构, ultralytics 只读它认识的字段, 多出来的它无视。这正是用 yaml 而不是 JSON Schema 的好处——扩展友好。

# 阶段 9: orchestrator.py (端到端编排)

到此 6 个底层组件全到位:

```
data_pipeline/
├── registry.py     ← @register / get_converter / ConvertOptions
├── service.py      ← convert_data_to_yolo (5 行调度)
├── core/
│   ├── pascal_voc.py
│   ├── coco.py
│   └── yolo.py
└── split/
    ├── manifest.py
    ├── splitter.py        ← split_pairs (纯函数)
    ├── materializer.py    ← materialize + SplitOutputDirs (DI)
    └── yaml_writer.py     ← write_dataset_yaml (含 odp_meta)
```

orchestrator 把它们串起来, 暴露成**用户视角的"一次操作"**: 给一个数据集名 → 给一份训练用的 yaml。

## 9.1 端到端流程

```
输入: data/raw/<dataset>/{images, annotations}/

  1. 校验 raw 目录结构              ← orchestrator
  2. 调 converter 把 annotations 转 yolo txt (放 tempfile)  ← service.py
  3. 找匹配的 (image, label) 对     ← orchestrator
  4. 调 split_pairs 划分             ← split/splitter.py
  5. 调 materialize 落盘             ← split/materializer.py
  6. 调 write_dataset_yaml 生成 yaml ← split/yaml_writer.py

输出: data/train/{images,labels}/, data/val/..., data/test/...
      apps/platform/configs/datasets/<dataset>.yaml
```

orchestrator 只串顺序, **不实现任何业务规则**——所有规则都在它调的 6 个组件里。

## 9.2 第一版 orchestrator

`apps/platform/src/odp_platform/data_pipeline/orchestrator.py`:

```python
"""DatasetPipeline——端到端数据集准备编排器。

只串顺序, 不实现业务规则。所有规则都在它调的 6 个组件里。
"""
from __future__ import annotations

import logging
import tempfile
from pathlib import Path
from typing import List, Optional

from odp_platform.common.constants import (
    DEFAULT_RANDOM_STATE, IMAGE_EXTENSIONS, Task,
)
from odp_platform.common.paths import (
    TRAIN_IMAGES_DIR, TRAIN_LABELS_DIR,
    VAL_IMAGES_DIR,   VAL_LABELS_DIR,
    TEST_IMAGES_DIR,  TEST_LABELS_DIR,
    raw_dataset_root, dataset_yaml_path,
)
from odp_platform.data_pipeline.registry import (
    ConvertOptions, get_converter,
)
from odp_platform.data_pipeline.service import convert_data_to_yolo
from odp_platform.data_pipeline.split.manifest import PairList
from odp_platform.data_pipeline.split.materializer import (
    SplitOutputDirs, materialize,
)
from odp_platform.data_pipeline.split.splitter import split_pairs
from odp_platform.data_pipeline.split.yaml_writer import write_dataset_yaml

logger = logging.getLogger(__name__)


class DatasetPipeline:
    """端到端编排: raw -> yolo txt -> split -> 落盘 -> yaml。"""

    def __init__(
        self,
        dataset_name: str,
        annotation_format: str,
        *,
        task: str = Task.DETECT,
        train_rate: float = 0.8,
        val_rate: float = 0.1,
        classes: Optional[List[str]] = None,
        coco_cls91to80: bool = False,
        random_state: int = DEFAULT_RANDOM_STATE,
    ):
        self.dataset_name = dataset_name
        self.annotation_format = annotation_format
        self.task = task
        self.train_rate = train_rate
        self.val_rate = val_rate
        self.random_state = random_state

        # ⚠️ "用户传入的 classes" 是入参不可变状态;
        #    "最终的 classes (含 converter 自动发现的)" 是运行时确定的状态——
        #    分开存避免互相污染
        self._user_classes: Optional[List[str]] = classes
        self._final_classes: List[str] = []

        self._options = ConvertOptions(
            task=task,
            classes=classes,
            coco_cls91to80=coco_cls91to80,
        )

        # 路径全部从 paths.py 取, 不在这里拼字符串
        self.raw_root = raw_dataset_root(dataset_name)
        self.raw_images = self.raw_root / "images"
        self.raw_annotations = self.raw_root / "annotations"
        self.output_dirs = SplitOutputDirs(
            train_images=TRAIN_IMAGES_DIR,
            train_labels=TRAIN_LABELS_DIR,
            val_images=VAL_IMAGES_DIR,
            val_labels=VAL_LABELS_DIR,
            test_images=TEST_IMAGES_DIR,
            test_labels=TEST_LABELS_DIR,
        )
        self.yaml_out = dataset_yaml_path(dataset_name)

    def run(self) -> dict:
        """跑完端到端。返回 {train/val/test 计数 + yaml 路径}。"""
        logger.info(
            f"开始处理数据集 {self.dataset_name!r} "
            f"(format={self.annotation_format}, task={self.task})"
        )

        # 1. 校验 raw 目录 (本节先简单做, 阶段 10 再加覆盖率)
        self._check_raw()

        # 2. 校验 converter 支持当前 task
        entry = get_converter(self.annotation_format)
        if not entry.supports(self.task):
            raise ValueError(
                f"格式 {self.annotation_format!r} 不支持 task={self.task!r}。"
                f"支持: {entry.supported_tasks}"
            )

        # 3. 用 tempfile 中转: converter 写到临时目录, 不污染 data/raw/
        with tempfile.TemporaryDirectory(prefix="odp_pipe_") as tmp:
            staging = Path(tmp) / "labels"
            classes = convert_data_to_yolo(
                input_dir=self.raw_annotations,
                output_labels_dir=staging,
                annotation_format=self.annotation_format,
                options=self._options,
            )
            self._final_classes = classes
            logger.info(f"转换得到 {len(classes)} 个类别")

            # 4. 配对
            pairs = self._pair_images_with_labels(staging)
            logger.info(f"图像-标签配对: {len(pairs)} 对")

            # 5. 划分
            manifest = split_pairs(
                pairs,
                train_rate=self.train_rate,
                val_rate=self.val_rate,
                random_state=self.random_state,
            )
            logger.info(f"划分结果: {manifest.summary()}")

            # 6. 落盘
            counts = materialize(manifest, self.output_dirs)

            # 7. 写 yaml
            write_dataset_yaml(
                self.yaml_out,
                dataset_root=self.output_dirs.train_images.parent.parent,
                classes=classes,
                manifest=manifest,
                dataset_name=self.dataset_name,
                source_format=self.annotation_format,
                task=self.task,
            )

        return {"counts": counts, "yaml": str(self.yaml_out)}

    # ------------------------------------------------------------
    # 内部辅助
    # ------------------------------------------------------------
    def _check_raw(self) -> None:
        """阶段 9 第一版: 只检查目录存在。阶段 10 会加覆盖率前置。"""
        if not self.raw_root.is_dir():
            raise FileNotFoundError(
                f"数据集目录不存在: {self.raw_root}"
            )
        if not self.raw_images.is_dir():
            raise FileNotFoundError(
                f"缺少 images 子目录: {self.raw_images}"
            )
        if not self.raw_annotations.is_dir():
            raise FileNotFoundError(
                f"缺少 annotations 子目录: {self.raw_annotations}"
            )

    def _pair_images_with_labels(self, labels_dir: Path) -> PairList:
        """按 stem 配对 raw_images/ 下的图像 和 labels_dir 下的 yolo txt。"""
        # 建立 image stem -> Path 索引
        # ★ 大小写处理: pathlib.glob 在 Linux 是大小写敏感的, "*.jpg" 不会匹配
        #   IMG_001.JPG (相机默认大写)。改用 iterdir + suffix.lower() 一次列全部,
        #   再按 IMAGE_EXTENSIONS 过滤——跨平台一致, 且只扫一次目录。
        image_exts_lower = {ext.lower() for ext in IMAGE_EXTENSIONS}
        image_index = {}
        for img in self.raw_images.iterdir():
            if img.is_file() and img.suffix.lower() in image_exts_lower:
                image_index[img.stem] = img

        pairs: PairList = []
        for lbl in sorted(labels_dir.glob("*.txt")):
            img = image_index.get(lbl.stem)
            if img is None:
                logger.debug(f"标签 {lbl.name} 无对应图像, 跳过")
                continue
            pairs.append((img, lbl))

        return pairs
```

## 9.2.1 陷阱预警: 入参与状态分离 (一个真实痛点)

> ⚠️ **陷阱预警 (`_user_classes` vs `_final_classes`)**: 这是一个**最容易被忽视**的设计——为什么 classes 要分两个属性?

### 反面写法 (新手最常踩)

```python
def __init__(self, classes=None):
    self.classes = classes   # ← 既存"用户传入的", 又准备装"converter 跑完后最终的"

def run(self):
    if self.classes is None:
        self.classes = []
    self.classes = convert_data_to_yolo(
        ..., options=ConvertOptions(classes=self.classes)
    )
    # ★ 这一行执行完, self.classes 从"用户传入的"悄悄变成"converter 探测到的"——
    #   调用方读不出来, 二次 run() 行为不同
```

### 为什么这是 Bug 的源头

1. **调用方读到的 `self.classes` 含义不明**——他不知道这是"我传进去的"还是"converter 又探测后改过的"。读类的属性时, 一个变量两种含义 = 灾难。

2. **二次调用 `run()` 行为不一致**——
   - 第一次跑: `self.classes=None` → converter 自动发现 → 跑完变成 `['cat','dog','person']`
   - 第二次调 `run()` 时: `self.classes` 已经是 `['cat','dog','person']` 了 → converter 当成"用户传入的白名单"严格过滤
   - **同样的对象, 不同的行为**, 这种 bug 调到怀疑人生

3. **测试很难写**——验证"入参没被破坏"时, 你都不知道初值在哪一帧被冲掉的

### 正解: 入参像合同, 状态走自己的字段

```python
self._user_classes: Optional[List[str]] = classes   # ← 用户契约, 全程不变
self._final_classes: List[str] = []                  # ← 运行时状态, 跑 converter 后填
```

**入参像合同, 一旦签订就不再修改**; 状态可以变, 但走自己的字段。两个字段的命名也明确: `_user_*` = 来自用户, `_final_*` = 系统确定的最终值。

### 普适迁移

这个模式不只用于 classes。**任何"可选参数 + 运行时可能确定"的场景**都该这样分:

| 场景 | 入参(契约) | 状态(运行时确定) |
|---|---|---|
| 类别列表 | `_user_classes` (None 表示自动) | `_final_classes` (运行后填) |
| 输出路径 | `_user_output_dir` (None 表示用默认) | `_resolved_output_dir` (走 paths.py 解析后) |
| 随机种子 | `_user_seed` (None 表示自动生成) | `_final_seed` (生成后, 写入 yaml 用) |
| 超时时间 | `_user_timeout` (None 表示无限) | `_effective_timeout` (合并配置后) |

> **金句**: **"一个变量同时承担'输入参数'和'输出结果'两种身份, 是 90% '它跑了一次就坏了' bug 的源头。"**

## 9.4 回看: orchestrator 现在多瘦?

到这一步, 我们建议你打开 `orchestrator.py` **数一下行数**——大约 **120 行**(含注释和空行)。然后做一个**反事实想象**:

如果**不拆 split/** 子包, orchestrator 自己实现"分发 + 划分 + 落盘 + yaml 序列化", 它会有多少行?

| 假想"一锅炖"版本 | 行数估计 |
|---|---|
| 自己实现 if/elif 格式分发 | ~30 |
| 自己实现划分逻辑 (含 5 个边界) | ~50 |
| 自己实现 copy 三组到目标目录 | ~30 |
| 自己实现 yaml 序列化 + odp_meta 拼装 | ~40 |
| 校验 + 配对 + 日志 + __init__ | ~80 |
| **合计** | **~230 行** |

**对比实际的"委托版"**:

```
DatasetPipeline 自己只做 4 件事:
  1. 参数校验 (__init__)
  2. _check_raw (含覆盖率前置, 阶段 10 加强)
  3. _pair_images_with_labels  
  4. 串顺序 (run())

其余全部委托:
  - 转换 → service.convert_data_to_yolo (注册表分发)
  - 划分 → split.splitter.split_pairs (纯函数, sub-second 可测)
  - 落盘 → split.materializer.materialize (DI, 测试零污染)
  - yaml → split.yaml_writer.write_dataset_yaml (含 odp_meta)
```

| 维度 | "一锅炖"版 | "委托"版(当前) |
|---|---|---|
| 行数 | ~230 | ~120 |
| 单元测试可行性 | 难(要造数据集) | 易(每个组件可独测) |
| 加新格式要改 | orchestrator + 自己的 if/elif | 只加 `core/<format>.py` |
| 改划分策略要改 | orchestrator | 只改 `split/splitter.py` |
| 改 yaml schema 要改 | orchestrator | 只改 `split/yaml_writer.py` |

**减半行数还在其次, 关键是 orchestrator 现在只有一个职责: 串顺序**。它不知道 VOC XML 怎么解析, 不知道 sklearn 怎么划分, 不知道 yaml 怎么写——这些知识都封装在它调用的小工具里。

> **金句**: **"好的代码不是'写得多漂亮', 而是'读起来像目录'——一眼看出每件事属于谁。"**

### 一个反直觉的观察

新手看到 `orchestrator.run()` 里这种"6 步串顺序"的代码, 经常会觉得**它太薄了, 没什么内容**。但**这就是它该有的样子**:

```python
def run(self):
    self._check_raw()                  # 1. 自己做(业务规则)
    with tempfile.TemporaryDirectory(...) as tmp:
        classes = convert_data_to_yolo(...)  # 2. 委托
        pairs = self._pair_images_with_labels(...)   # 3. 自己做(IO)
        manifest = split_pairs(pairs, ...)           # 4. 委托
        counts = materialize(manifest, ...)          # 5. 委托
        write_dataset_yaml(...)                      # 6. 委托
```

**"薄"是因为别的地方都有归属——这才是关注点分离做对的样子**。如果你的 orchestrator 看起来很胖, 那不是因为业务复杂, 是因为还没拆完。

## 10.3 修复: 覆盖率前置校验

在 orchestrator 的 `_check_raw()` 里加上覆盖率检查——**在调 converter 之前**就检出问题, 节省你 30 分钟训练 + 2 小时 debug。

打开 `orchestrator.py`, 把 `_check_raw` 改成:

```python
    def _check_raw(self) -> None:
        """阶段 10 加强版: 目录检查 + 图像-标注覆盖率前置。"""
        from odp_platform.common.constants import (
            AnnotationFormat,
            COVERAGE_HARD_THRESHOLD, COVERAGE_SOFT_THRESHOLD,
            IMAGE_EXTENSIONS,
        )

        # ----- 基础目录检查 -----
        if not self.raw_root.is_dir():
            raise FileNotFoundError(f"数据集目录不存在: {self.raw_root}")
        if not self.raw_images.is_dir():
            raise FileNotFoundError(f"缺少 images 子目录: {self.raw_images}")
        if not self.raw_annotations.is_dir():
            raise FileNotFoundError(
                f"缺少 annotations 子目录: {self.raw_annotations}"
            )

        # ----- 覆盖率前置 -----
        # 数图像 (大小写不敏感, 见 _pair_images_with_labels 同款处理)
        image_exts_lower = {ext.lower() for ext in IMAGE_EXTENSIONS}
        n_images = 0
        image_stems = set()
        for img in self.raw_images.iterdir():
            if img.is_file() and img.suffix.lower() in image_exts_lower:
                n_images += 1
                image_stems.add(img.stem)

        if n_images == 0:
            raise FileNotFoundError(f"images/ 目录为空: {self.raw_images}")

        # 数标注 (按格式过滤扩展名, 但这里粗略统计 stem 即可)
        ann_ext_map = {
            AnnotationFormat.PASCAL_VOC: "*.xml",
            AnnotationFormat.COCO:       "*.json",
            AnnotationFormat.YOLO:       "*.txt",
        }
        pattern = ann_ext_map.get(self.annotation_format, "*")
        annotation_stems = {
            p.stem for p in self.raw_annotations.glob(pattern)
        }

        # ★ 对 coco, 一个 json 含整个数据集——覆盖率检查不适用,
        #   跳过即可 (coco 的覆盖率问题是另一种, 不在这里管)
        if self.annotation_format == AnnotationFormat.COCO:
            logger.debug("COCO 格式跳过 stem 覆盖率检查")
            return

        # 算覆盖率: 有标注的图像数 / 总图像数
        matched = len(image_stems & annotation_stems)
        coverage = matched / n_images

        logger.info(
            f"图像-标注覆盖率: {matched}/{n_images} = {coverage:.1%}"
        )

        if coverage < COVERAGE_HARD_THRESHOLD:
            raise ValueError(
                f"图像-标注覆盖率过低: {coverage:.1%} "
                f"(硬阈值 {COVERAGE_HARD_THRESHOLD:.0%})。\n"
                f"  总图像数: {n_images}\n"
                f"  有标注图像: {matched}\n"
                f"建议检查 annotations/ 目录是否完整, 或确认 --format 参数正确。"
            )
        elif coverage < COVERAGE_SOFT_THRESHOLD:
            logger.warning(
                f"⚠️  图像-标注覆盖率偏低: {coverage:.1%} "
                f"(软阈值 {COVERAGE_SOFT_THRESHOLD:.0%})。"
                f" 训练可继续, 但建议核对数据完整性。"
            )
```

阈值用的是 `constants.py` 里已经定好的 `COVERAGE_HARD_THRESHOLD=0.5` 和 `COVERAGE_SOFT_THRESHOLD=0.9`:

- < 50%: **直接 raise**, 用户必须修数据集才能继续
- 50%-90%: 警告, 但允许继续
- ≥ 90%: 正常通过

## 11.2 CLI 代码

`apps/platform/src/odp_platform/cli/transform_data.py`:

```python
"""odp-transform 命令——D3 端到端 CLI。"""
from __future__ import annotations

import argparse
import logging
import sys
from typing import List, Optional

from odp_platform.common.constants import AnnotationFormat, Task
from odp_platform.common.logging_utils import get_logger
from odp_platform.common.paths import LOGGING_DIR
from odp_platform.data_pipeline import (
    DatasetPipeline, list_capabilities,
)

logger = logging.getLogger(__name__)


def _build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(
        prog="odp-transform",
        description=(
            "ODPlatform 数据集转换 + 划分工具。\n"
            "把 data/raw/<dataset>/{images,annotations}/ 转换、划分,\n"
            "并生成 ultralytics 训练用的 <dataset>.yaml。"
        ),
        formatter_class=argparse.RawDescriptionHelpFormatter,
        epilog=_capability_matrix_text(),
    )
    parser.add_argument("--dataset", "-d", required=True,
                        help="数据集名 (= data/raw/ 下的子目录名)")
    parser.add_argument("--format", "-f", required=True,
                        choices=list(AnnotationFormat.all()),
                        help="原始标注格式")
    parser.add_argument("--task", "-t", default=Task.DETECT,
                        choices=list(Task.all()),
                        help="任务类型 (默认 detect)")
    parser.add_argument("--train-rate", type=float, default=0.8)
    parser.add_argument("--val-rate", type=float, default=0.1,
                        help="(test 自动 = 1 - train - val)")
    parser.add_argument("--classes", nargs="+", default=None,
                        help="类别白名单 (YOLO 格式必传)")
    parser.add_argument("--random-state", type=int, default=42,
                        help="随机种子 (会写入 yaml 的 odp_meta)")
    parser.add_argument("--coco-cls91to80", action="store_true",
                        help="(COCO 专属) 91 类映射到 80 类")
    return parser


def _capability_matrix_text() -> str:
    """生成 --help 末尾的能力矩阵——从 registry 实时读。"""
    try:
        caps = list_capabilities()
    except Exception:
        return ""
    lines = ["格式能力矩阵:"]
    for fmt, tasks in caps.items():
        lines.append(f"  {fmt:<15} -> {tasks}")
    return "\n".join(lines)


def main(argv: Optional[List[str]] = None) -> int:
    args = _build_parser().parse_args(argv)

    # CLI 入口装配 odp_platform 根 logger——之后所有 getLogger(__name__) 冒泡继承
    get_logger(base_path=LOGGING_DIR, log_type="transform")

    try:
        result = DatasetPipeline(
            dataset_name=args.dataset,
            annotation_format=args.format,
            task=args.task,
            train_rate=args.train_rate,
            val_rate=args.val_rate,
            classes=args.classes,
            coco_cls91to80=args.coco_cls91to80,
            random_state=args.random_state,
        ).run()
        logger.info(f"成功完成: {result}")
        return 0

    except (ValueError, NotImplementedError) as e:
        logger.error(f"参数/能力错误: {e}")
        return 1
    except FileNotFoundError as e:
        logger.error(f"数据集问题: {e}")
        return 2
    except Exception as e:
        logger.exception(f"未预期异常: {e}")
        return 3


if __name__ == "__main__":
    sys.exit(main())
```

> ⚠️ **陷阱预警 (`get_logger` 必须传 `base_path`)**:
> D2 设计的 `get_logger` 第一个参数是 `base_path: Path`(必填), 第二个才是 `log_type: str`(可选, 默认 `"general"`)。**少传 `base_path` 会立刻 TypeError**, 拿不到日志。
>
> 这里的标准用法是 `get_logger(base_path=LOGGING_DIR, log_type="transform")`——`LOGGING_DIR` 来自 paths.py(D2 已有), 是日志写盘的根目录。

## 12.3 ADR-002: 记录架构决策

ADR (Architecture Decision Record) 是工程里用来记录"我们为什么这样选, 而不是那样选"的标准格式。**半年后新人加入团队**, 看 ADR 就能理解项目骨架的来由, 而不是去读 commit log 拼凑。

`docs/architecture/ADR-002-data-pipeline.md`:

```markdown
# ADR-002: data_pipeline 子系统架构

**Date**: 2026-03-15 (D3)
**Status**: Accepted

## Context

ODPlatform 需要把多种格式的原始标注数据集 (VOC / COCO / YOLO)
转换成 ultralytics 训练所需的 YOLO 格式 + 划分 + yaml, 暴露成
一条 CLI 命令。

## Decision

采用**单顶层目录 + 双内部子包**结构:

```
data_pipeline/
├── registry.py       (框架: 注册表 + ConvertOptions)
├── service.py        (框架: 调度层)
├── orchestrator.py   (端到端编排)
├── core/             (实现: 各格式 converter)
└── split/            (划分/落盘/yaml)
```

九条核心原则:

1. **注册表 + 自动发现**: 加新格式 = 加新文件, 不改老代码 (开闭原则)。`@register` 装饰器登记到 `_REGISTRY`, `_lazy_init` 用 `pkgutil.iter_modules` 扫 `core/` 包自动 import——新增 `core/<format>.py` 后注册表代码一行都不用动
2. **接口等价性**: yolo 直通也"写"到 output_labels_dir, 调用方看到所有 converter 行为一致
3. **统一参数包**: ConvertOptions 统一签名, 各 converter 按需取用
4. **能力声明**: @register(supported_tasks=...), 不匹配在 service 层 fail-fast
5. **纯函数 + IO 分离**: split_pairs 纯函数 (sub-second 测试), materializer 才碰盘
6. **依赖注入**: materializer 不 import paths, 通过 SplitOutputDirs 接收目标目录
7. **元数据持久化**: yaml 的 odp_meta 块记录 random_state / 划分比例 / 时间, 支持复现
8. **入参与状态分离**: `_user_classes` (契约, 不变) vs `_final_classes` (运行时确定); 避免"同一变量两种含义"的 bug
9. **会话级临时状态用 tempfile**: 中转目录不放项目内 (`data/raw/` 洁癖铁律), 用系统临时目录, 自动清理

## Alternatives Considered

### A. 顶层平铺 (data_converter/ + data_splitter/ + pipelines/ 三个顶层目录)

**否决**: 顶层目录数量会随业务阶段(D6 验证 / D7 训练 / D8 推理)
线性膨胀。每个新阶段都加 2-3 个顶层目录, 学员看顶层就看不清"业务地图"。

我们坚持**按业务阶段拆顶层目录**:
```
data_pipeline/    (D3, 本节)
data_validation/  (D6, 下节)
training/         (D7)
inference/        (D8)
```

而**技术职能拆分(converter / splitter / orchestrator)留在子目录内部**——
顶层回答"项目有哪几件大事", 子目录回答"这件大事内部分几步"。

### B. 单文件 data_pipeline.py

**否决**: 加格式时整个文件越来越胖, 多人协作 merge 冲突频繁,
违反单一职责。一个 2000 行的 data_pipeline.py 没有任何工程师想 review。

### C. 让 ultralytics.convert_coco 直接写到 output_labels_dir

**否决**: ultralytics 的输出结构跟我们需要的不一致 (它会建子目录),
直接写会污染 output_labels_dir 的结构。改用 tempfile.mkdtemp()
中转 — 不污染 data/raw/, 不污染仓库, 进程结束自然清理。

### D. 让 yolo "converter" 啥也不做, 由 orchestrator 处理位置

**否决**: 这种方案让 orchestrator 处处要写 `if format == "yolo"` 特殊处理。
违反开闭原则的"分发漂移"问题会重演——每加一种"原生格式"都要在 orchestrator 加分支。
改用**接口等价性**: yolo 也通过硬链接/复制"写"到 output_labels_dir, 
orchestrator 一行 if 都不需要。

### E. 用工厂模式而不是注册表

**否决**: 工厂内部还是 `if/elif`, 加新格式还是要改工厂。注册表+@register 才真正
做到"加新文件 = 加新格式", 老代码一行不动。

### F. materializer 直接 import paths.py 而不用 DI

**否决**: 单元测试时无法把输出导向临时目录, 要么污染真实 `data/train/`, 要么
monkey-patch paths 模块——两个都是 hack。多输出目标(挂载点/网络盘)时也要改
materializer 本身。改用 SplitOutputDirs 注入, 调用方决定路径, 测试和复用都简单。

## Known Boundaries

- **task=segment 端到端**: 目前只有 coco converter 支持 segment 输出,
  但 orchestrator 的 yaml schema 未适配 polygon 标签 — 推迟到 D3.2。
- **K-fold 划分**: 当前只支持 train/val/test 三分。K-fold 是
  split_pairs 的扩展, 不是 orchestrator 的 — 推迟到 D3.x。
- **数据验证**: 当前只在 orchestrator 入口做覆盖率前置, 更深入的
  yolo txt 越界 / 类别分布检查推迟到 D6 data_validation/。

## References

- D3 讲义 (本文档配套)
- ultralytics docs: https://docs.ultralytics.com/datasets/
- sklearn train_test_split: 注意 train_size 必须在开区间 (0.0, 1.0)
```

## 直接给陷阱预警的设计点

下面 7 个不再让学员先撞再修, 而是讲义讲清"为什么这样写":

| 陷阱 | 直接给的解药 |
|---|---|
| 各 converter 参数不一致 | ConvertOptions 统一参数包 |
| task 不该是 coco 专属 | task 是 ConvertOptions 一等字段 + supported_tasks 能力声明 |
| 临时目录污染 data/raw/ 或仓库 | tempfile.mkdtemp() 中转 |
| yolo 直通让调用方写 if/elif | 接口等价性: yolo 也写 output_labels_dir |
| 浮点精度让划分错位 | RATE_EPSILON + 5 个边界判断 |
| materializer 硬依赖全局路径 | 依赖注入 SplitOutputDirs |
| random_state 只活在内存 | yaml 加 odp_meta 块持久化 |

## 学到的"工业级模式"

| 模式 | 用在 | 你的工具箱 |
|---|---|---|
| 注册表 (Registry) | data_pipeline.registry | 任何插件化系统 |
| 装饰器自动注册 | @register | Flask / pytest 也用 |
| 框架 vs 实现物理分离 | core/ 子包 | 所有插件化系统的标准结构 |
| 对外公共 API 收口 | __init__.py re-export | 任何"对外简洁、对内分层" |
| 统一参数包 (dataclass) | ConvertOptions / SplitOutputDirs | 多实现共用接口 |
| 能力声明 | supported_tasks | 实现各有所能 |
| 接口等价性 | yolo 直通 | 特殊 case 想被当普通处理 |
| tempfile 暂存 | coco / orchestrator | 会话级中间产物 |
| 延迟初始化 + 自动发现 | registry._lazy_init / pkgutil | 解决循环 import + 插件化扩展 |
| 单一真相来源 (SSoT) | paths.py / constants.py | 重复定义会打架 |
| 关注点分离 (SRP) | converter / split / orchestrator | 类做 3+ 件事就该拆 |
| 纯函数 + IO 分离 | split_pairs vs materialize | 可测试性最简单的提升 |
| 依赖注入 (DI) | SplitOutputDirs | 测试友好 + 灵活 |
| 入参与状态分离 | _user_classes vs _final_classes | 可选参数后期可能确定 |
| Fail-fast 前置 | _check_raw 覆盖率 | 尊重用户的时间 |
| 元数据持久化 | yaml.odp_meta | 半年后能回查 |
| Unix 退出码 | CLI 0/1/2/3 | 任何 CLI 工具的标配 |

**17 个工业级模式**——D3 一天的收获放到任何 Python 项目都能直接复用。

## 11 句"工程素养金句"汇总 (送给你贴到工位)

D3 散落在各章节的金句, 集中起来一遍:

1. **"约定的本质是【消除歧义】。一个项目里能用'数据集名'定位一切, 胜过靠多个参数拼凑路径。"** — 阶段 0.1
2. **"paths.py 是项目唯一的【路径真相来源】——绕开它就等于在项目里立了第二个版本的真相, 早晚会跟第一个版本打架。"** — 阶段 0.3
3. **"if/elif 的本质问题不是'丑', 是'分支信息会跨文件复制'——同一个'我支持哪些格式'的真相, 散在 N 个文件里, 改一处忘一处, 必然漂移。"** — 阶段 1.3
4. **"工厂、ABC、注册表不是同一题的三个答案——是三道不同的题。下次设计模块时, 先想清楚你要解决哪一题。"** — 阶段 1.6
5. **"API 设计有三个一等公民——【主流场景的默认值】【小众场景的可选参数】【两种 None 语义的区分】。"** — 阶段 2.1.1
6. **"在项目里给每个'临时需求'都建个目录的人, 三个月后会回过头发现项目根下多了 12 个 `.cache_xxx/`、`temp_xxx/`、`_staging_xxx/`——没人知道哪个还在用, 没人敢删。"** — 阶段 4.0.1
7. **"接口等价性 = 让特殊 case 看起来跟普通 case 一样——不是消灭特殊性, 是消灭【特殊性对外可见】。"** — 阶段 5
8. **"架构决策的第一题不是'怎么拆', 是'按什么轴拆'。按业务阶段拆出来的顶层目录, 比按技术职能拆出来的, 更经得起项目演化。"** — 阶段 6.0
9. **"测得快的代码就是好代码——能用纯函数表达的逻辑, 永远不该带 IO 进来。"** — 阶段 6.3
10. **"DI 的本质是【翻转资源获取的方向】——从'我自己去找' 变成 '调用我的人告诉我'。一个参数换三种灵活性, 永远划算。"** — 阶段 7.1.2
11. **"一个变量同时承担'输入参数'和'输出结果'两种身份, 是 90% '它跑了一次就坏了' bug 的源头。"** — 阶段 9.2.1
12. **"沉默的丢弃 = 工程犯罪。任何【数据被丢弃但不告知】的代码, 都该被改成 fail-fast。"** — 阶段 10.2
13. **"Fail-fast 不是悲观, 是尊重——尊重用户的时间, 尊重 GPU 的电费。"** — 阶段 10.4
14. **"好的代码不是'写得多漂亮', 而是'读起来像目录'——一眼看出每件事属于谁。"** — 阶段 9.4
15. **"工程师和程序员的差别——程序员让代码跑起来, 工程师让代码【可以解释自己为什么是这样】"** — 给你的最后一段话

## 练习 2: 加一种新格式 (必做)

CreateML 的 `(x, y)` 是**中心点**(不是左上角), 一个 JSON 含整个数据集。

> 💡 **注意 registry.py 也不用改**: 因为 `_lazy_init` 用 `pkgutil.iter_modules` 扫 `core/` 包, 你新建的 `createml.py` 会被自动 import, `@register` 装饰器会自动登记到 `_REGISTRY`。装饰器 + 自动发现的真正威力就在这里——**完全不碰老代码**, 才叫开闭原则。
>
> 验收方法: 写完后跑 `python -c "from odp_platform.data_pipeline import list_capabilities; print(list_capabilities())"`, 应该看到 `'createml'` 出现在输出里。
