## 1 引言

### 1.1 编写目的

本文档是 ODPlatform 通用目标检测开发平台"运行配置子系统"的软件需求规格说明书(以下简称"本文档"),用于明确定义该子系统的功能需求、非功能需求及外部接口需求,作为后续概要设计、详细设计、编码实现、系统测试和项目验收的依据。

本文档预期读者为:

- 软件架构师与系统设计人员(作为设计输入)
- 软件开发工程师(作为实现依据)
- 软件测试工程师(作为测试依据)
- 项目管理人员(作为进度与变更管理依据)
- 客户代表与领域专家(作为需求确认依据)

### 1.2 文档范围

#### 1.2.1 本文档涵盖的内容

本文档描述 ODPlatform 中"运行配置子系统"应当具备的能力,具体包括:

- 训练任务、验证任务、推理任务三类运行任务的参数配置定义
- 多来源配置(代码默认值、配置文件、命令行参数等)的加载与合并
- 配置值的合法性验证
- 配置参数的来源追溯
- 配置模板文件的自动生成与维护
- 与下游服务子系统(训练服务、验证服务、推理服务)的接口

#### 1.2.2 本文档不涵盖的内容

下列内容不在本文档范围内,由其他文档定义:

- 训练、验证、推理三类任务的执行逻辑(由对应的服务子系统 SRS 定义)
- 数据集准备与数据校验(由数据流水线子系统与数据校验子系统的 SRS 定义)
- 模型权重的存储与版本管理(由模型管理子系统的 SRS 定义)
- 日志记录、路径管理等基础设施(由公共基础子系统的 SRS 定义)
- 具体技术选型、模块划分、类设计、数据结构(由概要设计说明书与详细设计说明书定义)

## 2 总体描述

### 2.1 产品背景

#### 2.1.1 业务现状

ODPlatform 是基于 Ultralytics YOLO 算法库的通用目标检测开发平台。平台已建成的子系统包括:公共基础子系统、数据流水线子系统、数据校验子系统。当前需建设的运行配置子系统是面向训练、验证、推理三类核心运行任务提供统一参数管理能力的基础子系统。

#### 2.1.2 现存问题

通过对平台既有实现的调研与对内部用户的访谈,识别出运行配置管理领域存在如下五项业务问题:

**问题 P1:参数定义分散,维护成本高**

同一参数的定义当前分散于命令行解析代码、默认值常量、YAML 配置文件三处。新增或修改一个参数需在多处同步,导致维护工作量大、易遗漏、易出现定义漂移。

**问题 P2:非法参数值在运行时才暴露**

用户提供的非法参数值(如图像尺寸不符合算法要求、批次大小为零、设备名拼写错误等)未在配置阶段被拦截。错误在任务运行到中途才以底层异常的形式暴露,导致用户难以定位问题根因,排查耗时长。

**问题 P3:参数值的来源不可追溯**

任务运行完成后,用户无法判断某参数的最终生效值来自命令行、YAML 还是代码默认值。这导致:调参排查困难、实验结果难以复现、跨成员协作时配置归属不清。

**问题 P4:配置模板与代码定义漂移**

YAML 配置模板由人工维护,模板中的字段注释、字段集合与代码中的实际参数定义之间长期存在不一致。新人通过阅读模板理解参数含义后,在实际运行时常发现行为与模板描述不符。

**问题 P5:配置文件缺失时静默继续**

当用户预期使用的 YAML 配置文件不存在时,平台既有实现自动生成默认模板并继续运行任务。在 CI/CD 流水线、批量实验等无人值守场景下,用户可能在数小时之后才发现实际运行所用的并非预期配置,造成时间与计算资源的浪费。

#### 2.1.3 建设目标

运行配置子系统需以系统化的方式解决上述五项问题,具体目标为:

- **G1**:实现参数定义的唯一真相来源,消除多处维护
- **G2**:在配置阶段拦截非法参数,使错误尽早暴露
- **G3**:提供完整的参数来源追溯能力,支撑调参、复现与协作
- **G4**:由代码定义自动生成配置模板,杜绝注释与代码漂移
- **G5**:在配置缺失等异常情况下立即停止并指引用户,杜绝静默继续

### 2.2 产品功能概述

运行配置子系统应提供以下五类核心功能:

| 功能类别 | 功能简述 | 对应解决的问题 |
|----------|----------|----------------|
| 配置定义与结构管理 | 提供训练、验证、推理三类任务的参数定义,以及参数的描述、示例、调参建议等附加信息 | P1, P4 |
| 配置输入与加载 | 从 YAML 文件、命令行参数等多种来源加载用户提供的配置 | P5 |
| 配置合并与溯源 | 按优先级合并多来源配置,并记录每个参数的完整来源历史 | P3 |
| 配置验证 | 验证参数值的合法性、字段名的正确性、跨字段组合的一致性 | P2 |
| 配置模板生成 | 根据参数定义自动生成带注释的 YAML 模板供用户编辑 | P4 |

## 3 具体需求

### 3.1 外部接口需求

#### 3.1.1 用户接口

##### UI-01 命令行接口

子系统应提供命令行接口供用户主动触发配置模板生成功能。该接口应:

- 接受任务类型参数(训练 / 验证 / 推理),用以指定生成哪一类配置模板
- 接受输出路径参数,允许用户指定生成文件的位置
- 接受覆盖选项参数,控制目标文件已存在时的行为
- 接受备份选项参数,控制覆盖时是否保留原文件副本
- 支持通过 Python 模块入口和项目注册的短命令两种方式调用,二者行为一致
- 提供帮助信息,使用户在不查阅外部文档的情况下了解全部参数

#### 3.1.2 软件接口

##### SI-01 与上游子系统的接口

| 依赖子系统 | 接口用途 |
|------------|----------|
| 路径管理模块 | 获取配置文件目录路径、根据配置名解析具体文件路径 |
| 词汇定义模块 | 获取任务类型等领域词汇的合法取值集合 |
| 日志模块 | 输出面向运维的系统日志 |

##### SI-02 与下游子系统的接口

下游的训练服务、验证服务、推理服务子系统通过本子系统提供的编程接口获取已验证的配置。接口的具体形态由设计阶段确定,但必须满足:

- 下游调用方以最少的代码量完成配置准备工作,理想情形为单次调用
- 单次调用应同时返回"已验证的配置对象"与"配置溯源信息对象"
- 调用方无需了解 YAML 解析、命令行解析、合并算法、验证规则等内部实现
- 配置对象应可转换为可直接传递给 Ultralytics 接口的参数集合

#### 3.1.3 通信接口

本子系统不涉及网络通信。

### 3.2 功能需求

#### 3.2.1 配置定义与结构管理

##### FR-01 多任务类型的配置定义【M】

**需求描述**:系统应分别为训练任务、验证任务、推理任务提供独立的配置定义。

**详细要求**:

- 每类任务的配置应涵盖该任务在 Ultralytics 接口中可用的全部参数
- 三类任务共有的参数应在结构上统一,避免在各任务的配置中重复定义
- 配置定义应支持下游消费方以编程方式访问任一参数的当前值

##### FR-02 配置字段的元数据【M】

**需求描述**:每个配置字段除其值之外,应携带元数据,用于支撑文档生成、模板生成与审计报告。

**详细要求**:

字段元数据应至少包含以下信息:

| 元数据项 | 用途 |
|----------|------|
| 描述 | 简要说明字段含义 |
| 默认值 | 未被任何来源指定时使用的值 |
| 示例取值 | 若干典型可选值 |
| 调参建议 | 面向用户的使用提示,可包含多条 |
| 分组归属 | 字段所属的功能分组,用于在配置模板中按组排列 |
| 敏感标识 | 标识字段是否包含敏感信息,默认为否 |

字段元数据应与字段定义同处一地,以保证两者不发生漂移(对应建设目标 G1、G4)。

##### FR-03 任务语义与实验标识的分离【M】

**需求描述**:配置中"任务类型"与"实验标识"应由两个独立字段承担,避免一个字段同时承担两类正交职责。

**详细要求**:

- **任务类型字段**:
  - 取值受限于平台词汇定义模块提供的合法集合
  - 任何不在该集合中的取值应被拒绝
  - 拒绝时的错误信息应明确告知用户:如意图是命名本次实验,应使用实验标识字段

- **实验标识字段**:
  - 取值为自由字符串,允许为空
  - 不参与对 Ultralytics 的接口调用,仅用于平台内部的输出目录组织等用途

##### FR-04 框架内部字段的隔离【M】

**需求描述**:系统应区分"对外字段"与"框架内部字段",在向 Ultralytics 传递参数时仅传递对外字段。

**详细要求**:

- "框架内部字段"是指仅供平台内部使用、不应也不能传递给 Ultralytics 的字段(如实验标识)
- 不同任务类型对"哪些字段属于框架内部"的判定可能不同(例如任务类型字段:训练时需传递,验证与推理时由权重自动推断,不应传递)
- 各任务类型应能独立声明自己的框架内部字段集合,且该机制应避免在不同任务类型间产生代码重复

#### 3.2.2 配置输入与加载

##### FR-05 YAML 配置文件加载【M】

**需求描述**:系统应支持从 YAML 文件加载配置。

**详细要求**:

- 应接受绝对路径、相对路径和仅文件名(后两者基于约定的配置目录解析)三种形式
- 默认采用 UTF-8 编码读取;若解码失败,应在记录系统日志后尝试系统默认编码
- 空文件或仅包含注释的文件视为合法输入,等同于"全部使用默认值"
- YAML 顶层结构若非字典(如为列表、纯量等),应立即终止加载并报告错误
- YAML 文件中显式标记为空的字段不参与后续合并,等同于"未提供"

##### FR-06 YAML 文件不存在的处理【M】

**需求描述**:当用户指定的 YAML 配置文件不存在时,系统应立即终止,不得静默生成默认模板继续运行。

**详细要求**:

- 系统应抛出明确的错误并停止后续流程
- 错误信息中**必须**包含生成默认模板所需的完整命令,使用户可直接复制执行
- 错误信息中**必须**包含该文件期望出现的完整路径
- 本需求对应解决问题 P5

##### FR-07 命令行参数加载【M】

**需求描述**:系统应支持从命令行参数加载配置。

**详细要求**:

- 应自动排除非配置类的控制参数(如帮助开关、配置文件路径参数、调试开关等)
- 应自动排除以特定前缀(如下划线)标识的内部参数
- 用户未显式提供的参数(对应值为空)不应进入合并流程,以免错误覆盖其他来源
- 应支持参数名映射机制,允许将用户友好的命令行参数名映射为配置定义中的字段名

##### FR-08 配置来源的可扩展性【S】

**需求描述**:配置加载机制应支持将来增加新的配置来源(如环境变量、远程配置中心等)。

**详细要求**:

- 新增配置来源不应要求修改既有的加载、合并、验证逻辑
- 新增配置来源应能融入既有的优先级机制与溯源机制

#### 3.2.3 配置合并与溯源

##### FR-09 多来源配置合并【M】

**需求描述**:系统应支持将多个配置来源按优先级合并为最终配置。

**详细要求**:

- 默认优先级:命令行 > YAML > 代码默认值,优先级高的来源覆盖优先级低的来源
- 优先级应由调用方在调用时显式表达,而非硬编码于系统内部
- 来源数量不限于上述三档,应支持调用方指定任意数量的来源(配合 FR-08)
- 空值不参与覆盖,但显式的"假值"(布尔的否、数值零、空字符串、空列表等)必须被视为有效的覆盖值

##### FR-10 配置溯源【M】

**需求描述**:合并完成后,系统应能对任一字段提供完整的来源追溯能力。

**详细要求**:

对任一字段,系统应能回答:

- 当前生效值由哪个来源提供
- 在该来源生效之前,先前的值是什么、来自哪个来源
- 完整追溯过程应能回溯至代码默认值

溯源信息应以两种形式可用:

- **人类可读形式**:适合在终端展示或写入日志,使开发者能够直接阅读理解
- **结构化形式**:适合写入审计系统、数据库或日志聚合平台供机器处理

本需求对应解决问题 P3、建设目标 G3。

##### FR-11 验证错误的溯源增强【M】

**需求描述**:当某字段值在验证阶段被拒绝时,系统输出的错误信息应包含该字段的来源追溯链。

**详细要求**:

用户在看到验证错误时,无需自行回溯命令行历史与 YAML 文件,即可判断错误值的来源,从而快速定位是命令行输入错误、配置文件错误还是默认值不合理。

#### 3.2.4 配置验证

##### FR-12 字段值合法性验证【M】

**需求描述**:系统应在配置交付给下游消费方之前,验证所有字段值的合法性。

**详细要求**:

验证内容应至少包括:

- **取值范围**:数值字段的上下界、字符串字段的长度等
- **取值类型与格式**:字段值的类型符合声明
- **受限集合**:对应取值受限的字段,值应属于预定义集合
- **特殊语义**:如批次大小不应为零或负数(除非有特殊语义,如表示"自动")

验证失败时应明确指出失败字段、当前值、失败原因。本需求对应解决问题 P2、建设目标 G2。

##### FR-13 跨字段一致性验证【M】

**需求描述**:系统应能识别字段之间的语义依赖,对组合不合理的配置作出响应。

**详细要求**:

跨字段问题应区分两类处理:

- **互相矛盾型**:用户的意图自相矛盾(如声明"不保存检查点"同时声明"按周期保存检查点"),系统应**拒绝**并要求用户澄清
- **冗余无害型**:用户的意图明确但配置中存在不会生效的字段(如某项数据增强已禁用,却同时配置了仅在该增强启用时才有效的辅助参数),系统应**警告**但允许继续

##### FR-14 未知字段的拒绝【M】

**需求描述**:若用户在 YAML 或命令行中提供了配置定义中未声明的字段(如字段名拼写错误),系统应明确拒绝。

**详细要求**:

- 系统不得静默忽略未知字段
- 拒绝时应明确指出该字段名,便于用户排查拼写错误
- 本需求是对问题 P2 的补充覆盖

##### FR-15 警告与错误的渠道分离【M】

**需求描述**:由不同源头产生的非致命提示应通过不同渠道发出。

**详细要求**:

| 提示类型 | 输出渠道 | 受众 |
|----------|----------|------|
| 由用户输入引发的提示(字段值边缘、配置组合冗余等) | 面向用户的渠道,可被标准机制升级为错误 | 终端用户 |
| 由系统行为引发的提示(如编码降级、自动备份等) | 面向运维的日志渠道 | 运维人员 |

两类渠道应可被独立配置与捕获,不应混用。

#### 3.2.5 配置模板生成

##### FR-16 配置模板的自动生成【M】

**需求描述**:系统应能根据任一任务类型的配置定义,自动生成对应的 YAML 配置模板文件。

**详细要求**:

生成的模板应:

- 按字段所属的功能分组组织,分组在视觉上明显区分
- 每个字段附带其元数据中的描述、示例取值、调参建议等作为 YAML 注释
- 字段以其默认值作为初始值
- 文件头部包含生成时间与模板用途说明
- 文件尾部包含常见问题的简要解答

本需求对应解决问题 P4、建设目标 G4。

##### FR-17 生成模板的格式正确性【M】

**需求描述**:生成的 YAML 模板必须语法正确且语义自洽。

**详细要求**:

- 生成的模板必须可被本系统的 YAML 加载器正常解析
- 解析结果应与对应配置的默认值等价
- 布尔值、空值、列表等特殊类型必须按 YAML 标准格式输出
- 字符串中含 YAML 特殊字符时必须正确转义或加引号

##### FR-18 模板生成的安全策略【M】

**需求描述**:模板生成功能应保护用户已编辑的配置文件不被意外覆盖。

**详细要求**:

- **默认行为**:目标文件已存在时,跳过生成并明确提示用户该文件已存在
- **强制覆盖**:仅当用户显式指定覆盖选项时,才执行覆盖
- **覆盖时自动备份**:执行覆盖时,默认将原文件备份为带时间戳的副本
- **关闭备份**:仅当用户显式关闭备份时,才不进行备份

本需求对应约束 C-07。

##### FR-19 模板生成的命令行入口【M】

**需求描述**:系统应提供命令行入口供用户主动生成配置模板。

**详细要求**:参见 [3.1.1 UI-01 命令行接口](#ui-01-命令行接口)。

#### 3.2.6 对下游消费方的接口

##### FR-20 公开接口的明确界定【M】

**需求描述**:子系统应明确区分对外公开的接口与内部实现细节。

**详细要求**:

- 公开接口构成对下游消费方的稳定承诺,其变更应遵循版本兼容策略
- 内部实现细节可在不通知的情况下变更,下游消费方不应依赖
- 公开接口必须具备完整的文档说明,使用户可通过 IDE 或交互式帮助直接查阅

##### FR-21 一站式配置构建接口【M】

**需求描述**:系统应为训练、验证、推理三类任务分别提供一站式接口,在一次调用中完成完整的配置准备流程。

**详细要求**:

一次调用应完成以下流程:

1. 加载 YAML 配置文件(若指定)
2. 加载命令行参数(若指定)
3. 按优先级合并所有来源(允许调用方插入额外的自定义来源,见 FR-08)
4. 执行字段值合法性验证(FR-12)与跨字段一致性验证(FR-13)
5. 返回已验证的配置对象与配置溯源信息

该接口还应提供"仅预览不验证"的模式,使下游消费方能在执行验证前查看合并结果。

##### FR-22 配置对象到 Ultralytics 参数的转换【M】

**需求描述**:已验证的配置对象应能转换为可直接传递给 Ultralytics 接口的参数集合。

**详细要求**:

转换过程中应:

- 自动过滤值为空的字段
- 自动剔除框架内部字段(参见 FR-04)
- 输出结果可作为关键字参数传递给 Ultralytics 的训练、验证或推理接口

#### 3.2.7 配置审计与复现

##### FR-23 实验复现的配置快照【S】

**需求描述**:系统应能将一份配置导出为完整的、可序列化的快照,以支持实验复现。

**详细要求**:

- 快照应包含全部字段的值
- 快照应包含配置类型的标识信息,使日后可据此精确恢复
- 从快照恢复时应执行完整的验证流程,以便在配置定义后续演进时能立即发现不兼容情况

##### FR-24 敏感字段的脱敏【S】

**需求描述**:在产生人类可读报告或外发日志时,被标记为敏感的字段应自动脱敏。

**详细要求**:

- 被标记为敏感的字段(参见 FR-02 中的"敏感标识")在人类可读报告中应以脱敏字符串替代
- 在结构化审计日志中应避免包含敏感字段的值
- 仅在用户主动调用未脱敏的访问接口时才返回原始值

## 4 验收准则

本章定义子系统通过验收应满足的可观察行为。验收测试以黑盒方式针对公开接口和命令行入口进行,不要求验收人员了解内部实现细节。

### 4.1 功能验收准则

| 编号 | 验收场景 | 通过标准 | 关联需求 |
|------|----------|----------|----------|
| AC-01 | 不提供任何来源,直接构建训练配置 | 返回的配置对象中所有字段为其设计默认值 | FR-01, FR-09 |
| AC-02 | 提供合法的 YAML 文件构建训练配置 | YAML 中指定的字段值生效,未指定字段保持默认 | FR-05, FR-09 |
| AC-03 | 同时提供 YAML 与命令行,二者对同一字段给出不同值 | 命令行的值最终生效 | FR-09 |
| AC-04 | 指定的 YAML 文件不存在 | 系统报告错误并停止;错误信息中包含生成模板的完整命令与文件期望路径 | FR-06 |
| AC-05 | YAML 顶层结构为列表 | 系统报告错误并停止 | FR-05 |
| AC-06 | YAML 中存在配置定义未声明的字段 | 系统报告错误并指出该字段名 | FR-14 |
| AC-07 | 任务类型字段被赋予不在合法集合内的值 | 系统报告错误,错误信息明确建议改用实验标识字段 | FR-03 |
| AC-08 | 命令行向某数值字段传入越界值 | 系统报告错误,且错误信息中包含该字段的来源追溯链 | FR-11, FR-12 |
| AC-09 | 配置组合自相矛盾(如禁用保存却要求周期保存) | 系统拒绝该配置 | FR-13 |
| AC-10 | 配置组合冗余无害(如某增强已关闭却配置了关联参数) | 系统发出警告但允许配置构建成功 | FR-13 |
| AC-11 | 命令行某参数显式置为假值(布尔否、零、空字符串等) | 该值正确覆盖 YAML 或默认值,而非被视为"未提供" | FR-09 |
| AC-12 | 通过命令行入口生成训练配置模板 | 在期望路径生成 YAML 文件,内容包含分组结构、字段注释、默认值、文件头与文件尾 | FR-16, FR-19 |
| AC-13 | 用本系统加载新生成的 YAML 模板 | 加载结果与该任务类型的默认配置等价 | FR-17 |
| AC-14 | 目标配置文件已存在时再次执行生成命令(未指定覆盖选项) | 跳过生成,文件内容不变,终端有明确提示 | FR-18 |
| AC-15 | 目标配置文件已存在时使用覆盖选项执行生成命令 | 原文件被备份为带时间戳的副本,新文件被写入 | FR-18 |
| AC-16 | 对任一配置字段查询其来源追溯信息 | 返回完整的来源覆盖链,可回溯至代码默认值,且支持人类可读与结构化两种形式 | FR-10 |
| AC-17 | 将一份已验证的配置导出快照后,再从快照恢复 | 恢复出的配置对象与原配置在字段值上完全等价 | FR-23 |
| AC-18 | 配置中存在敏感字段,生成人类可读报告 | 报告中该字段的值被脱敏 | FR-24 |

## 附录 A 典型业务场景

本附录列出五个典型业务场景,用于辅助理解需求。本附录内容不构成额外的需求条目。

### 场景 A.1 新工程师首次使用平台跑训练

**场景描述**:某算法工程师刚加入项目,本地代码克隆完成后尚无任何 YAML 配置文件。

**期望流程**:

1. 工程师运行训练命令
2. 系统报告"配置文件不存在",并给出生成命令的具体形式
3. 工程师复制命令并执行,获得带注释的默认 YAML 模板
4. 工程师阅读模板中的字段注释,理解各参数的含义与典型取值
5. 工程师编辑关心的字段并保存
6. 工程师再次运行训练命令,系统加载 YAML 并开始训练

**涉及需求**:FR-06, FR-16, FR-17, FR-19, NFR-07, NFR-09

### 场景 A.2 工程师调参时疑惑某参数为何取此值

**场景描述**:工程师查看训练日志,发现学习率为某具体数值,但不确定该值是 YAML 中的设置还是自己在命令行中临时改的。

**期望流程**:

1. 工程师查看日志或溯源报告
2. 报告中明确显示该字段的完整来源追溯链
3. 工程师无需回溯命令行历史或对比 YAML 文件即可定位来源

**涉及需求**:FR-10, FR-11

### 场景 A.3 CI 流水线意外缺失配置文件

**场景描述**:CI 镜像构建脚本未将 YAML 配置文件打包进镜像。流水线触发训练任务。

**期望流程**:

1. 流水线启动训练
2. 系统立即报告配置文件缺失并停止
3. 流水线日志中明确显示缺失的文件路径与修复方式
4. 运维人员据此修复 CI 脚本

**不可接受的流程**:流水线静默生成默认配置并继续运行数小时,直至发现结果不符预期。

**涉及需求**:FR-06

### 场景 A.4 YAML 中字段名拼写错误

**场景描述**:工程师将训练轮数字段名 `epochs` 误拼写为 `epchs`。

**期望流程**:

1. 工程师运行训练
2. 系统在配置加载阶段立即报告未知字段 `epchs`
3. 工程师立即发现拼写错误并修正

**不可接受的流程**:系统静默忽略 `epchs`,使用默认轮数运行,工程师以为自己设置生效。

**涉及需求**:FR-14

### 场景 A.5 Ultralytics 升级后新增训练参数

**场景描述**:Ultralytics 发布新版本,新增一个训练参数 `new_param`。

**期望流程**:

1. 维护者在配置定义的唯一位置添加该字段及其元数据
2. 下次运行模板生成命令时,新模板自动包含该字段及其注释
3. 现有的加载、合并、验证、溯源等公共逻辑无需任何修改即可支持新字段

**涉及需求**:FR-02, NFR-15, NFR-16
