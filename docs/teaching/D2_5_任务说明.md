# D2.5 任务说明:开发 reset_project 项目重置工具

> **课程**: D2.5(D2 课程的补充章节)
> **形式**: 个人任务
> **预计工时**: 4–6 小时
> **前置**: 完成 D2,Python ≥ 3.12
> **交付物**: 一个可工作的 `odp-reset` 命令 + 完整 git 历史

---

## 一、背景:为什么要做这个工具

完成 D2 之后,你会发现一个尴尬的事:**测试 `init_project` 是一份反复劳动**。

每次想再跑一遍 `odp-init` 看效果,你都要手工敲:

```bash
rm -rf data/train data/val data/test
rm -rf runs models/checkpoints apps/platform/logging
# ...或者一刀切 rm -rf,然后忘了自己删过哪些
```

**这套手工流程藏着三个雷**:

1. **删错** — 不小心删到代码、或者删到 `data/raw/` 里你辛苦标注的数据
2. **删漏** — 下次 `odp-init` 提示"目录已存在",白跑一次
3. **删过的事没记录** — 半年后回头查"我那批训练结果哪去了",根本没线索

更要命的是教学场景:学员想再撞一次某个墙(例如重做 D2 的某个练习),没有一个干净的回滚点。

**所以我们需要一个工具**——把 `init_project` 创建的所有运行时产物可控地、安全地、留痕地清掉,让项目回到"刚 clone"的状态。它应该和 `odp-init` 是一对对称的兄弟命令:`odp-init` 建,`odp-reset` 拆。

## 二、产品目标

交付一个名为 `odp-reset` 的 CLI 命令,满足:

- **能正确清理**:删除所有运行时产物目录(训练划分、checkpoints、runs、业务日志)
- **绝对安全**:对源代码、文档、配置、原始数据、`.git` 等任何 git 追踪的内容**零误伤**
- **用户透明**:默认 dry-run,只打印计划,不动文件;真删必须显式确认
- **跨平台**:Linux / macOS / Windows 都能跑,Windows 上的只读文件场景也要处理
- **可观测**:每次执行都留下审计日志,日志本身不会被这个工具自己清掉
- **可追溯**:审计日志能回答"是谁,在什么时间,跑了哪个版本的 reset,带了什么参数"

## 三、功能需求清单

### 3.1 数据层(`paths.py`)

需要新增三组定义:

| 新增内容 | 作用 | 类型 |
|---|---|---|
| `get_dirs_to_reset()` | **白名单** — 显式列出哪些目录可以被清理 | 函数 |
| `PROTECTED_DIRS` + `is_protected()` | **黑名单兜底** — 即便白名单填错,也拦住对 ROOT、`.git`、代码目录的删除 | 常量 + 函数 |
| `META_DIR` / `META_LOGGING_DIR` | 工具自身的元数据目录,与业务数据严格隔离 | 常量 |

**关键设计原则**:

- 白名单 **不是** 从 `get_dirs_to_initialize()` 减出来的——而是**重新写一份**,明确列举"能删什么"。
- `is_protected()` 必须做**双向检查**:既检查路径本身是不是受保护目录,也检查它**是不是某个受保护目录的祖先**(否则给个 `ROOT_DIR` 就能间接删一切)。
- `META_LOGGING_DIR` 必须满足三个条件:
  - **在** `get_dirs_to_initialize()` 中(启动时建出来)
  - **在** `PROTECTED_DIRS` 中(任何 reset 操作都不能动它)
  - **绝对不在** `get_dirs_to_reset()` 中

### 3.2 CLI 层(`cli/reset_project.py`)

主入口 `reset_project(yes, force, dry_run) -> int` + argparse 封装 `main()`。

功能需求:

1. **默认 dry-run** — 不加任何参数时,只扫描 + 打印计划,**不删任何东西**
2. **`--yes`** — 显式同意删除;不加这个就永远是 dry-run
3. **交互式二次确认** — 用户必须**精确输入大写 `RESET`**;输入任何其他内容(包括 `yes` / `y` / 小写 `reset` / 空字符串 / Ctrl+C / EOF)一律取消
4. **`--force`** — 跳过交互确认(给 CI / 自动化用,**仅在 `--yes` 时有效**)
5. **`--dry-run`** — 显式声明 dry-run(默认行为本就如此,这个 flag 是给可读性用的);**与 `--yes` 冲突时,以 `--dry-run` 优先并警告**
6. **扫描阶段** — 遍历 `get_dirs_to_reset()`,统计每个目录的**文件数**和**总大小**;再过一遍 `is_protected()`,拦截受保护目录(应该不会有,但作为运行期防御)
7. **计划展示** — 以表格形式打印每个待删目录的路径、文件数、大小,以及合计;同时声明哪些重要目录(`data/raw/`、`models/pretrained/`、git 追踪的所有内容)**不会被动**
8. **删除阶段** — 逐目录删除,每个目录开始前打印 `[i/N] 删除 xxx`;对**大于 1 GiB** 的目录额外打印一行预警提示用户"这可能需要一会"
9. **错误恢复** — 使用 `shutil.rmtree(path, onerror=_on_rm_error)`,在回调里处理 Windows 上的只读文件(`os.chmod(path, stat.S_IWRITE)` 后重试)
10. **成功失败分开汇总** — 删除完成后分别列出成功的目录数、失败的目录列表 + 失败原因(**不能一个失败就整体退出,会让剩下的目录无法处理**)

### 3.3 审计层(集成在 CLI 中)

每次运行必须记录一段"审计上下文",字段:

- `user` — 当前操作系统用户名(`getpass.getuser()`)
- `pid` — 当前进程号(`os.getpid()`)
- `git_rev` — 当前 git HEAD 的 commit SHA(`git rev-parse HEAD`);非 git 仓库时记 `"(not a git repo)"`
- `argv` — `sys.argv`,即"被怎么调起来的"
- `cwd` — `os.getcwd()`

这段上下文在 `reset_project()` 开头打印到日志(`META_LOGGING_DIR/reset_project/`),并随后续操作一起持久化。

**为什么要这个?** 出事时(例如有人喊"我的 data 没了!"),审计日志是第一份证据。光看到"reset 完成"不够——必须能回答"**谁**,在**哪个版本的代码**上,**怎么**跑的"。这也对应到企业合规体系(SOX / HIPAA / ISO 27001)里对"破坏性操作可追溯"的标准要求。

### 3.4 入口层

#### `scripts/reset_project.py`(脚本入口,薄壳)

```python
import sys
from pathlib import Path
REPO_ROOT = Path(__file__).resolve().parent.parent
sys.path.insert(0, str(REPO_ROOT / "apps" / "platform" / "src"))
from odp_platform.cli.reset_project import main
if __name__ == "__main__":
    sys.exit(main())
```

#### `pyproject.toml`(注册 console_script)

```toml
[project.scripts]
odp-init  = "odp_platform.cli.init_project:initialize_project"
odp-reset = "odp_platform.cli.reset_project:main"   # 新增
```

#### `.gitignore`(排除 meta 目录)

```gitignore
# 工具元数据(reset/doctor 等元工具的审计日志)
.odp-meta/
```

## 五、验收标准(DoD)

提交之前对照下面这份清单**逐项打勾**,缺一项不算完成:

### 功能正确性

- [ ] 在干净环境里跑 `odp-init`,再跑 `odp-reset`(不加参数) — 看到计划表格,**没有任何目录被删**
- [ ] 跑 `odp-reset --yes`,看到二次确认提示,**输入 `y` 取消**,确认没删
- [ ] 跑 `odp-reset --yes`,输入 `RESET` 确认,看到逐目录的 `[i/N]` 删除日志,完成后所有训练划分、runs、checkpoints、业务日志被清掉
- [ ] 跑 `odp-reset --yes --force`,无需交互直接完成
- [ ] `odp-reset --dry-run --yes` 应该给出冲突警告,并以 dry-run 为准
- [ ] 重置完成后 `data/raw/` 完好无损 — 把它当作"用户标注数据"测一遍
- [ ] 重置完成后 `git status` 输出干净 — **没有任何 git 追踪的文件被改动**

### 安全机制

- [ ] 故意往 `get_dirs_to_reset()` 加 `ROOT_DIR`,跑 `odp-reset --yes --force` — `is_protected()` 应该拦下并打印"⛔ 拒绝处理受保护目录",**项目根目录完好**
- [ ] 故意往 `get_dirs_to_reset()` 加 `DOCS_DIR`,同上验证
- [ ] 故意往 `get_dirs_to_reset()` 加 `META_LOGGING_DIR`,同上验证(双重保护:既是黑名单,又不在白名单)

### 审计追踪

- [ ] 跑过任意一次 reset 后,检查 `.odp-meta/logs/reset_project/` 存在,里面有 `.log` 文件
- [ ] 日志里能看到 `user / pid / git_rev / argv / cwd` 五个字段
- [ ] 连续跑两次 reset,**第二次没有清掉第一次的日志**(审计追踪不可被自身销毁)
- [ ] `.odp-meta/` 被 `.gitignore` 忽略,`git status` 不应该看到它

### 工程化

- [ ] `pip install -e ./apps/platform --force-reinstall --no-deps` 之后,`odp-reset --help` 能跑
- [ ] `odp-reset` 和 `python scripts/reset_project.py` 行为完全一致

### 跨平台

- [ ] 在你的主开发平台跑通所有测试
- [ ] 至少在**另一个平台**(VM / WSL / 同学的机器)上跑通 `odp-reset --yes --force` 的完整流程

## 六、提交规范

按"做完一件完整的、安全可发布的事就 commit 一次"的节奏拆 commit,**禁止把所有改动堆到一个 commit**。

参考节奏(可微调,但 commit 数不应 < 4):

| Commit | 内容 |
|---|---|
| 1 | `feat(common): paths.py 加 reset 目标清单 + 保护机制` |
| 2 | `feat(common): paths.py 加 META_DIR 存放工具元数据` |
| 3 | `chore: 把 .odp-meta/ 加入 gitignore` |
| 4 | `feat(cli): 新建 reset_project.py(默认安全 + 审计上下文)` |
| 5 | `feat(platform): 注册 odp-reset 控制台命令` |

**判断标准**: 在每一次 commit 之前问自己——"现在 checkout 这个 commit 出来,会不会让任何人出事?"如果会,就还不能 commit。这条标准对带有 `delete` / `remove` / `clean` / `reset` 字样的工具尤为重要。

Commit message 用 conventional commits 格式(`type(scope): 简述`),body 至少写清楚**为什么这么改**——不只是"加了 xx 函数",而是"为了 xx 场景"。

## 九、提示与陷阱

- **不要复制 D2.5 教程里 v0 / v1 / v2 / v3 的代码** — 那几版是教学用的"反面教材",直接照搬会通过部分测试但漏掉审计、漏掉保护。**只对照本任务说明的需求写代码**。
- **`is_protected()` 的"祖先检查"很容易漏** — 自测时一定要把 `ROOT_DIR` 加进白名单试一次。
- **入口脚本很容易写错** — `scripts/reset_project.py` 必须调 `main()`(走 argparse),不能直接调 `reset_project()`,否则所有 flag 都拿不到。
- **改了 `pyproject.toml` 必须重装** — `pip install -e ./apps/platform --force-reinstall --no-deps`,否则 `odp-reset` 命令不会注册到 PATH。
- **日志 base_path 必须是 `META_LOGGING_DIR`** — 如果手抖写成 `LOGGING_DIR`,reset 会把自己的日志一并删掉,Windows 上直接报 WinError 32,Linux 上则静默丢日志。这是这次任务最容易踩的雷。
