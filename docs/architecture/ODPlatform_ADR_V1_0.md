>
> 版本:V1.0 | 适用阶段:实训全周期及后续维护 | 文档性质:架构决策记录(Architecture Decision Records)

## 文档定位与使用说明

### 这份文档是什么?

这是 ODPlatform 项目所有**架构决策的官方档案**。每一份 ADR(Architecture Decision Record)记录一项重大工程决策的**背景、选项、决策、理由、后果**——既是历史档案,也是未来维护的依据。

```
┌─────────────────────────────────────────────────────────────┐
│  这份 ADR 集合回答的问题                                      │
│                                                              │
│   ─ 我们当初为什么这么决定?                                  │
│   ─ 当时考虑过哪些备选方案?为什么排除了它们?                │
│   ─ 这个决策付出了什么代价?                                  │
│   ─ 什么情况下我们应该重新评估这个决策?                      │
│                                                              │
│   这些问题如果不写下来,半年后就没人记得;                    │
│   一年后所有人都会按"现状"理解,失去回退能力。               │
└─────────────────────────────────────────────────────────────┘
```

### 这份文档与《项目结构设计指南》的关系

| 文档 | 性质 | 关注点 |
|---|---|---|
| **《ODPlatform 项目结构设计指南》** | 设计纲领 + 教学讲义 | "WHY + WHAT + HOW"——为什么做、做什么样、怎么落地 |
| **本文档(ADR 集合)** | 决策档案 + 维护手册 | "WHEN + WHY + WHAT-IF"——什么时候决定、为什么这么决定、未来出现什么情况要重新评估 |

两份文档**互为补充**:

- 《结构设计指南》是给**学习者和新人**的——通过它建立对项目的整体理解
- **ADR 集合**是给**维护者和决策者**的——它是项目长期演进过程中"对话的依据"

如果一份决策**只**写在指南里,它会被理解成"现状的一部分"——读者不会去质疑;但如果它被写进 ADR,读者会自然意识到"这是一次有备选方案的选择,我可以重新评估它"。这是 ADR 制度真正的价值所在。

### 谁应该读这份文档?

| 读者 | 读哪部分 | 阅读时机 |
|---|---|---|
| **讲师** | 全文。ADR-012 是讲师专属的教学策略决策 | 备课、课后复盘 |
| **学员**(进阶阶段) | 第一部分(ADR 制度)+ 第二部分(挑选感兴趣的具体决策) | 实训中后段(D5+),建立工程文化感知 |
| **未来贡献者 / 接手者** | 全文 | 接手项目第一周内 |
| **其他项目的工程师**(取经) | 第一部分(可迁移到任何项目)+ 选读特定 ADR | 自己项目需要建立 ADR 制度时 |

### 阅读建议

第一次读这份文档时,**不要试图记住每一份 ADR 的细节**——而是关注**"为什么要写 ADR"以及"一份好的 ADR 长什么样"**。后续在工作中遇到具体决策点时,再回头翻对应的 ADR。

ADR 不是"读完就归档"的文档,**它是项目长期演进过程中反复回看的对话伙伴**——每次重大重构、版本升级、新人 onboarding 都会回到这里找答案。

# 第一部分:ADR 制度本身

> 这部分讲的不是 ODPlatform 的具体决策,而是**"如何写决策"本身**——是关于决策的元决策。任何一个工程项目想要建立 ADR 制度,这部分都可以直接套用。

## 1. 什么是 ADR?

### 1.1 ADR 的起源

ADR(**A**rchitecture **D**ecision **R**ecord,架构决策记录)的概念由 Michael Nygard 在 2011 年的一篇博客《Documenting Architecture Decisions》中正式提出。它的核心动机非常具体:

> 在项目演进了几年之后,**没人记得当初为什么这么做了**——团队成员有的离职、有的换岗,文档散落各处,有的过期、有的从未存在。新加入的人面对一个怪异的设计选择,只能猜测"也许有什么历史原因",然后要么沿用、要么推翻——但不管选哪条路,**都没有真正的依据**。

ADR 的作用,就是把"当时为什么这么决定"用**结构化、可查阅、版本化**的方式保存下来,让任何未来的人都能回溯决策的完整上下文。

### 1.2 ADR 解决的具体问题

没有 ADR 的项目,通常会出现以下症状:

| 症状 | 没有 ADR 时的表现 |
|---|---|
| **历史决策无法追溯** | "这个目录为什么这样组织?"——"我也不知道,反正一直是这样" |
| **重复犯错** | 三年前否定过的方案,被新人重新提出来,讨论一遍,再被否定一次 |
| **新人 onboarding 困难** | 新人需要"野生口耳相传"才能理解项目的潜在约束 |
| **重构时缺乏依据** | 想改某个设计,但没人能说清楚"原本不这么做的理由是什么" |
| **跨团队协作冲突** | 不同团队对"为什么这样设计"理解不一致,产生潜在矛盾 |
| **"权威专家"瓶颈** | 项目重大决策依赖某一两个老员工的记忆,他们一离开项目就失去判断能力 |

ADR 把所有这些隐性知识**外化为文档**——它的存在让项目从"靠人记"升级为"靠制度记"。

### 1.3 ADR 的核心价值

ADR 真正的价值,**不是写完后归档供查**,而是它的**写作过程本身就是质量保证**:

```
┌───────────────────────────────────────────────────┐
│  写 ADR 这个动作,会逼你自己想清楚:                │
│                                                    │
│   ─ 当前到底有哪几个备选方案?                     │
│   ─ 每个方案的真实代价是什么?                     │
│   ─ 为什么我倾向于其中一个?                       │
│   ─ 这个决策可逆吗?如果不可逆,我准备好了吗?     │
│   ─ 什么样的未来情况会让我推翻这个决定?           │
│                                                    │
│  90% 的工程错误,起源于"没把这些问题想清楚就动手"。│
│  ADR 的写作本身,就是把这些问题摆到台面上的过程。  │
└───────────────────────────────────────────────────┘
```

> 💡 **一个值得记住的观点**:
>
> **"写不出 ADR 的决策,通常是没想清楚的决策。"**
>
> 如果你发现自己"写 ADR 比做决策还累"——那不是 ADR 制度有问题,而是你**还没真正做完那个决策**。继续想下去,直到能写出清晰的 ADR——这才是真正的"想清楚"。

### 1.4 ADR 不是什么

为了避免误解,也明确说一下 ADR **不**是什么:

- **不是项目计划**——它不规划"未来要做什么",只记录"当前已经决定了什么"
- **不是会议纪要**——它不记录讨论过程,只记录讨论结论与依据
- **不是技术教程**——它不教你"怎么用 Pydantic v2",只记录"我们为什么选择 Pydantic v2"
- **不是承诺书**——它不阻止未来变更,反而是为变更准备好"上一次的判断依据"
- **不是个人观点**——它代表团队/项目的官方决策,需要通过评审进入 Accepted 状态

把这些边界划清楚,ADR 才不会变成无所不包的"杂物文档"。

## 2. ADR 的状态机

每一份 ADR 在它的生命周期内,会经历不同的状态。这些状态构成一个清晰的**决策状态机**:

```
┌─────────────┐
│  Proposed   │ ← 提出阶段:已起草,等待评审
└──────┬──────┘
       │ 评审通过
       ▼
┌─────────────┐
│  Accepted   │ ← 采纳阶段:决策生效,代码按此执行
└──────┬──────┘
       │
       ├──────► 时过境迁,有更好的方案 ─► ┌────────────┐
       │                                   │ Superseded │
       │                                   └────────────┘
       │                                   (被 ADR-XXX 替代)
       │
       └──────► 决策失效,且无替代 ─────► ┌────────────┐
                                          │ Deprecated │
                                          └────────────┘
                                          (废弃,不再适用)
```

### 2.1 各状态的含义与触发条件

| 状态 | 含义 | 触发条件 | 文档处理 |
|---|---|---|---|
| **Proposed** | 已起草,等待评审 | 任何人发起新决策 | 加入 `docs/architecture/` 目录,索引中标记 [Proposed] |
| **Accepted** | 决策生效,代码按此执行 | 评审通过(由架构维护者签字) | 索引中标记 [Accepted] + 日期 |
| **Superseded** | 被新的 ADR 替代 | 有新 ADR 明确表示"取代 ADR-XXX" | 旧 ADR 不删除,在头部加 "Superseded by ADR-YYY",新 ADR 中"背景"说明取代关系 |
| **Deprecated** | 决策失效但无替代 | 项目方向变化使原决策不再适用 | 头部加 "Deprecated as of YYYY-MM-DD",说明失效原因 |

### 2.2 关键规则:已采纳的 ADR 永不删除

ADR 文档**只追加,不删除**——即使一份 ADR 被废弃或被替代,它的物理文件也保留。原因有三:

1. **审计需要**——重大事故复盘时,可能需要回溯"当时为什么这么决定"
2. **新人理解需要**——某些"看起来奇怪"的代码,可能是历史 ADR 的产物。能查到原文,新人才能真正理解
3. **避免重复犯错**——三年前否定过的方案,如果新人不知道,可能再次提出。保留旧 ADR 让"否定的理由"也能被检索到

### 2.3 状态变更需要新 ADR

值得强调:**状态变更本身需要写一份新的 ADR**。

例如:
- 想把 ADR-005(数据格式插件化)从 Accepted 改成 Superseded —— 不要直接改 ADR-005 的状态。**写一份新的 ADR-013(新的数据格式策略),在其中说明取代 ADR-005**;然后在 ADR-005 的头部加一句 "Superseded by ADR-013"。

这条规则保证了**每一次状态变更都有完整的决策依据**,不会出现"某天某个 ADR 莫名其妙被改了状态,没人记得为什么"的尴尬。

## 3. ADR 的写作模板

### 3.1 标准字段

ODPlatform 采用以下标准字段:

| 字段 | 必填? | 说明 |
|---|---|---|
| **ID** | ✅ | ADR-XXX,数字递增,永不复用 |
| **标题** | ✅ | 动词开头,简短描述决策内容 |
| **状态** | ✅ | Proposed / Accepted / Superseded / Deprecated |
| **日期** | ✅ | 进入当前状态的日期 |
| **决策者** | ✅ | 主要决策者(可以是个人或团队) |
| **受影响范围** | ✅ | 决策影响哪些模块/文件/角色 |
| **背景** | ✅ | 为什么需要做这个决策?当前面临什么问题或选择? |
| **选项** | ✅ | 至少列出 2 个备选方案,包括最终未采纳的 |
| **决策** | ✅ | 最终采纳的方案 |
| **理由** | ✅ | 为什么选这个,不选其他?要列得具体到能说服未来的人 |
| **后果** | ✅ | 正面后果(预期收益)+ 代价(已知的牺牲) |
| **演进路径** | 选填 | 如果决策需要分阶段落地,记录阶段计划 |
| **何时重新评估** | ✅ | 什么样的未来情况会让我们重新评估这个决策?(防止决策"沉睡") |
| **相关 ADR** | 选填 | 与本决策相关的其他 ADR 编号 |
| **延伸资料** | 选填 | 相关博客、论文、规范、社区讨论 |

### 3.3 关键字段的写作要领

#### 「背景」字段

**好的背景**:讲清楚问题是什么、约束是什么,让读者**自然产生"我也想做这个决策"的代入感**。

**坏的背景**:只说"我们决定做 X"——读者无法判断这个决策是否合理。

#### 「选项」字段

**关键原则:必须列出至少 2 个备选方案,即使你已经知道选哪个**。

如果你只列一个方案,这就不是"决策"——这是"通知"。**真正的决策必然包含权衡**,而权衡需要至少两个对象才能成立。

如果你想不出第二个方案,通常意味着两件事之一:
1. 你对问题理解不够深(继续调研)
2. 这件事根本不需要 ADR(只是个常规实现细节)

#### 「理由」字段

理由要**具体到能说服一个不在场的工程师**。

❌ **空洞的理由**:
- "因为 Monorepo 是当前主流"
- "因为这样更优雅"
- "因为团队倾向于这个方案"

✅ **具体的理由**:
- "团队规模 1-3 人,Monorepo 的复杂度收益比 Multi-repo 高 4 倍(具体:CI 维护减少 70%,跨端原子提交价值高)"
- "Web backend 会直接 `import platform.inference`——Multi-repo 下需要把 platform 发到内部 PyPI,版本管理摩擦大"

#### 「何时重新评估」字段

这是 ADR 最容易被忽略、但最重要的字段。**没有"重新评估条件"的决策,会沉睡成"现状",失去自我修正能力**。

好的"重新评估"条件具有:
- **可验证性**——能用客观指标判断是否触发(团队人数、CI 时长、错误率)
- **针对性**——直接对应"如果触发,当前决策就不再合理"
- **数量适中**——通常 3-5 条,太多会冲淡核心,太少会有盲区

## 4. ADR 的写作规范

### 4.1 可读性原则:为"未来读者"写作

ADR 的真正读者**不是当前的你和团队**——是**6 个月后、3 年后、甚至 5 年后**的某个人。这个人:

- 可能不在当时的会议上
- 可能不知道当时的口头共识
- 可能完全没有当时的项目背景
- 可能正在面对一个棘手的重构,**急需理解原本的决策依据**

所以 ADR 写作时要**假设"读者一无所知"**——把背景、约束、术语、上下文都写清楚。

```
❌ 假设读者懂的写法:
"考虑到我们的 RSOD 数据流,选择 Monorepo"
→ 6 个月后读者不一定知道"RSOD 数据流"是什么意思

✅ 自包含的写法:
"考虑到我们使用 RSOD 公开数据集(包含飞机/油罐/操场/立交桥四类、
 共 ~1GB)作为本期实训的训练数据,这份数据需要被多个 app 共享访问。
 在这种场景下,选择 Monorepo..."
```

### 4.2 简洁原则:能短就别长

虽然要详细,但**不要废话**。一份 ADR 通常 60-150 行就够了——超过 200 行说明你**可能在一份 ADR 里塞了多个决策**,应该拆分。

| ADR 长度 | 含义 |
|---|---|
| < 30 行 | 太短,可能没把背景和理由说清楚 |
| **60-120 行** | **理想长度** |
| 150-200 行 | 偏长,检查是否有冗余 |
| > 200 行 | 通常应该拆分为多个 ADR |

### 4.3 中立原则:写决策,不写信仰

ADR 应该写**这个项目当前情境下的最佳决策**,而不是**作者的技术偏好**。

```
❌ 个人信仰式:
"FastAPI 是最好的 Python Web 框架,所以我们选 FastAPI"

✅ 项目情境式:
"我们需要 Python Web 框架,候选 Django / Flask / FastAPI。考虑到:
  - 我们有 Pydantic v2 配置体系,FastAPI 原生集成 Pydantic
  - 我们重 IO 操作(模型推理),需要 async 原生支持
  - 团队 1-3 人,Django 的批电池综合症会拖累速度
 因此选 FastAPI..."
```

中立的 ADR 即使作者离开,**新加入的工程师也能继续基于同样的逻辑做后续决策**——这才是制度的价值。

### 4.4 可证伪原则:写"什么情况下我会反悔"

这是最容易被忽略、但最体现专业度的原则——**写下你的决策的"反例"**。

每份 ADR 都应该回答:**"什么样的未来情况,会让我们认为今天这个决策是错的?"**

例子:
- 选 Monorepo 的反例:"如果团队成长到 50 人以上、各端独立发布节奏、CI 总时间 > 30 分钟,就重新评估 Multi-repo"
- 选 Pydantic v2 的反例:"如果 Pydantic v2 在 18 个月内没有在重大项目中被验证,且性能开销超过 5%,就重新评估"

写下反例的好处:
1. 强迫自己**承认决策的边界**,避免认为"这是永恒真理"
2. 给未来的维护者**明确的回溯触发条件**——他们不需要凭感觉判断"是不是该重新评估了"
3. 让决策**自带过期机制**,而不是慢慢失效到没人敢动

> 💡 **一句话总结**:
>
> **"一份不写'什么时候会推翻自己'的决策,是傲慢的;一份写了反例的决策,才真正经得起时间。"**

## 6. ADR 与其他文档的边界

ADR 容易与几种文档混淆。这里把边界划清楚:

| 文档类型 | 关注点 | 示例 |
|---|---|---|
| **ADR** | 为什么这么决定?有哪些备选? | "为什么选 Monorepo 而不是 Multi-repo" |
| **设计文档(Design Doc)** | 系统怎么实现?有哪些组件? | "推理流水线的 4 个阶段如何衔接" |
| **API 文档** | 怎么调用这个函数/接口? | `train(config: TrainConfig) -> TrainResult` |
| **用户手册** | 怎么使用这个工具/系统? | "如何运行 yolo_train.py" |
| **教程 / Tutorial** | 怎么从零开始上手? | "30 分钟用 ODPlatform 跑通第一个训练" |
| **CHANGELOG** | 这个版本变更了什么? | "v1.2.0: 新增 RT-DETR 支持" |

### 6.1 关键区分:ADR vs 设计文档

这两者最容易混淆。简单的判断标准:

```
如果我把"选项"和"理由"删掉,文档还成立吗?
  ─ 仍然成立 → 它是设计文档(只描述"是什么")
  ─ 不成立   → 它是 ADR(本质是"为什么这么选")
```

**ADR 的灵魂是「权衡」,设计文档的灵魂是「构造」**。两者可以互相引用——例如 ADR-008 决定"推理流水线异步化",对应的设计文档详细描述"4 个阶段、2 个队列、3 个线程"的具体结构。

### 6.2 不要把 ADR 写成"产品说明书"

一份常见的反模式是:**ADR 里大量描写"系统是什么",却没有讲"为什么这样选"**。

```
❌ 反模式 ADR(写成产品说明书):
"## 推理流水线
 我们的推理流水线由 4 个阶段组成:
 1. 帧采集
 2. 模型推理
 3. 结果绘制
 4. 显示输出
 各阶段通过队列连接..."
 
✅ 正确 ADR(写成决策档案):
"## 决策
 推理流水线采用 4 阶段异步队列模式。
 
 ## 选项
 - 选项 A:同步串行(简单但 GPU 利用率低)
 - 选项 B:异步队列流水线(复杂但充分利用资源)
 - 选项 C:多进程独立处理(扩展性好但通信开销大)
 
 ## 理由
 1. 业务需要 ≥80 FPS,选项 A 实测 52 FPS 不达标
 2. 单机部署不需要选项 C 的多进程开销
 3. 异步队列能让 GPU 在 CPU 处理上一帧时不空转..."
```

**记住:ADR 的价值在于权衡,不在于描述。**

## ADR-001:仓库布局策略

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 + 架构组 |
| 受影响范围 | 整个项目(代码组织、CI、协作流程) |
| 相关 ADR | ADR-002(路径定位)、ADR-003(命名)、ADR-010(工程化基线) |

### 背景

ODPlatform 当前为单一 Python 项目(通用目标检测平台,本期实训数据集为 RSOD)。SRS 文档预期未来扩展为多端系统:
- V1.1:Web 后端 + Web 前端管理界面
- V1.x:可能扩展到其他视觉任务(分类、分割等)
- V2.0:分布式训练、桌面端

需要在项目早期决定**代码组织方式**,避免未来扩展时大规模重构。代码组织方式不仅影响开发效率,也直接决定了 CI/CD 设计、协作模式、依赖管理等所有后续工程决策。

**核心矛盾**:今天只有 1 个端,但未来可能有多个端。怎样的设计**既能满足今天的简洁,又能容纳未来的扩展**?

### 选项

#### 选项 A:Multi-repo(每个端独立 Git 仓库)
每个 app(platform、web-backend、web-frontend、desktop)独立维护 Git 仓库,通过 PyPI 包发布共享代码。

- **优点**:各端发布独立、权限隔离、CI 简单
- **缺点**:跨仓库改动摩擦大、共享代码必须先发包、版本管理复杂、新人 onboarding 需要 clone 多个仓库
- **代价**:每次跨端改动需要至少 7 步操作(改代码 → 写 changelog → 打 tag → 发包 → 改下游版本号 → reinstall → test)

#### 选项 B:Monorepo(所有端在同一仓库,通过 workspace 共享)
所有 app 在同一 Git 仓库,通过 uv workspace 管理依赖,各 app 之间通过本地路径直接引用。

- **优点**:跨端原子提交、统一工具链、共享资产单一来源、新人一次 clone 看全貌
- **缺点**:CI 需要 path filter 优化、大文件需要 Git LFS、所有人共享同一份历史
- **代价**:学习 workspace 配置(约 20 行 toml)、CI 优化的初始投入

### 决策

**采纳选项 B(Monorepo)**,仓库保持当前命名 `ODPlatform`。

### 理由

1. **团队规模匹配**:本项目团队规模 1-3 人(教学场景下扩展到几组学员),完全在 Monorepo 的"高收益区间"——团队规模 < 50 人时,Monorepo 几乎总是更优选择。
2. **物理依赖关系**:未来 web-backend 会**直接 `import platform.inference`**——这是物理依赖。Multi-repo 下需要把 platform 发到内部 PyPI,每次接口改动都要走"发布-升级"流程,持续制造摩擦。
3. **共享资产难管理**:RSOD 数据集 ~1GB、模型权重 ~MB-GB 级、训练 runs 海量产物——这些资产**只能存在一份**。Multi-repo 下要么重复存储,要么找复杂的共享机制。Monorepo 下放在仓库根的 `data/` 和 `models/` 即可。
4. **教学价值清晰**:学员一次 clone 看到完整产品全貌——这种"产品全景视图"对建立工程认知有显著价值。
5. **跨端原子变更**:未来一次 PR 可能同时改 backend API + frontend 调用 + 共享 schema——Monorepo 下这是一个原子提交,Multi-repo 下需要协调三个 PR 的合并顺序。
6. **业界最佳实践参考**:Google、Meta、字节跳动、Anthropic 都使用 Monorepo,且团队规模都在百千人级。我们 1-3 人的项目不存在"规模超出 Monorepo 适用区间"的担忧。

### 后果

#### 正面后果
- 跨端原子 PR 成为可能
- 统一工具链(全仓一套 ruff / pytest / mypy)
- 共享资产单一来源,无重复存储
- 新人 onboarding 一次 clone 全貌
- 重构跨多个 app 时能保证原子性

#### 代价
- 需要学习 uv workspace 概念(约 20 行 toml 配置)
- CI 必须配置 path filter,否则改一个 app 会触发全仓 CI
- 大文件(数据集、模型权重)需要 Git LFS 或外部存储
- 对外发布(独立包到 PyPI)需要额外脚本支持

### 演进路径

```
Stage 0(实训前):marker file 就位,目录暂保持平铺
Stage 1(实训后):重构为 src/ 布局 + apps/platform/ 结构(可选)
Stage 2(V1.1):新增 apps/web-{backend,frontend}/
Stage 3(V2.0):新增 apps/desktop/、扩展多任务
```

### 何时重新评估

如果出现以下任一情况,应重新评估是否切换到 Multi-repo:

1. **团队规模 > 50 人**,且各端由完全独立的团队负责
2. **各端发布节奏完全不同**(例如平台每周发,Web 半年发)
3. **CI 总时间持续 > 30 分钟**,path filter 已无法优化
4. **仓库总大小 > 10GB** 且 Git LFS 解决不了
5. **某个 app 需要独立的安全/合规边界**(例如医疗合规要求隔离)

V1.x 阶段命中 0/5 → Monorepo 是当前的正确选择。

- Google Engineering Practices: "Why Google Stores Billions of Lines of Code in a Single Repository"(CACM 2016)
- Meta Engineering: "Scaling Mercurial at Facebook"
- Nx / Turborepo / uv workspace 等 Monorepo 工具生态

## ADR-002:路径定位策略

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 + 架构组 |
| 受影响范围 | `common/paths.py` 及所有依赖路径定位的代码 |
| 相关 ADR | ADR-001(仓库布局)、ADR-010(工程化基线) |

### 背景

任何"读数据、写日志、加载配置、保存 checkpoint"的代码,都需要知道**项目根在哪里**。ODPlatform 在初期使用了 `Path(__file__).resolve().parent.parent.parent` 这种"数 parent 数量"的方式定位项目根。

随着项目演进(尤其是未来要迁移到 Monorepo `apps/platform/src/odp_platform/common/paths.py` 的位置后),`.parent` 的层级数会变化——任何文件移动都会让 `paths.py` 失效。

需要一个**对文件位置无敏感**的路径定位方案。

### 选项

#### 选项 0:硬编码绝对路径(立刻排除)
```python
ROOT_DIR = "/home/user/projects/ODPlatform"
```
跨机器立崩,不予考虑。

#### 选项 1:`Path(__file__).parent.parent.parent`
基于当前文件位置数 parent 层级。
- **优点**:简单、零依赖
- **缺点**:文件挪一层即失效;迁移到 Monorepo 后 parent 计数变化;无法 mock 测试;视觉不直观

#### 选项 2:Marker File 向上查找
仓库根放一个空文件 `.odp-workspace`,paths.py 从当前文件位置向上查找直到找到这个文件。
- **优点**:文件挪到任意层级都工作;与 git/pytest/poetry/ruff 等工业工具的同款机制对齐;便于单元测试
- **缺点**:仓库根多一个看似无用的空文件;新人需要 5 分钟理解 marker file 概念

#### 选项 3:环境变量优先
`os.environ.get("ODP_WORKSPACE_ROOT")`。
- **优点**:容器/生产部署友好
- **缺点**:开发期每个开发者要自己配置环境变量,容易遗忘;不能单独使用

### 决策

**采纳选项 2(Marker File)作为主方案,叠加选项 3(环境变量)作为生产部署的覆盖**。

具体实现:
- 仓库根放空文件 `.odp-workspace` 作为 marker
- `paths.py` 从当前文件位置向上递归查找 marker
- 容器/生产环境可设置 `ODP_WORKSPACE_ROOT` 环境变量,优先级高于 marker file

### 理由

1. **文件位置无关**:文件挪到任意目录都能工作——这是 ODPlatform 未来 Stage 1 重构(目录结构调整)的关键保障。如果用 `parent.parent.parent`,Stage 1 重构需要改一次 `paths.py`,Stage 2 再改一次,每次都要担心数错。
2. **工业标准对齐**:Git 用 `.git/`、pytest 用 `pytest.ini`、poetry 用 `pyproject.toml`、Ruff 用 `pyproject.toml` 或 `ruff.toml`、Bazel 用 `WORKSPACE`、Yarn/pnpm 用 `package.json`、Cargo 用 `Cargo.toml`——**所有现代工程工具都用 marker file 模式**。这不是巧合,是因为它真正解决了路径定位的所有实际难题。
3. **可测试性**:单元测试中可以通过临时目录 + 创建/删除 marker 文件来 mock 不同的工作区根。`parent.parent.parent` 方案下要 mock 路径定位需要 monkey patch 模块属性,非常脆弱。
4. **支持"两个根"区分**:Monorepo 之后会自然出现两种根——WORKSPACE_ROOT(整个 monorepo)和 PROJECT_ROOT(当前 app)。Marker file 模式可以同时支持(WORKSPACE_ROOT 用 `.odp-workspace`,PROJECT_ROOT 用最近的 `pyproject.toml`)。
5. **环境变量叠加保留生产部署灵活性**:容器化部署时 marker file 可能因挂载路径不同而难以定位,环境变量提供了"逃生口"。两者优先级关系明确(env 优先于 marker)。

### 后果

#### 正面后果
- 路径定位代码极其健壮,任何文件挪动都不会让它失效
- 跨开发/CI/生产环境一致(开发用 marker file,容器用环境变量)
- 与业界主流工具的工程认知一致,新人不需要学一套陌生约定
- 单元测试容易写,fixtures 友好

#### 代价
- 仓库根多一个空文件 `.odp-workspace`(对项目无任何运行时影响)
- 新加入的工程师需要花 5 分钟理解 marker file 概念
- 极端情况下,如果有人在嵌套子目录里手动创建了同名 marker,会有定位歧义(实际几乎不会发生)

### 演进路径

```
现在:仓库根创建 .odp-workspace 空文件,paths.py 切换为 marker 查找模式
未来:容器化部署时,Dockerfile 设置 ODP_WORKSPACE_ROOT 环境变量
长期:维持现状,这个方案对项目结构变化天然鲁棒
```

### 何时重新评估

1. 如果出现"嵌套 marker 误识别"的 bug(实际未见过案例)
2. 如果 Python 标准库或 importlib 提供了更优雅的"项目根定位"原生支持
3. 如果项目演进到使用某种打包形态(zipapp / pyinstaller),marker file 不再可访问

V1.x 阶段命中 0/3 → Marker file 是当前的正确选择。

### 延伸资料

- pytest 的 rootdir 发现机制(`pytest --rootdir`)
- Cargo 的 `WORKSPACE` 概念(Rust 生态)
- Bazel 的 `WORKSPACE.bzl` 设计

## ADR-003:命名约定

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | 整个项目(目录、包名、命令、文件、变量) |
| 相关 ADR | ADR-001(仓库布局)、ADR-002(路径定位) |

### 背景

ODPlatform 项目对外名称为 ODPlatform(Object Detection Platform),需要在第一天就确定**完整的三层命名约定**(产品名 / 仓库名 / Python 包名)以及配套的 marker 文件、CLI 命令前缀、环境变量前缀等命名细节。

如果不在早期统一,各模块开发者会自发产生不同的命名习惯——半年后会出现 `odp_*`、`odplatform_*`、`od_*`、`platform_*` 多种前缀混用的局面,纠正成本极高。

### 选项

#### 选项 A:不约定,各模块自由
每个开发者自己选命名风格,最后再统一。

- **优点**:早期省事
- **缺点**:必然产生不一致,后期纠正代价巨大;影响代码可读性与品牌一致性

#### 选项 B:简单约定(只定 Python 包名前缀)
只规定 Python 包名前缀,其他不约束。

- **优点**:轻量
- **缺点**:仓库名、CLI 命令、环境变量等仍可能混乱

#### 选项 C:完整三层命名 + 全套配套约定
明确产品名 / 仓库名 / 组件名 / Python 包名 / Marker 文件 / CLI 前缀 / 环境变量前缀的完整规范。

- **优点**:命名一致性高,品牌识别度强,可隔离性好,可演进性好
- **缺点**:文档稍长,需要新人理解一次

### 决策

**采纳选项 C(完整三层命名 + 全套配套约定)**。

具体方案:

| 层次 | 名字 | 用在哪 |
|---|---|---|
| **产品名** | ODPlatform | README 标题、对外文档、PPT |
| **仓库名** | ODPlatform(保持当前命名) | GitHub/GitLab 仓库地址 |
| **组件名(apps)** | platform / web-backend / web-frontend / desktop | `apps/` 下的子目录(未来) |
| **组件名(packages)** | shared-schemas / shared-utils 等 | `packages/` 下的子目录(未来) |
| **Python 包名前缀** | `odp_` | `odp_platform`、`odp_schemas`、`odp_web_backend` |
| **Marker 文件** | `.odp-workspace` | 仓库根的空文件,paths.py 用它定位 |
| **CLI 命令前缀(未来 entry-points)** | `odp-` | `odp-train`、`odp-infer` 等 |
| **CLI 脚本(本期实训)** | `yolo_*.py` | 实训方案规定,保留 |
| **环境变量前缀** | `ODP_` | `ODP_LOG_LEVEL`、`ODP_WORKSPACE_ROOT` |

### 理由

1. **避免标准库冲突**:Python 包名加 `odp_` 前缀,避免与标准库 `platform` 冲突。如果包就叫 `platform`,`import platform` 会产生歧义,且 IDE 跳转可能跳到错的地方。
2. **品牌一致性**:`odp_*`、`odp-*`、`ODP_*` 这套统一前缀让任何工程师在阅读代码、查日志、看进程时,看到 odp-/ODP_ 立刻知道是 ODPlatform 的产物。多项目共存的服务器或开发环境里,这种视觉标识符能省下大量定位问题的时间。
3. **可隔离性**:环境变量加 `ODP_` 前缀避免与其他工具/项目冲突;`.odp-workspace` marker 避免与其他 monorepo 工具(nx、turborepo)的 marker 文件冲突。
4. **可演进性**:如果未来某个组件要独立发布(例如 `odp_schemas` 发到内部 PyPI 给其他项目用),命名前缀让"独立发布"几乎零成本——不需要改包名、不需要担心冲突。
5. **业界标准做法**:google.protobuf、django.contrib、azure.storage、aws.* 都是"组织前缀 + 模块名"的命名模式。这不是为了好看,是工业界经过大量项目验证下来的实践。
6. **产品名和组件名分开管理**:产品名(ODPlatform)是品牌,对外;组件名(platform / web-backend)是内部模块,对开发者。多个组件共同构成同一个产品身份。

### 后果

#### 正面后果
- 命名一致性高,品牌识别度强
- 与业界主流工具命名习惯对齐
- 命名约定从第一天起就到位,无需后期统一改造
- 未来扩展时,所有新模块只要遵循约定即可

#### 代价
- 文档稍长,新人需要理解一次三层命名层次
- `odp_` 前缀让 import 语句略长几个字符(但极短,几乎不影响)

### 关于本期实训保留 `yolo_*.py` 命名的说明

实训方案中,入口脚本命名为 `yolo_trans.py`、`yolo_train.py`、`yolo_val.py`、`yolo_infer.py`。这是本期实训的显式约定,理由:

- 让学员一眼看出"这些是项目的命令行入口"
- 与 Ultralytics YOLO 这个核心算法栈保持视觉关联
- 实训方案文档已经写定,不变更

未来如果项目要支持除 YOLO 之外的检测算法栈(RT-DETR、DINO 等),CLI 命名约定可以演进为 `odp-train`(算法无关),通过 `pyproject.toml` 的 entry-points 实现——但这是本期实训之后的事。

### 关于未来扩展的预留

ODPlatform 的核心定位是**通用目标检测开发平台**。如果未来业务需要扩展到目标检测之外的视觉任务(分类、分割等),届时再视情况讨论是否需要新的上位品牌——**不在当前阶段决策**。

当前阶段的关键判断是:**不预先建设、不过早抽象**。命名约定能容纳"目标检测"这个核心范畴的所有合理扩展,这就够了。

### 何时重新评估

1. 如果项目方向发生根本变化(从"目标检测平台"扩展到"全视觉 AI 平台"),需要重新评估是否启用上位品牌
2. 如果 `odp_` 前缀与某个常用第三方库发生冲突(目前未发现)
3. 如果团队成长到需要更精细的命名空间隔离

V1.x 阶段命中 0/3 → 当前命名约定是合适的。

### 延伸资料

- PEP 8 - Style Guide for Python Code(包名规范)
- 《项目结构设计指南》第二部分(命名作为架构决策)

## ADR-004:配置系统设计

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `config/` 模块及所有需要参数配置的子模块 |
| 相关 ADR | ADR-007(训练编排)、ADR-008(推理流水线) |

### 背景

视觉 AI 项目的"四大顽疾"之一是**配置散落**——超参数硬编码在代码里、YAML 文件散落各处、CLI 参数随意添加。其后果是**实验无法精确复现**,昨天 mAP 0.85 的模型今天怎么也跑不出来。

ODPlatform 需要一套**配置系统**作为整个平台的中枢——它需要回答几个具体问题:

- 配置写在哪?代码里?CLI 参数?YAML 文件?三者如何组合?
- 配置的类型如何保证?用户传错类型会发生什么?
- 实验复现时,如何精确知道"当时的所有参数是什么"?
- 默认值放哪里?如何被用户覆盖?

### 选项

#### 选项 A:argparse + 字典
传统 Python 方式,argparse 接收 CLI 参数,转成字典传递。
- **缺点**:无类型保证;参数变化时要改多处;复杂嵌套配置笨拙

#### 选项 B:Hydra + OmegaConf
Facebook 出品的配置框架,支持 YAML 组合和命令行覆盖。
- **优点**:成熟、社区大
- **缺点**:学习成本高;运行时类型检查;过于灵活反而难维护

#### 选项 C:Pydantic v2 + 三源合并
用 Pydantic v2 定义类型化配置模型,实现 DEFAULT → YAML → CLI 三源合并,带来源溯源能力。
- **优点**:类型安全(IDE 提示 + 静态检查);Pydantic 生态成熟;与 FastAPI 等天然兼容
- **缺点**:需要写一些样板代码

### 决策

**采纳选项 C(Pydantic v2 + 三源合并)**。

具体设计:
- 每类配置都有对应的 Pydantic 模型(`TrainConfig`、`InferConfig`、`ValConfig` 等)
- 三源优先级:**CLI 参数 > YAML 文件 > 代码默认值**
- 每个字段记录其**来源**(`ConfigSource` 枚举:DEFAULT / YAML / CLI),用于实验复现时追溯
- 配置验证在加载时同步完成,错误立即报告

### 理由

1. **类型安全**:Pydantic v2 在加载时立即校验类型——传错类型(例如把 `lr0=0.01` 写成 `lr0="0.01"`)会立刻报错,而不是训练 30 分钟后崩溃。
2. **IDE 友好**:Pydantic 模型有完整类型提示,IDE 自动补全字段名、参数类型,减少配置错误。
3. **可序列化**:Pydantic 模型可序列化为 JSON / YAML,实验记录、checkpoint 元数据、远程任务参数传递都天然支持。
4. **来源溯源是关键创新**:传统配置系统知道"最终值是什么",但不知道"每个值是怎么来的"——是默认值?是 YAML 写的?是 CLI 强制覆盖的?有了来源标记,实验复现时可以精确还原"当时哪些参数是用户主动改了,哪些是默认值"。
5. **生态整合**:未来 V1.1 的 web-backend 用 FastAPI(Pydantic 原生集成),配置体系无缝复用。
6. **性能 OK**:Pydantic v2 是 Rust 实现的核心,验证速度极快,不会成为瓶颈。

### 后果

#### 正面后果
- 实验严格可复现(配置 + 来源 = 完整复现依据)
- IDE 全程类型提示,配置错误前置发现
- 配置可序列化,跨环境传递无障碍
- 与未来 Web 后端无缝整合

#### 代价
- 每类配置需要先定义 Pydantic 模型(20-50 行模型代码)
- 学员需要学习 Pydantic v2 语法(实训 D3 教学内容)

### 何时重新评估

1. 如果 Pydantic 出现重大不向后兼容变更(参考 v1 → v2 的迁移痛苦)
2. 如果配置规模膨胀到 100+ 字段,需要更结构化的组合方式(届时考虑加 Hydra-style 组合)
3. 如果性能分析显示配置加载成为瓶颈(目前不可能)

### 延伸资料

- Pydantic v2 官方文档:https://docs.pydantic.dev
- 《项目结构设计指南》第十二章(配置系统设计)

## ADR-005:数据格式转换的插件化

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `data_converter/` 模块 |
| 相关 ADR | ADR-006(数据验证) |

### 背景

不同数据集采用不同的标注格式:Pascal VOC(XML)、COCO(JSON)、YOLO(TXT)、自定义格式。模型训练通常需要 YOLO 格式作为输入,但**原始数据可能来自任意格式**。

ODPlatform 需要支持**多格式转换**,且未来可能扩展(支持 Open Images、LabelMe 等更多格式)。怎么设计才能"今天支持的格式干净,未来加新格式不动旧代码"?

### 选项

#### 选项 A:If-else 分支
在一个 `convert(format, ...)` 函数里用 if-else 处理所有格式。
- **缺点**:加新格式要改 `convert` 函数;违反开闭原则;测试覆盖率难保证

#### 选项 B:每种格式独立转换器,工厂模式选择
为每种格式实现独立的 Converter 类,通过工厂函数 `get_converter(format)` 选择。
- **优点**:加新格式只需新增类,不动旧代码(开闭原则);易测试;符合策略模式
- **缺点**:稍微复杂一些

#### 选项 C:Pluggy / 第三方插件框架
用 Pluggy(pytest 同款插件机制)做完全可扩展的插件系统。
- **优点**:第三方可发布插件
- **缺点**:本期实训过度设计;学习成本高

### 决策

**采纳选项 B(独立转换器 + 工厂)**。

具体设计:
- `data_converter/core/` 下每种格式一个文件:`pascal_voc.py`、`coco.py`、`yolo.py`
- 共同遵循 `BaseConverter` 接口:`def convert(input_path, output_path, ...) -> ConversionResult`
- `data_converter/service.py` 提供工厂函数 `get_converter(format)`,内部维护格式 → 转换器的映射

### 理由

1. **开闭原则**:加新格式(例如 LabelMe)只需新建 `labelme.py` 并注册到映射,**不动任何旧代码**——这是降低重构风险的关键。
2. **测试隔离**:每种格式可独立单元测试。VOC 转换测试不需要 mock COCO 转换的依赖。
3. **当前实训需要**:本期 RSOD 数据集是 VOC 格式,需要转 YOLO 给训练用;未来可能引入 COCO 数据集——架构要能容纳。
4. **避免过度设计**:Pluggy 是好东西,但本期实训用不到"第三方插件"这个能力。简单工厂+约定接口已经满足"加格式不动旧代码"的核心需求。
5. **可读性**:学员看 `core/pascal_voc.py` 立刻知道这是 VOC 转换实现,文件路径本身就是文档。

### 后果

#### 正面后果
- 加新格式零回归风险
- 单元测试简洁清晰
- 学员易理解(每个文件职责清晰)

#### 代价
- 比单函数 if-else 多一些样板代码
- 需要遵守接口约定(实际上是优点伪装成代价)

### 何时重新评估

1. 如果格式数量膨胀到 10+,且需要第三方贡献新格式 → 考虑 Pluggy
2. 如果某个格式有显著的特殊处理需求,无法纳入统一接口 → 重新设计接口

### 延伸资料

- 《Design Patterns》策略模式
- pluggy 官方文档(未来如有需要)

## ADR-006:数据质量验证策略

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `data_validation/` 模块 |
| 相关 ADR | ADR-005(数据转换) |

### 背景

业界共识:**"模型不行,经常其实是数据不行"**。但传统视觉 AI 项目的数据质量检查通常是**事后人工排查**——训出来效果不好,才回头看数据有没有问题。

ODPlatform 需要一套**主动式数据质量验证体系**——在数据进入训练前,系统化地暴露数据问题:标注缺失、坐标越界、类别不平衡、图像损坏、重复标注等。

### 选项

#### 选项 A:不做数据验证,交给训练时报错
传统做法,训练时数据加载器会报错。
- **缺点**:错误暴露太晚(训练数小时后);错误信息可能误导(以为是模型问题)

#### 选项 B:简单脚本检查,只覆盖少数维度
写几个简单脚本,检查"标签文件存在"、"路径正确"等基础维度。
- **优点**:简单
- **缺点**:覆盖不全,问题可能漏过

#### 选项 C:多维度验证体系 + 隔离清洗 + 可视化报告
完整的数据质量验证体系,包括 validators(规则检查)、analyzers(分布分析)、cleaners(隔离清洗)、visualizers(标注可视化)、reporters(HTML 报告)。
- **优点**:全面、专业、可教学
- **缺点**:开发投入较大

### 决策

**采纳选项 C(完整数据验证体系)**。

具体模块组成:
- `validators.py`:规则检查(标注完整性、坐标合法性、类别有效性)
- `analyzers.py`:统计分析(类别分布、目标尺寸分布、图像分辨率分布)
- `cleaners.py`:隔离清洗(把问题数据隔离到 `quarantine/` 目录,不污染主数据)
- `visualizers.py`:标注可视化(生成带标注的图像,人眼复核)
- `chart_generator.py`:图表生成(分布直方图、热力图)
- `reporters.py`:HTML 质量报告(综合呈现以上结果)

### 理由

1. **数据问题前置**:训练前 5 分钟跑完验证,比训练 5 小时后崩溃强 60 倍。
2. **"隔离清洗"是关键设计**:有问题的数据不直接删除——而是移到 `quarantine/` 目录。这保留了"数据原貌",支持事后人工复核。直接删除是不可逆的工程错误。
3. **可视化对教学价值大**:HTML 报告让学员**眼见为实**地看到"原来这个标注是错的"——比讲师讲一万句"数据要清洗"都有效。
4. **业界对标**:工业界的数据质量平台(如 Label Studio、CVAT、Encord)都包含类似能力,我们做的是其中最核心的子集。
5. **未来可扩展**:`reporters.py` 输出的报告可以集成到 web-backend 作为 Dashboard 展示。

### 后果

#### 正面后果
- 数据问题暴露时间从"训练后"提前到"训练前"
- 学员通过实操学到"数据质量"是工程话题,而非算法话题
- HTML 报告作为教学资产,可复用到后续课程

#### 代价
- 模块本身代码量较大(~500-1000 行)
- 需要约定"质量报告的格式标准"

### 何时重新评估

1. 如果数据集类型扩展到非图像数据(视频、点云),需要新增维度的验证
2. 如果发现现有 6 个维度仍无法覆盖某类常见数据问题
3. 如果性能成为瓶颈(数据集 > 10 万图)

### 延伸资料

- Label Studio / CVAT / Encord 等数据平台的设计参考
- 《Data Quality at Scale》(Google SRE 系列)

## ADR-007:训练编排服务化

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `training/` 模块 |
| 相关 ADR | ADR-004(配置系统) |

### 背景

视觉 AI 训练有"算法层"和"工程层"两类工作:
- **算法层**:神经网络结构、损失函数、优化器(由 Ultralytics YOLO 等算法库提供)
- **工程层**:超参数管理、checkpoint 归档、训练监控、错误处理、产物落盘约定

ODPlatform 应该聚焦工程层,**不重新发明算法**。如何在保持灵活性的同时,把工程层做扎实?

### 选项

#### 选项 A:把 Ultralytics 直接暴露给用户
用户直接调用 `ultralytics.YOLO(...).train(...)`。
- **优点**:零成本
- **缺点**:工程层缺失(超参溯源、归档、监控都得用户自己做);失去平台价值

#### 选项 B:重写训练循环
不用 Ultralytics,自己写 `for epoch in ...` 训练循环。
- **优点**:完全可控
- **缺点**:工程量巨大,与算法库脱离,会落后于业界算法演进

#### 选项 C:服务层包装 + 组合,不改核心
保留 Ultralytics 作为算法核心,在外层加 `TrainService` 服务对象,负责工程层(配置注入、产物归档、状态报告)。
- **优点**:充分利用 Ultralytics 的算法成熟度;工程层独立可演进;算法栈未来可替换(RT-DETR、DINO)
- **缺点**:需要清晰的"哪些是算法层、哪些是工程层"的边界

### 决策

**采纳选项 C(服务层包装)**。

具体设计:
- `training/service.py` 定义 `TrainService` 类
- `TrainService.train(config: TrainConfig) -> TrainResult` 是对外唯一入口
- 内部:配置注入 Ultralytics → 调用 `model.train()` → 收集结果 → 归档产物 → 返回 TrainResult
- TrainResult 包含 checkpoint 路径、metrics、配置快照、运行环境信息

### 理由

1. **聚焦工程层**:把"超参溯源、产物归档、状态报告"这些工程价值集中到一处,与算法核心解耦。
2. **算法可替换**:未来扩展支持 RT-DETR / DINO 时,只需新增 `RTDETRTrainService`,实现同样的接口——`platform` 服务消费方零修改。
3. **测试友好**:`TrainService` 可以 mock(集成测试不需要真训练),也可以小规模真实训练(端到端测试用极小数据集)。
4. **CLI / Web 双入口共用同一服务**:本期实训用 CLI(`yolo_train.py`),未来 Web 用 HTTP API——两者都调用同一个 `TrainService.train()`,业务逻辑零重复。
5. **状态可观测**:服务层是注入日志、监控、错误处理的天然位置。

### 后果

#### 正面后果
- 算法层和工程层职责清晰
- 算法栈可替换,平台价值长期保持
- CLI 与未来 Web 后端共用业务逻辑
- 易测试

#### 代价
- 比"直接调 Ultralytics"多一层封装
- 需要定义并维护 `TrainConfig`、`TrainResult` 等数据契约

### 何时重新评估

1. 如果发现 Ultralytics 提供的能力已经全面覆盖工程层(目前没有迹象)
2. 如果服务层成为性能瓶颈(几乎不可能,训练时间是分钟到小时级)
3. 如果业务需求要求训练循环完全自定义(届时考虑混合架构)

### 延伸资料

- Ultralytics 官方文档
- "Service Layer" 模式(Patterns of Enterprise Application Architecture, Fowler)

## ADR-008:推理流水线异步化

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `inference/` 模块 |
| 相关 ADR | ADR-009(frame_source) |

### 背景

视觉 AI 推理对实时性要求高。简单的"读帧 → 推理 → 绘制 → 显示"串行循环,GPU 在 CPU 处理 IO 时是空转的——典型表现:**单线程推理 52 FPS,GPU 利用率 35%**。

业务场景(实时监控、安全预警)通常要求 ≥ 80 FPS。需要架构上的优化方案。

### 选项

#### 选项 A:同步串行(简单方案)
读帧 → 推理 → 绘制 → 显示,在主线程依次执行。
- **优点**:代码极简,易理解
- **缺点**:GPU 利用率低,达不到实时性要求

#### 选项 B:异步队列流水线
4 个阶段(读帧 / 推理 / 绘制 / 显示)各自独立线程,通过队列连接。
- **优点**:GPU 不空转,充分利用资源;FPS 显著提升
- **缺点**:线程同步复杂;需要处理背压(producer 比 consumer 快时);调试困难

#### 选项 C:多进程独立处理
每个阶段独立进程,通过 multiprocessing.Queue 通信。
- **优点**:绕开 GIL;扩展性好
- **缺点**:进程间通信开销大;启动慢;模型权重不能跨进程共享(需要重复加载)

### 决策

**采纳选项 B(异步队列流水线)作为主方案,选项 A(同步)作为简单场景的备选**。

具体设计:
- `inference/service.py`:同步推理服务(简单场景、单图片推理、单元测试用)
- `inference/pipeline.py`:异步流水线(高 FPS 场景),4 阶段独立线程 + 2-3 个有界队列
- `inference/components.py`:各阶段的具体组件(读帧、推理、绘制、显示)
- 用户可根据场景选择:`infer_service.predict_image(...)` 或 `pipeline.run(source=..., display=True)`

### 理由

1. **业务需要 ≥ 80 FPS**:同步串行实测 52 FPS,达不到。异步流水线把 GPU 利用率从 35% 提到 70-80%,FPS 自然上升。
2. **不需要选项 C 的多进程开销**:本期实训单机部署,不需要跨节点。多进程的代价(启动慢、模型重复加载)不值得。
3. **同步 + 异步双方案并存**:简单场景(单图推理、unit test)用同步避免线程开销;复杂场景(实时视频)用异步流水线。两者通过统一的 InferConfig 配置切换。
4. **背压处理**:有界队列(maxsize=10)在 producer 太快时会自然阻塞,避免内存爆炸。这是工业级流水线的标准做法。
5. **可扩展性**:未来增加新阶段(如"目标跟踪")时,在流水线中插入新组件即可,不影响其他阶段。

### 后果

#### 正面后果
- 高 FPS 场景达到业务要求
- GPU 利用率显著提升
- 简单场景仍然有简单方案可用
- 流水线架构便于未来扩展

#### 代价
- 异步流水线代码复杂度高于同步方案(需要处理线程同步、异常传播、优雅关闭)
- 调试困难(线程错误堆栈不直观)
- 学员需要理解异步队列概念(实训 D9 内容)

### 何时重新评估

1. 如果业务从"单机实时"扩展到"集群处理",需要选项 C(多进程)或更复杂的分布式方案
2. 如果 Python 移除 GIL 后(PEP 703),线程方案的优势进一步放大,无需重新评估;反之如果某种新方案(如 CUDA 流原生支持)出现,可能简化设计
3. 如果发现某个阶段是固定瓶颈(例如绘制阶段),需要重新平衡阶段拆分

### 延伸资料

- 《项目结构设计指南》第十二章(推理流水线设计)
- DeepStream / FFmpeg / GStreamer 等业界流水线框架的设计参考

## ADR-009:frame_source 输入源抽象

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `common/frame_source/` 模块 |
| 相关 ADR | ADR-008(推理流水线) |

### 背景

YOLO(Ultralytics)官方推理接口 `model.predict(source=...)` 看似方便,但其内部 `LoadStreams` 实现存在严重局限:

- 无法配置摄像头分辨率(高速摄像头被强制降到默认分辨率)
- 无法设置帧率(720p / 90fps 摄像头帧率被限制)
- 无法选择视频后端(MSMF / DShow / V4L2)
- 不支持视频文件 seek(无法跳到第 2 小时 30 分)
- 不同输入源(摄像头/视频/图片/文件夹)需要写不同的处理代码

社区流传的 workaround 是**手动改 Ultralytics 源码**——但这一升级版本就丢失,无法长期维护。

### 选项

#### 选项 A:每次需要时直接写 cv2.VideoCapture 代码
- **缺点**:推理代码到处是 cv2 调用,无法测试,无法切换输入源

#### 选项 B:Fork Ultralytics,改它的 LoadStreams
- **缺点**:维护负担巨大,版本升级时丢失改动

#### 选项 C:抽出独立 frame_source 模块
为 ODPlatform 实现独立的输入源抽象,统一摄像头/视频/图片/文件夹四种输入源的接口。
- **优点**:控制完全在我们手里;支持完整参数配置;统一接口降低下游代码复杂度
- **缺点**:需要投入开发

### 决策

**采纳选项 C(独立 frame_source 模块)**。

具体设计:
- `common/frame_source/base.py`:抽象基类 + 迭代器协议(`__iter__`, `__next__`)
- `common/frame_source/camera.py`:摄像头(支持完整 cv2 参数配置)
- `common/frame_source/video.py`:视频文件(支持 seek)
- `common/frame_source/images.py`:图片文件夹
- `common/frame_source/factory.py`:`create_frame_source(spec)` 工厂函数,自动识别类型

接口示意:
```python
source = create_frame_source(
    type="camera", 
    device=0, 
    resolution=(1280, 720), 
    fps=90,
    backend="dshow"
)
for frame in source:
    result = model.predict(frame)
```

### 理由

1. **填补底层框架空白**:Ultralytics 不会为我们改,我们也不应该 Fork。在合适的边界(我们自己控制的代码)做抽象,是工程上正确的应对方式。
2. **统一抽象的工程价值**:下游推理代码用 `for frame in source` 一行循环,可以无修改地切换摄像头/视频/图片——这是大幅降低复杂度的设计。
3. **解耦设计**:输入源处理独立于推理引擎,可独立单元测试,不受 Ultralytics 版本升级影响。
4. **支持业务关键能力**:高速摄像头参数配置、视频 seek 功能,在某些业务场景(传送带检测、事故复盘)是不可替代的能力。
5. **教学价值大**:这个模块的存在直接演示了"通用平台 vs 算法脚本"的本质区别——平台的价值在于识别框架的局限,在合适边界做出自己的抽象。

### 后果

#### 正面后果
- 完整的输入源参数配置能力
- 下游推理代码统一接口,切换输入源零修改
- 不依赖 Ultralytics 内部实现,版本升级风险低
- 模块独立可测试

#### 代价
- 模块本身代码量(估计 200-500 行)
- 需要维护与 cv2 不同后端的兼容性

### 何时重新评估

1. 如果 Ultralytics 官方接口未来开放参数配置(目前 GitHub Issue 长期未解决)
2. 如果出现某种新输入源(如 ROS Topic、网络流),需要扩展接口
3. 如果某种重要参数(如硬件解码加速)需要绕过当前抽象

### 延伸资料

- Ultralytics GitHub Issues #1757、#9402、#1446(用户长期反映的问题)
- OpenCV VideoCapture API 文档
- 《项目结构设计指南》0.1.2 节(frame_source 案例)

## ADR-010:工程化标准基线

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | 整个项目(代码风格、依赖管理、CI、提交规范) |
| 相关 ADR | ADR-001(仓库布局)、ADR-011(测试策略) |

### 背景

ODPlatform 作为一个工程化教学项目,需要在第一天就建立**清晰的工程化基线**——所有代码都按统一标准编写,所有提交都通过统一检查。

如果不在早期建立基线,会出现:
- 代码风格混乱(每人一种缩进、命名习惯)
- 依赖版本不锁(同一个项目,不同人装出不同结果)
- 没有 lint,低级错误反复出现
- 没有 CI,问题代码进主分支

需要决定**从 V1.0 第一天开始,有哪些工程化要求是基线?**

### 选项

#### 选项 A:不强制,自由开发
- **缺点**:必然导致风格混乱、质量参差不齐

#### 选项 B:只规范代码风格(black + isort)
- **优点**:轻量
- **缺点**:覆盖面太窄,其他工程问题(类型检查、测试、依赖管理)仍然散乱

#### 选项 C:完整工程化基线(六根支柱 + 配套)
PEP 621 单一配置源、src/ 布局、可安装包、依赖锁文件、ruff/mypy/pytest 三件套、CI as code。
- **优点**:全面、专业、与业界主流对齐
- **缺点**:学习曲线略高(但每一项都是必修课)

### 决策

**采纳选项 C(完整工程化基线)**。

具体清单:

| 支柱 | 工具/标准 | 配置文件 |
|---|---|---|
| **统一配置源** | PEP 621 + pyproject.toml | `pyproject.toml` |
| **包布局** | src/ 布局 | `apps/*/src/` |
| **依赖锁** | uv.lock | `uv.lock`(提交到 Git) |
| **代码风格 + Lint** | ruff(替代 black + isort + flake8) | `pyproject.toml` 内 `[tool.ruff]` |
| **类型检查** | mypy(渐进引入,核心层先严格) | `pyproject.toml` 内 `[tool.mypy]` |
| **测试框架** | pytest | `pyproject.toml` 内 `[tool.pytest]` |
| **Git Hooks** | pre-commit(运行 ruff + mypy + 标准格式) | `.pre-commit-config.yaml` |
| **CI as Code** | GitHub Actions / GitLab CI | `.github/workflows/` |
| **大文件管理** | Git LFS | `.gitattributes` |
| **结构化日志** | logging_utils.py 提供 JSON 日志 | `common/logging_utils.py` |
| **ADR 决策记录** | docs/architecture/ 集合 | 本文档 |

### 理由

1. **现代化基线是当前最佳实践的最小集合**:这些工具/标准都是 Python 生态 2025-2026 年的主流,不是过度工程化。
2. **ruff 替代多个工具**:ruff 一个工具替代 black + isort + flake8 + 部分 pylint,速度极快(Rust 实现),配置简单。
3. **类型检查渐进引入**:`common/` 和 `config/` 等核心层先严格(有 `py.typed` 标记 + 完整类型注解),`scripts/` 等入口层暂时宽松——这种渐进式策略避免一次性投入过大。
4. **CI 不是可选项**:即使是教学项目,也应该让学员体验"提交代码 → CI 自动跑 → 通过/失败"的完整工作流。这是工业界基础。
5. **从第一天就到位的价值**:第一行代码就是在标准基底上写的,**避免了"先野蛮生长再补标准"的痛苦**——改造已有代码比写新代码难得多。

### 后果

#### 正面后果
- 所有代码风格统一
- 依赖严格可复现
- 工程文化从第一天形成
- 学员体验"工业级标准开发流程"

#### 代价
- 学员需要学习 8-10 种工具/标准(实训 D7-D8 教学内容)
- 配置文件较多(pyproject.toml、.pre-commit-config.yaml、CI yml 等)
- 提交代码会被 hooks 拦下来——初期会有摩擦

### 演进路径

```
Stage 0(D1-D2):pre-commit + ruff 先行,学员习惯"提交前自动格式化"
Stage 1(D3-D6):mypy 在核心层强制,渐进推广
Stage 2(D7-D8):专门一天讲工程化,引入 CI as code 概念
Stage 3(实训后):未来如果开放贡献,加入 SECURITY.md、CODEOWNERS 等社区标准
```

### 何时重新评估

1. 如果某个工具被官方废弃或社区废弃(例如 black 已被 ruff 替代,这种情况要持续观察)
2. 如果 Python 生态出现新的主流标准(如未来 PEP 提案)
3. 如果项目规模膨胀到当前 CI 配置不够用(届时考虑分层 CI、缓存优化)

### 延伸资料

- PEP 621(pyproject.toml 标准化)
- ruff 官方文档
- 《项目结构设计指南》第十七章(现代化工程六根支柱)

## ADR-011:测试分层策略

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | `tests/` 目录 |
| 相关 ADR | ADR-010(工程化基线) |

### 背景

测试是工程化的另一根支柱——没有可信的测试,所有重构都是危险的;没有分层的测试,测试本身就会变成维护负担。

ODPlatform 需要一个**可执行、可教学、可演进**的测试体系——既要覆盖真实场景,又不能让测试变成项目的负担。

### 选项

#### 选项 A:不分层,所有测试混在一起
- **缺点**:运行慢(每次都跑全集);维护负担大;反馈周期长

#### 选项 B:按模块分(每个模块一个 tests/)
- **缺点**:仍然不解决"快慢混合"问题

#### 选项 C:按测试性质分层(单元/集成/端到端)
经典测试金字塔,按运行速度和覆盖范围分层。
- **优点**:不同场景跑不同层;CI 可只跑快层做 PR check;反馈快
- **缺点**:需要约定"什么算单元/集成/端到端"

### 决策

**采纳选项 C(测试金字塔三层)**。

具体设计:

```
tests/
├── unit/                ← 毫秒级,纯函数,无 IO 无外部依赖
│   ├── common/
│   ├── config/
│   ├── data_validation/
│   └── ...
├── integration/         ← 秒级,多模块协作,有受控外部依赖
│   ├── data_pipeline/   (转换+验证联合测试)
│   ├── training_flow/   (配置→训练→产物)
│   └── inference_flow/  (输入源→推理→输出)
├── e2e/                 ← 分钟级,完整端到端场景
│   ├── full_train_minimal_dataset/
│   └── full_inference_video/
├── conftest.py          ← pytest 全局 fixtures
└── fixtures/            ← 测试数据(极小数据集)
    └── tiny_voc/        ← 10 张图,用于端到端测试
```

### 理由

1. **测试金字塔是经典理论**:Mike Cohn 在 2009 年提出,经过 15+ 年验证,不存在"过时"。
2. **运行速度梯度**:
   - 单元测试:毫秒级,可在每次保存代码时自动跑(IDE 集成)
   - 集成测试:秒到分钟级,提交时本地跑或 PR 时 CI 跑
   - 端到端:分钟级,夜间或发版前跑
3. **PR CI 只跑前两层**:大幅缩短反馈周期。e2e 测试在夜间 CI 跑。
4. **fixtures 数据集是关键**:一个**极小的、版本受控的、可提交进 Git 的**数据集(tiny_voc/),让端到端测试不依赖外部环境。
5. **教学价值清晰**:学员能看到"什么是单元测试 / 什么是集成测试 / 什么是端到端测试",不会把它们混在一起。
6. **未来 Web 端的契约测试自然衔接**:V1.1 增加 Web 后端时,可以在 `tests/contract/` 加契约测试层。

### 后果

#### 正面后果
- 反馈速度快(单元测试 < 1 秒)
- 测试维护负担可控
- 学员理解测试金字塔概念
- CI 设计清晰

#### 代价
- 需要约定"什么算单元/集成"的边界(初期会有讨论)
- fixtures 目录需要维护(数据集格式变化时需更新)

### 何时重新评估

1. 如果 e2e 测试时间膨胀到超过 1 小时,需要考虑测试并行化或拆分
2. 如果某些测试始终位于"灰色地带"(不像单元也不像集成),考虑增加新层级(如"组件测试")
3. 如果 fixtures 数据集大小超过 100MB,需要迁移到 Git LFS

### 延伸资料

- Mike Cohn《Succeeding with Agile》(测试金字塔提出)
- pytest 官方文档
- 《项目结构设计指南》第十九章(分层测试体系)

## ADR-012:教学落地策略

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-05 |
| 决策者 | 讲师 |
| 受影响范围 | 课程节奏、教学组织、考核 |
| 相关 ADR | 全部前述 ADR(教学要承接所有架构决策) |

### 背景

ODPlatform 不仅是一个工程项目,**它同时承担两周实训的教学使命**。教学场景下的成功标准与纯工程项目不同——除了"代码能跑、架构合理",还要让学员**真的学到东西、产生工程认知的跃迁**。

需要决策:在两周(10 个工作日)内,如何安排课程节奏、组织协作、设计考核,让学员不仅"用 ODPlatform",更要"理解 ODPlatform"?

### 选项

#### 选项 A:线性教学(从 D1 到 D10 按目录顺序讲)
按代码目录从底层到顶层讲。
- **缺点**:学员前几天看不到"完整产品",失去动力;基础层抽象,初学者难以投入

#### 选项 B:任务驱动教学(每天一个任务)
每天发一个任务,学员动手做。
- **优点**:有产出感
- **缺点**:容易陷入"只懂局部不懂全局"

#### 选项 C:架构先行 + 任务驱动 + 反思制造
D1 完整讲架构(本文档对应内容),D2-D9 按模块动手,D10 答辩反思。每天**至少制造一个"啊哈时刻"**。
- **优点**:全局观先建立,然后落地;反思环节巩固认知
- **缺点**:D1 信息密度大,需要讲师驾驭能力

### 决策

**采纳选项 C(架构先行 + 任务驱动 + 反思)**。

具体节奏:

```
D1: 项目立项 + 架构思维 + 命名 + 仓库布局 + 完整目录设计(本文档第〇至第三部分)
D2: common/ 基础层 + paths.py marker file 模式("啊哈时刻 4")
D3: config/ 配置系统(Pydantic v2 + 三源合并)
D4: data_converter/ 数据格式转换插件化(ADR-005 实践)
D5: data_validation/ 数据质量验证体系(ADR-006 实践)
D6: training/ 训练编排服务化(ADR-007 实践)
D7: 工程化日(ruff + mypy + pre-commit + CI)
D8: 测试日(unit + integration + e2e 三层落地)
D9: inference/ 推理流水线 + frame_source(ADR-008 + ADR-009 大融合)
D10: 答辩日:每组讲一份自己写的 ADR,讲师与同学交叉评审
```

每天必须制造的认知锚点("啊哈时刻"):

| 时刻 | 内容 | 教学价值 |
|---|---|---|
| 1 | "原来工业界的做法这么不一样" | 工程化 vs 算法脚本的本质差异 |
| 2 | "原来名字会限制未来" | 命名是架构决策(配合开放讨论题) |
| 3 | "原来 Monorepo 不是仓库变大" | 原子变更的核心价值 |
| 4 | "原来今天这一步不动,以后会很惨"(D2 marker file 演示) | 工程债务的复利效应 |

### 理由

1. **架构先行的根本理由**:学员如果先动手再讲架构,会形成"我已经会了,你讲的是多余的"的抵触心理;先讲架构再动手,他们落地时会**带着问题去印证**,投入感更强。
2. **每天一个"啊哈时刻"**:认知翻转比知识灌输有效 10 倍。30 秒的 marker file 现场演示 = 30 分钟道理讲解。
3. **D10 答辩日的关键作用**:让学员**自己写一份 ADR**,讲师同学交叉评审。这一步的价值在于:他们必须**把所学内化成"自己的决策依据"**——这种内化才是真学习。
4. **承接所有前述 ADR**:每个模块的教学日(D2-D9)都对应一份 ADR——学员既看到决策,又看到代码,形成"决策-实现"的双向验证。
5. **工程文化与课程内容统一**:学员不只学技术,也学**"工业界的工作方式"**(写 ADR、做评审、跑 CI)——后者是简历竞争力。

### 后果

#### 正面后果
- 学员既学到具体技术,也建立工程思维
- D10 答辩产出可作为简历素材
- 课程节奏与项目结构互相印证
- "啊哈时刻"建立长期记忆点

#### 代价
- D1 信息密度大,讲师备课负担重
- 需要现场演示能力(marker file 30 秒演示)
- 答辩日需要安排同学评审,组织成本高

### 何时重新评估

1. 如果某届学员普遍反馈"D1 信息太密",考虑把 D1 拆成 1.5 天
2. 如果"啊哈时刻"效果不达预期(学员无明显认知翻转),需要重新设计现场演示方式
3. 如果 ODPlatform 后续演进到下一阶段(V1.1 增加 Web),教学节奏需要相应扩展

### 延伸资料

- 《项目结构设计指南》第八部分(教学落地)
- 经验:每个 D 都要预留 5-10 分钟"开放问答时间",学员问得越多,他们越投入

## ADR-013:运行配置子系统

| 字段 | 内容 |
|---|---|
| 状态 | **Accepted** |
| 日期 | 2026-05-22 |
| 决策者 | 实训团队 |
| 受影响范围 | `run_config/` 模块、CLI `odp-config` 命令 |
| 相关 ADR | ADR-004(配置系统设计)、ADR-007(训练编排)、ADR-008(推理流水线) |

### 背景

ADR-004 确立了"三源合并(代码默认值 → YAML → CLI) + 来源溯源"的配置设计方向,但留下了具体实现方式的选择。

ADR-004 设想的方案是每类任务一个 Pydantic 模型(TrainConfig、ValConfig 等),但随着 D5 实训推进,遇到几个具体问题:

1. **跨任务共享字段**:model、device、workers 等字段在不同任务中重复定义,Pydantic 模型继承体系会让字段溯源变得复杂
2. **配置溯源需要每个字段独立跟踪来源**:Pydantic 模型在校验时会丢失"这个值来自哪个源头"的信息
3. **字段按任务分组**:同一个字段名在不同任务中可能有不同的默认值或约束(如 batch 在 train 中默认 16,在 val 中默认 16 但含义不同)
4. **实验复现需要可序列化的快照**:快照需要包含完整配置 + 时间戳 + 版本号,Pydantic 模型序列化会携带类结构信息

这些问题的本质是:**配置管理需要"字段为中心"的视角,而非"任务为中心"的视角**。

### 选项

#### 选项 A:Pydantic 模型继承体系(ADR-004 原方案)

每类任务一个 Pydantic 模型,共享字段通过基类继承。

```
BaseConfig (model, device, workers, ...)
├── TrainConfig (epochs, batch, lr0, ...)
├── ValConfig (batch, imgsz, conf, ...)
└── PredictConfig (source, save_txt, ...)
```

- **优点**:类型安全、IDE 提示好、Pydantic 生态成熟
- **缺点**:
  - 字段溯源需要额外包装(Pydantic 不原生支持"每个字段来自哪里")
  - 同名字段在不同任务中有不同约束(如 batch 的 group 不同),继承体系难以表达
  - 新增字段需要修改模型定义,违反开闭原则
  - 序列化会携带 Pydantic 模型结构信息,快照不纯净

#### 选项 B:ConfigField 注册表 + 懒加载字段模块 ✅

用 `ConfigField` 定义单个字段(名称 + 类型 + 默认值 + 分组 + 校验规则),通过注册表统一管理,按需懒加载。

```
run_config/
├── registry.py     # 注册表:register_field / get_field / list_fields
├── fields/         # 字段定义模块(按任务分组)
│   ├── __init__.py # 通用字段(task, model, device, workers, ...)
│   ├── train.py    # 训练专用字段(epochs, batch, lr0, ...)
│   ├── val.py      # 验证专用字段(batch, imgsz, conf, ...)
│   └── predict.py  # 推理专用字段(source, save_txt, ...)
├── schema.py       # 数据结构(ConfigField, ConfigBundle, ConfigSnapshot, TraceRecord)
├── loader.py       # YAML 加载 + CLI 参数解析
├── merger.py       # 三源合并(默认值 → YAML → CLI)
├── validator.py    # 字段级校验 + 跨字段校验
└── service.py      # 编排层(build_config / restore_from_snapshot)
```

- **优点**:
  - **字段为中心**:每个字段独立定义,同名字段首次注册生效,后续静默跳过
  - **按任务分组**:通过 group 字段(如 `"train_训练"`)支持 `list_fields(task)` 过滤
  - **溯源原生支持**:merge 过程中自动记录每个字段的来源链
  - **开闭原则**:新增字段只需新建 .py 文件注册,不改旧代码
  - **快照纯净**:`ConfigSnapshot` 只存 `{field: value}` 字典,不携带类结构
- **缺点**:
  - 比 Pydantic 多一些样板代码(每个字段要写 ConfigField 定义)
  - 没有 IDE 静态类型提示(配置值运行时才确定类型)
  - 同名字段冲突用"首次注册胜出"策略,可能丢失任务特化约束

#### 选项 C:JSON Schema + 动态校验

用 JSON Schema 定义配置结构,用 jsonschema 库校验。

- **优点**:语言无关、生态广
- **缺点**:Python 类型提示丢失;溯源需要额外实现;动态校验错误信息不够友好

### 决策

**采纳选项 B(ConfigField 注册表 + 懒加载字段模块)**。

### 理由

1. **溯源是三源合并的核心价值**:TraceRecord 在 merge 过程中自然产生,每个字段都记录"最终值"和"来源链"。这是 Pydantic 方案需要额外包装才能做到的。
2. **字段懒加载让新增字段零侵入**:在 `fields/` 目录下新建 .py 文件,调用 `register_field()` 即可。不需要修改 registry、merger、validator 等框架代码。
3. **register_field 幂等设计**:同名字段在不同模块中重复定义时,首次注册胜出,后续静默跳过。这让共享字段(如 device、batch)可以在 `__init__.py` 中统一定义,任务模块中的同名定义被安全忽略。
4. **ConfigSnapshot 为实验复现而生(FR-23)**:纯键值对字典 + 版本号 + 时间戳,序列化为 JSON 后体积小、可读性强、跨语言兼容。
5. **敏感字段脱敏(FR-24)**:ConfigField 的 `sensitive=True` 标记在 `to_report_dict(mask_sensitive=True)` 时自动将值替换为 `"***"`,适用于 API Key 等凭据。
6. **CLI 一体化**:`odp-config` 子命令(generate / validate / trace / snapshot)全部基于同一套 registry 和 service,命令行体验与 Python API 一致。

### 后果

#### 正面后果
- 溯源成为配置系统的自然产物,非额外功能
- 新增一个配置字段 = 在 `fields/` 下加一个注册调用,不影响任何现有代码
- 快照 JSON 可以直接用于跨环境传递和存档
- 敏感字段脱敏机制无需用户额外配置,声明即生效

#### 代价
- 字段定义散布在多个文件中,查看"某个任务有哪些字段"需要查多个文件(可通过 `odp-config generate --task <T>` 查看完整清单)
- 同名字段"首次注册胜出"策略意味着 `predict.py` 中定义的 device(group="predict_输入")不会生效,而是使用 `__init__.py` 中的通用定义(group="模型")。这意味着 `list_fields("predict")` 不会返回 device——但 build_config 使用时,`list_fields()`(无过滤)会包含所有字段,所以 device 仍然可用且校验通过。
- ConfigField 是运行时类型检查,不是静态类型检查——传错类型在配置加载时报错而非 IDE 提示

### 何时重新评估

1. 如果字段数量膨胀到 100+,文件碎片化成为维护负担 → 考虑按功能域合并字段模块
2. 如果需要对同一字段在不同任务中有不同约束(如 train batch 和 val batch 有不同 ge/le)→ 需要引入"任务感知"的字段定义机制(如 ConfigField(task_overrides=...))
3. 如果 Pydantic v3 或替代方案出现原生溯源支持 → 重新评估是否迁移

- ADR-004:配置系统设计(本 ADR 的具体实现)
- `odp_platform/run_config/` 源码

- [ ] **代价没有刻意隐瞒**——好的 ADR 会主动暴露决策的弱点
- [ ] **重新评估条件是可证伪的**——用客观指标判断,不靠"感觉"
- [ ] 决策不与已有 ADR 冲突;如有取代关系,已注明 "Supersedes ADR-XXX"

### B.4 文档维护

- [ ] 文件命名遵循 `ADR-XXX-slug.md` 约定
- [ ] 同步更新了 `INDEX.md` 索引
- [ ] Commit 信息清晰描述"ADR-XXX: [标题]"
- [ ] 如有相关 ADR,在两份 ADR 中互相引用

> 通过全部 4 组检查 → 这份 ADR 进入 Accepted 流程
> 任何一项失败 → 回去修改

## 附录 C:常见错误与反模式

### C.1 反模式 1:决策后才写 ADR(追溯式)

**症状**:
- 代码已经按某种方式写完,然后"为了规范"补一份 ADR
- ADR 的"选项"部分空洞,只是为了显得"考虑过其他方案"

**问题**:
追溯式 ADR 失去了**"逼自己想清楚"**的核心价值。它变成了"为已发生的事辩护",而不是"在做决策前想清楚"。

**正确做法**:
在做决策**之前**就写 ADR(以 Proposed 状态)。让"列选项、写理由"这个动作真正影响最终决策。

### C.2 反模式 2:ADR 写成产品说明书

**症状**:
- 大量篇幅描述"系统是什么样"
- 缺少"为什么这么选"的对比

**问题**:
这种文档是设计文档,不是 ADR。它没有权衡,只有描述。

**正确做法**:
专注"权衡"——为什么选 A 不选 B / C?把 70% 篇幅放在"理由"和"选项对比"上,而不是"实现细节"上。

### C.3 反模式 3:理由过于空洞

**症状**:
- "因为 X 是当前主流"
- "因为这样更优雅"
- "因为团队倾向于这个方案"

**问题**:
这些理由对未来读者毫无说服力。**未来读者会想"主流不主流跟我们项目有什么关系?"**。

**正确做法**:
理由要**项目情境化**——用项目的具体约束(团队规模、业务需求、技术栈、时间窗)解释决策。

### C.4 反模式 4:没有"何时重新评估"

**症状**:
- ADR 写完后,从此再没人看
- 5 年后,所有人都把它当"现状",没人想过质疑

**问题**:
没有"重新评估条件"的决策会**沉睡成现状**,失去自我修正能力。

**正确做法**:
每份 ADR 都列出 3-5 个**可证伪的、可量化的**触发条件——一旦某条件触发,有人就会回到这份 ADR 来重新评估。

### C.5 反模式 5:ADR 太多,失去焦点

**症状**:
- 一年内写了 50 份 ADR
- 每个微小决策都开 ADR(变量名、文件名、缩进风格)

**问题**:
ADR 是为**重大架构决策**准备的,不是为日常决策准备的。过多的 ADR 会让真正重要的决策被淹没。

**正确做法**:
**ADR 的判断标准——这个决策影响多大?**

```
─ 影响一个文件 → 不需要 ADR(commit message 说明即可)
─ 影响一个模块 → 不需要 ADR(模块 README 即可)
─ 影响整个项目 / 影响多人协作 / 决策不可逆 → 需要 ADR
```

### C.6 反模式 6:ADR 状态不维护

**症状**:
- 决策已经被替代了,但 ADR 还是 Accepted 状态
- 实际代码已经不按 ADR 写了

**问题**:
ADR 与代码现状脱节,文档失信。

**正确做法**:
- 重大变更前,先写新 ADR(标 Proposed),旧 ADR 改为 Superseded
- 把"维护 ADR 状态"作为 PR 检查项

### C.7 反模式 7:ADR 写得过早或过晚

**症状**:
- **过早**:还没遇到真实决策点,凭想象预先写一堆 ADR(候选 ADR 不算)
- **过晚**:决策已经做完、代码已经写完几个月,才补一份 ADR

**问题**:
- 过早 → 决策空泛,因为没有真实约束
- 过晚 → 失去"逼自己想清楚"的价值,变成形式主义

**正确做法**:
**决策前 1-3 天**写 ADR(以 Proposed 状态),让 ADR 写作过程真实影响决策。

### D.1 ADR 制度的经典文献

- **Michael Nygard, "Documenting Architecture Decisions" (2011)** —— ADR 概念的原始来源,值得任何工程师读一遍

- **Spotify** —— 公开了部分技术决策的 ADR 实践
- **AWS Well-Architected Framework** —— 每条架构最佳实践背后都有类似 ADR 的决策记录文化
- **ThoughtWorks Tech Radar** —— 技术选型决策的公开追溯

### D.3 与 ADR 相关的工程文化书籍

- 《Documenting Software Architectures》(Bass, Clements, Kazman)
- 《Building Evolutionary Architectures》(Ford, Parsons, Kua)
- 《Software Architecture in Practice》
