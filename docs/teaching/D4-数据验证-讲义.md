# ODPlatform D4: data_validation 子系统

> **课程定位**: 实训第 4 天讲义
>
> **核心叙事**: 从一个干净的 D3 仓库出发, 构建一个完整的【数据质检子系统】 `data_validation/`——把"D3 产出的 YOLO yaml + 一行命令"变成"一份数据质量报告 + 退出码"。
>
> **三面真撞墙**:
> 1. **一个 validate() 函数串到底, 第 N 个 check fail 阻断后面所有 check** → 推出注册表 + `@check` 装饰器 + CheckResult 聚合
> 2. **每个 check 自己扫一遍盘** → 推出 DatasetSnapshot 一次扫描多次复用
> 3. **跑完只剩 `[PASS] yaml_schema` 这种干瘪日志, 没人知道数据集长什么样** → 推出 ValidationReport(纯数据) + render.py(纯展示)分离
>
> **配套上一节**: 《D3: data_pipeline 子系统》
>
> **配套下一节**: 《D5: 训练子系统》(D4 守好数据这一关, D5 才能开始训)
>
> **设计原则**: **质检子系统只检测, 不修复**——发现问题归 data_validation, 修复问题归 data_pipeline 重跑产线(或独立的清理工具)。两件事混在一起会让"我刚跑了一次 validate, 数据是不是变了?"这种基本问题都说不清楚。
>
> **建议时长**: 6-8 课时

## D4 跟 D3 的关系: "生产数据" → "质检数据"

D3 教的是"怎么把原始数据集变成 YOLO 训练 yaml"——**产线**。
D4 教的是"那份 yaml 长出来的数据集质量怎么样"——**质检**。

这两个子系统**架构同源**——都是注册表 + 装饰器自动注册 + 框架/实现物理分离——但调度模式相反:

| 维度 | D3 data_pipeline | D4 data_validation |
|---|---|---|
| 调度模式 | **互斥分发** (按格式选 1 个 converter) | **聚合执行** (跑全部 check, 收集结果) |
| `@register` / `@check` 触发 | 一次注册被分发选中 | 一次注册被全部跑 |
| 失败时 | 立刻抛异常 (互斥模式只关心当下这个) | 装进 CheckResult 继续跑下一个 (聚合模式承诺没人能阻断他人) |
| 默认行为 | 修改数据 (生成产物) | 只读分析 (绝不修改原始数据) |
| 退出码 | 0 / 1 / 2 (成功 / 用户错误 / 系统错误) | 0 / 1 / 2 (PASS / 有 WARNING / 有 ERROR) |

**用同一种 idiom 解决两道不同的题**——这就是注册表模式的复用价值。学完 D4 你会对 D3 的注册表理解更深(从"工厂模式的一种"升级到"通用的插件化基础设施")。

> **金句**: **"产线和质检, 是数据工程的左右脚——D3 教你迈出去, D4 教你站得稳。"**

## 0.1 子系统目录约定: 框架 vs 实现 物理分离

跟 D3 的 `data_pipeline/core/` 同款 —— **框架代码** (registry/service/snapshot/report/render) 放外层, **实现代码** (各个 check) 放 `checks/` 子包:

```
data_validation/
├── __init__.py         ← 框架: 对外公共 API
├── registry.py         ← 框架: 注册表 + 数据契约
├── snapshot.py         ← 框架: 一次扫描的数据载体
├── service.py          ← 框架: 调度层 + 端到端函数
├── report.py           ← 框架: 聚合产出 (纯数据)
├── render.py           ← 框架: 展示层 (纯展示)
└── checks/             ← 实现: 每加一个 check 加一个文件
    ├── __init__.py
    ├── yaml_schema.py
    ├── pair_existence.py
    └── ...
```

外层的 6 个 .py 文件**写完不再动**——加新 check 只是往 `checks/` 加文件, 永远不污染框架。**这是 D3 留下的肌肉记忆**, 这里只是再用一次。

⚠️ **设计点 (为什么物理目录反映开闭原则)**: 不是为了"目录好看", 是为了**让代码审查的人一眼看出"这次提交在改框架还是加实现"**。框架变动是大事(影响所有 check), 实现变动是小事(只影响一个 check)——把它们放不同目录, 一行 `git diff --stat` 就能区分。这种"可读性靠目录"的设计, 比"靠 commit message"更可靠。

## 0.2 paths.py 增量(本节只新增 1 项)

> 💡 跟 D3 阶段 0.4 同样的惯例: **paths.py 的修改集中在"加新子系统"这一个时刻**, 不要散在阶段 5 加一条、阶段 11 又加一条。

D3 阶段 0 已经把 `RUNS_DIR` 立起来了(端私有, 每次训练 / 验证产出都进这里)。D4 只需要在它下面再立一个子目录辅助函数:

打开 `apps/platform/src/odp_platform/common/paths.py`, 末尾追加:

```python
# ============================================================
# D4 增量: 数据验证运行目录
# ============================================================
VALIDATION_RUNS_DIR: Path = RUNS_DIR / "data_validation"


def validation_run_dir(run_id: str) -> Path:
    """返回某次验证运行的产出目录: runs/data_validation/<run_id>/

    Args:
        run_id: 形如 "20260516_184523" 的时间戳 ID (由 validate_dataset 生成)

    Returns:
        Path 对象 (尚未创建, 调用方自己 mkdir)

    用法:
        run_dir = validation_run_dir("20260516_184523")
        run_dir.mkdir(parents=True, exist_ok=True)
        (run_dir / "report.json").write_text(...)
    """
    return VALIDATION_RUNS_DIR / run_id
```

注意**没有**把 `VALIDATION_RUNS_DIR` 加进 `get_dirs_to_initialize()`——因为它是"运行时按需创建", 不是"项目初始化必备目录"。`odp-init` 不该提前建一个空的 `runs/data_validation/`, 否则用户没跑过验证, 目录就先躺在那里, 误导性大。

⚠️ **设计点 (按需创建 vs 启动创建)**:
- **启动创建** (在 `get_dirs_to_initialize` 里): 项目结构的"骨架"目录, 比如 `configs/datasets/`, `data/raw/`——用户哪天会用, 先建好不冤枉。
- **按需创建** (调用方自己 mkdir): "用了才有"的目录, 比如 `runs/<run_id>/`, `tmp_xxx/`——没跑过的时候**不存在**才是对的状态(`ls runs/` 空就代表"还没跑过", 信息密度高)。

> **金句**: **"目录的'不存在'本身也是信息——别用预先 mkdir 抹掉它。"**

"""缺标签比例 >= 50% 时 pair_existence 触发 ERROR (流程崩了)"""

"""缺标签比例 >= 5% (但 < 50%) 时触发 WARNING; 低于此值仅 INFO"""
```

⚠️ **陷阱预警**: 不要在 `data_validation` 子系统内部再立一个 `TaskType.DETECTION / SEGMENTATION`。D3 立的是 `Task.DETECT / SEGMENT`(字符串值 `"detect"` / `"segment"`), 跟 ultralytics yaml 里的 `task:` 字段一致。**D4 直接 import 用就对了**——立第二份会:

```python
# ❌ 反模式
class TaskType:
    DETECTION    = "detection"     # ← 跟 Task.DETECT="detect" 字符串都不一样
    SEGMENTATION = "segmentation"  # ← 跟 Task.SEGMENT="segment" 也不一样
```

立刻产生两个版本的"任务名"真相, 测试时 `Task.DETECT == TaskType.DETECTION` 就 fail——SSoT 立马破。**遇到"我需要任务类型"的需求, 第一反应应该是 `from odp_platform.common.constants import Task`**, 而不是"我再立一个"。

> **金句**: **"SSoT 的纪律不是写出来的, 是【克制】出来的——每次想新立常量前先问'是不是已经有了'。"**

# 阶段 1: 朴素第一版——一个 validate() 函数串到底

D4 的核心架构是**注册表 + @check + CheckResult**。但**直接抄过来很难体会"为什么是这样"**——所以阶段 1 我们故意先写一版"看起来很自然"的朴素实现, 等它**长歪了**再砸掉。

跟 D3 阶段 1 的剧本同款——先写一个 `convert()` 函数, 再撞 if/elif 漂移的墙, 推出注册表。D4 的撞墙不一样, 但**先朴素再重构**的节奏一致。

## 1.1 任务设定: 4 个检查项

D4 要做的"数据质检", 翻译成具体动作就是 4 个**互相独立**的检查:

| # | 检查项 | 在问什么 |
|---|---|---|
| 1 | yaml_schema | yaml 字段对不对 (nc, names, names 长度==nc) |
| 2 | pair_existence | 每张图都有对应的 .txt 标签吗 |
| 3 | label_format | 每行 .txt 的格式合法吗 (5 字段, class_id 范围, 坐标 [0,1]) |
| 4 | split_uniqueness | train/val/test 之间有没有图像名重复 (数据泄露!) |

**互相独立**这点关键——`yaml_schema` 失败不该阻止 `split_uniqueness` 继续跑(它们查的是完全不同的东西)。这是后面撞墙①的核心。

阶段 1 先把这 4 个 check **直接写在一个文件、一个函数里**, 看看会变成什么样子。

## 1.3 第一版: 一个 validate() 函数串到底

新建 `apps/platform/src/odp_platform/data_validation/validate_v1.py`——这是**阶段性的废弃文件**, 阶段 2 砸掉, 现在用它体会"一坨怎么演变成"。

⚠️ **写在代码前 — 朴素 != 偷工减料**:

朴素第一版的"朴素"指的是**架构上**——一个函数串到底, 没有注册表 / 没有 snapshot / 没有分级 severity。**但代码里的细节仍然该一开始就写对**:

- **names 字段支持 list 和 dict 两种格式**: ultralytics 现役 yaml 既有 `names: [a,b,c]` 也有 `names: {0:a, 1:b, 2:c}` 两种写法(D3 的 yaml_writer 写的就是 dict 形式)。 朴素版**不能因为"我是反面教材"就只支持 list**, 否则学员跑 D3 产出的 yaml 第一步就报 "names 不是 list" 卡死, **撞错了墙**。
- **入参用数据集名而不是完整路径**: D3 已经立了 `dataset_yaml_path(name)`, 朴素版直接复用——`python -m ... rsod` 比 `python -m ... apps/platform/configs/datasets/rsod.yaml` 友好得多。 SSoT 原则不分朴素版还是正版。

阶段 1 要让学员撞的是**架构问题**(撞墙①), 不是"我没考虑 yaml 格式 / 路径拼错了" 这种**基础卫生问题**——这两件事一开始就写对。

```python
# apps/platform/src/odp_platform/data_validation/validate_v1.py
"""第一版: 朴素实现, 4 个 check 全塞一个函数。

这是一个【会被砸掉】的文件——阶段 2 用注册表替换它。
不进 git——见阶段 1.7 的设计点。
"""
from __future__ import annotations

import logging
from pathlib import Path
from typing import Dict

import yaml

from odp_platform.common.constants import IMAGE_EXTENSIONS

logger = logging.getLogger(__name__)


def validate(yaml_path: Path) -> bool:
    """一坨实现——4 个 check 串到底。

    Returns:
        True  → 全部通过
        False → 任意一项失败
    """
    # ============ check 1: yaml_schema ============
    if not yaml_path.exists():
        logger.error(f"yaml 文件不存在: {yaml_path}")
        return False

    try:
        with open(yaml_path, "r", encoding="utf-8") as f:
            cfg = yaml.safe_load(f)
    except yaml.YAMLError as e:
        logger.error(f"yaml 解析失败: {e}")
        return False

    if not isinstance(cfg, dict):
        logger.error(f"yaml 顶层不是 dict: {type(cfg).__name__}")
        return False

    names = cfg.get("names")
    nc    = cfg.get("nc")

    # ultralytics 接受两种 names 写法 (新旧都得支持):
    #   list[str]:        names: [cat, dog, bird]
    #   dict[int, str]:   names: {0: cat, 1: dog, 2: bird}   ← D3 yaml_writer 出的就是这种
    if isinstance(names, list) and names and all(isinstance(n, str) and n for n in names):
        names_count = len(names)
    elif (isinstance(names, dict) and names
            and all(isinstance(k, int) for k in names.keys())
            and all(isinstance(v, str) and v for v in names.values())):
        names_count = len(names)
    else:
        logger.error("names 缺失或不是合法的 list[str] / dict[int, str]")
        return False

    if not isinstance(nc, int) or nc <= 0:
        logger.error(f"nc 缺失或不是正整数: {nc!r}")
        return False

    if names_count != nc:
        logger.error(f"nc ({nc}) 跟 names 长度 ({names_count}) 不一致")
        return False

    logger.info(f"[PASS] yaml_schema: nc={nc}, names_count={names_count}")

    # 解析 data_root (yaml.path 字段, 或 yaml 同级目录兜底)
    path_str = cfg.get("path")
    if path_str:
        data_root = Path(path_str)
        if not data_root.is_absolute():
            data_root = (yaml_path.parent / data_root).resolve()
    else:
        data_root = yaml_path.parent.resolve()

    # ============ check 2: pair_existence ============
    missing_total = 0
    images_total  = 0
    for split in ("train", "val", "test"):
        split_rel = cfg.get(split)
        if not split_rel:
            continue
        split_dir = data_root / split_rel if not Path(split_rel).is_absolute() else Path(split_rel)
        if not split_dir.exists():
            continue
        for ext in IMAGE_EXTENSIONS:
            for img in split_dir.glob(f"*{ext}"):
                images_total += 1
                # YOLO 默认布局: images/<split>/foo.jpg → labels/<split>/foo.txt
                parts = list(img.parts)
                for i in range(len(parts) - 1, -1, -1):
                    if parts[i] == "images":
                        parts[i] = "labels"
                        break
                label = Path(*parts[:-1]) / (img.stem + ".txt")
                if not label.exists():
                    missing_total += 1

    if missing_total > 0:
        logger.error(f"[FAIL] pair_existence: {missing_total}/{images_total} 张图无标签")
        return False
    logger.info(f"[PASS] pair_existence: {images_total} 张图都有标签")

    # ============ check 3: label_format ============
    for split in ("train", "val", "test"):
        split_rel = cfg.get(split)
        if not split_rel:
            continue
        split_dir = data_root / split_rel if not Path(split_rel).is_absolute() else Path(split_rel)
        if not split_dir.exists():
            continue
        for ext in IMAGE_EXTENSIONS:
            for img in split_dir.glob(f"*{ext}"):
                parts_p = list(img.parts)
                for i in range(len(parts_p) - 1, -1, -1):
                    if parts_p[i] == "images":
                        parts_p[i] = "labels"
                        break
                label = Path(*parts_p[:-1]) / (img.stem + ".txt")
                if not label.exists():
                    continue
                with open(label, "r", encoding="utf-8") as f:
                    for line_no, line in enumerate(f, 1):
                        line = line.strip()
                        if not line:
                            continue
                        parts = line.split()
                        if len(parts) != 5:
                            logger.error(
                                f"[FAIL] label_format: {label.name} 第{line_no}行字段数!=5"
                            )
                            return False
                        try:
                            cls_id = int(parts[0])
                            coords = [float(x) for x in parts[1:5]]
                        except ValueError:
                            logger.error(
                                f"[FAIL] label_format: {label.name} 第{line_no}行类型错"
                            )
                            return False
                        if not (0 <= cls_id < nc):
                            logger.error(
                                f"[FAIL] label_format: {label.name} 第{line_no}行 cls_id={cls_id} 越界 (nc={nc})"
                            )
                            return False
                        if not all(0.0 <= c <= 1.0 for c in coords):
                            logger.error(
                                f"[FAIL] label_format: {label.name} 第{line_no}行坐标越界"
                            )
                            return False
    logger.info(f"[PASS] label_format")

    # ============ check 4: split_uniqueness ============
    stems_by_split: Dict[str, set] = {}
    for split in ("train", "val", "test"):
        split_rel = cfg.get(split)
        if not split_rel:
            continue
        split_dir = data_root / split_rel if not Path(split_rel).is_absolute() else Path(split_rel)
        if not split_dir.exists():
            continue
        stems = set()
        for ext in IMAGE_EXTENSIONS:
            stems.update(p.stem for p in split_dir.glob(f"*{ext}"))
        stems_by_split[split] = stems

    for s1, s2 in [("train", "val"), ("train", "test"), ("val", "test")]:
        if s1 not in stems_by_split or s2 not in stems_by_split:
            continue
        common = stems_by_split[s1] & stems_by_split[s2]
        if common:
            logger.error(
                f"[FAIL] split_uniqueness: {s1} 和 {s2} 之间有 {len(common)} 张重复——数据泄露!"
            )
            return False
    logger.info(f"[PASS] split_uniqueness")

    logger.info("✓ 全部 4 项检查通过")
    return True


if __name__ == "__main__":
    import sys
    from odp_platform.common.paths import dataset_yaml_path

    logging.basicConfig(level=logging.INFO, format="%(levelname)s %(message)s")

    if len(sys.argv) != 2:
        print("用法: python -m odp_platform.data_validation.validate_v1 <数据集名>")
        print("例如: python -m odp_platform.data_validation.validate_v1 rsod")
        sys.exit(1)

    dataset_name = sys.argv[1]
    yaml_path    = dataset_yaml_path(dataset_name)
    ok = validate(yaml_path)
    sys.exit(0 if ok else 1)
```

## 1.5 🧱 撞墙①: 看看现在的 validate() 函数长啥样

现在 `validate_v1.py` 的 `validate()` 函数 **160+ 行**。先列几个症状, 再分析本质。

**症状 A: 一份"对所有图像循环"重复了 3 次**

`pair_existence` / `label_format` / `split_uniqueness` 都要遍历 train/val/test 的所有图像——**同一份 for 循环写了 3 次**, 每次都重新 `glob` 一遍盘。10 万张图的数据集, 这是 10 万 ×3 = 30 万次文件系统操作。

**症状 B: 一个 fail 就 `return False`, 后面的 check 都不跑了**

```python
if missing_total > 0:
    logger.error(...)
    return False   # ← pair_existence 失败, 后面 2 个 check 都不查了
```

这才是**真正可怕的**。设想数据集既有"标签格式错"又有"训练集验证集图像重复"——用户:

1. 跑一次, 看到 `pair_existence 失败`, 修(花 10 分钟)
2. 跑第二次, 看到 `label_format 失败`, 修(再花 20 分钟)
3. 跑第三次, 看到 `split_uniqueness 失败`(数据泄露!), 才知道**最严重的问题原来一直在那**
4. ...

**本来一次能给齐的诊断, 被切成了 3 次串行**。用户的时间成本被工具浪费——这是**沉默的工程犯罪**。

**症状 C: 加第 5 个 check 会改这 160 行**

下次再加一个 check(比如"检测损坏图片"), 还得**继续往这个函数里加**——它的体积只会**单调增长**。永远不会变小。

**症状 D: 每个 check 的"长什么样"差别巨大**

- `yaml_schema` 主要在 logger.error 一行字符串
- `label_format` 在双重循环里 logger.error
- `split_uniqueness` 在循环外计算后 logger.error

**3 种风格混在一个函数里**, 改一种逻辑要先理解另外两种的上下文——**心智负担线性增长**。

## 1.6 撞墙①的本质

D3 撞墙①是"if/elif 调用方漂移"——**信息散在多文件**。
D4 撞墙①是"一个函数串到底"——**信息黏在单文件不能拆**。

但本质相同:

> **同一份"我支持哪些 X"的真相, 跟"调度逻辑"和"具体实现"耦合在一起, 改一个就要碰所有。**

D3 的解药是: 让每个 converter **声明**自己处理什么格式, 中间层只查表分发。
D4 的解药是: 让每个 check **声明**自己叫什么名字, 中间层**全部跑一遍**收集结果——而不是 fail 就停。

⚠️ **关键观察 (聚合模式 vs 互斥模式)**: D3 是**互斥分发**——一次调用选 1 个 converter。D4 是**聚合执行**——一次调用跑全部 check。两种调度模式的"注册表"机制是相同的, 但**调度层的承诺不同**:

- D3 service.convert: "我从注册表挑一个跑, 你只需要给我格式名"
- D4 service.run_all_checks: "我从注册表把每一个都跑一遍, **任何 check 抛异常我都接住, 不让它顶死其他 check**"

**第二句的后半句是 D4 调度层的核心承诺**——没有它, 聚合模式不成立(否则第一个 check 抛异常, 用户还是只看到一个错)。这条承诺会在阶段 2 的 `run_all_checks` 里用 `try/except Exception` 兑现。

> **金句**: **"聚合模式的本质不是'调用所有'——是'承诺没人能阻断所有'。后者比前者贵, 但缺了它聚合就不成立。"**

## 1.7 想清楚解药的形状

想清楚解药要长什么样, 再去实现:

1. **每个 check 是一个独立函数**, 不再塞在 `validate()` 里——这样 160 行的函数会拆成 4 个 40 行的小函数
2. **每个 check 返回一个统一的结果对象**(不是 `bool`), 这样调度层能 "失败也继续, 把结果攒成列表"
3. **结果对象要有 severity**——区分"这是 ERROR 阻塞 CI"和"这是 INFO 提醒一下", 因为不同 check 的紧张度不应该一样
4. **调度层只做"按注册表跑一遍"**, 不内嵌任何 check 的业务——以后加新 check 是**加新文件**, 不是改调度层
5. **加新 check 不应该改框架代码**——这意味着 `registry.py` 不能维护一个"硬编码 import 列表"

第 1-2 点直接对应 D3 注册表的复用。**第 3 点 (severity 分级)** 是 D4 特有的——D3 的 converter 只关心"成功 / 失败", D4 的 check 有"严重程度", 这是一个**新的设计变量**, 阶段 2 会重点讲。**第 5 点**是 D4 的真正开闭原则兑现——阶段 2 用 `pkgutil` 自动 import 解决。

# 阶段 2: 注册表机制——D4 子系统的骨架

阶段 2 把 D4 的**框架部分**一次写完。具体包括 4 个文件:

1. `registry.py` ← 数据契约 (CheckResult / CheckSeverity / CheckContext) + 注册表 + `@check` 装饰器 + **自动 import** 机制
2. `service.py` ← 调度层 (`run_all_checks` + 异常兜底)
3. `__init__.py` ← 对外公共 API
4. `checks/__init__.py` + `checks/_placeholder.py` ← 占位, 验证机制能跑通

## 2.1 先想清楚: registry.py 要装什么?

不要急着写代码。先列清单——`registry.py` 至少要解决 5 个问题:

| 问题 | 解决物 |
|---|---|
| 1. 怎么注册 check? | `@check("name")` 装饰器 |
| 2. 注册到哪里去? | 模块级 dict `_REGISTRY` |
| 3. check 函数返回什么? | `CheckResult` dataclass |
| 4. CheckResult 怎么表达"多严重"? | `CheckSeverity` 四级常量 |
| 5. check 函数收什么参数? | `CheckContext` dataclass (阶段 4 会加 snapshot 字段, 这里先放 yaml_path) |

5 个东西 + 1 个 `get_all_checks()` 查询 API + 1 个**自动 import 机制**(解决"加新 check 不改框架")——一个文件搞定。

## 2.2 CheckResult 字段设计的 4 个选择

这是 D4 整个子系统最重要的数据结构, 值得逐字段拆解。

字段名 + 类型 + 用途:

```python
@dataclass
class CheckResult:
    name:     str                       # check 的字符串名
    severity: str                       # ERROR / WARNING / INFO / PASS
    summary:  str                       # 一句话总结 (给人看)
    details:  Dict[str, Any]            # 结构化详情 (给机器看)

    @property
    def passed(self) -> bool:           # 派生属性, 不存值
        return self.severity in (CheckSeverity.PASS, CheckSeverity.INFO)
```

4 个字段, 每个都有别的方案, 但我们选了这个版本——下面逐项讲为什么。

### 选择 1: `severity` 是 enum (四级), 不是 `bool passed` (两级)

最朴素的设计是 `passed: bool`——通过 / 不通过。但实际产线上, **"不通过"至少分三种紧张度**:

- **ERROR**: 训练绝不能继续 (e.g. `split_uniqueness` 失败 = 数据泄露)
- **WARNING**: 能继续但要人工 review (e.g. `pair_existence` 缺 30%)
- **INFO**: 知道一下 (e.g. 缺 0.1% 标签 — 影响极小)

`bool` 装不下这种"同样是失败, 紧张度不同"的语义。**升级为四级 enum**:

```python
class CheckSeverity:
    ERROR   = "ERROR"     # CI 必须停
    WARNING = "WARNING"   # 能继续, 但看
    INFO    = "INFO"      # 知道一下
    PASS    = "PASS"      # 通过

    _ORDER = {"PASS": 0, "INFO": 1, "WARNING": 2, "ERROR": 3}

    @classmethod
    def rank(cls, level: str) -> int:
        return cls._ORDER.get(level, 0)
```

四级一次到位, 比"先两级再升三级再升四级"省掉两次重构。**rank() 方法**让 `ValidationReport.overall_severity = max(results, key=rank)` 一行搞定——这是后面阶段 8 的伏笔。

⚠️ **为什么是字符串常量类不是 `enum.Enum`?** 跟 D3 阶段 0.5 立的同款理由——字符串值跟 JSON / 日志 / 命令行参数对接最自然。`enum.Enum` 写起来要 `Severity.ERROR.value`, 太啰嗦。

⚠️ **为什么保留 INFO 而不是直接砍到三级?** 看似简化, 实际有代价——`pair_existence` 的"缺 0.1% 标签"既不该是 PASS(数据集**确实**有问题), 也不该是 WARNING(影响极小, 不应该闹眼前)。砍掉 INFO 强行三级会让"少量遗漏可容忍"无处表达——要么硬升 WARNING(假阳性), 要么硬贬 PASS(信号丢失)。**对齐 syslog / k8s events 的成熟分级习惯**, INFO 留着, 集成监控告警时也不别扭。

### 选择 2: `details` 是 `Dict`, 不是 `List[str]`

朴素方案:

```python
@dataclass
class CheckResult:
    passed: bool
    messages: List[str]   # 错误信息列表
```

这套方案的问题: **messages 是人话, 机器只能 grep**。考虑这个场景:

- `pair_existence` 失败要展示: `train 缺 152/1000 = 15.2%`
- 一份 JSON 报告要把这个数字单独取出来做趋势图: 上周缺 5%, 这周缺 15%, 下周?

如果 messages 是 `["train 缺 152/1000 = 15.2%"]`, **要正则解析才能拿到 15.2**——脆弱、易错、跨语言不通用。

`details: Dict` 的方案:

```python
{
    "total_images":  1000,
    "total_missing": 152,
    "missing_ratio": 0.152,
    "missing_per_split": {"train": 152},
}
```

- `summary: str` 给人看 ("train 缺 152 张标签 (15.2%)")
- `details: Dict` 给机器看 (上面的结构化字典)

两者职责分离。`summary` 短, `details` 厚。日志显示 `summary`, JSON 报告写 `details`, 监控趋势图读 `details` 里的具体字段。

### 选择 3: `passed` 是 `@property`, 不是存值

朴素:

```python
@dataclass
class CheckResult:
    severity: str
    passed:   bool   # ← 跟 severity 冗余, 容易不一致
```

如果两个字段都存, 万一 `severity=ERROR` 但 `passed=True`(写代码时手抖), 数据自相矛盾。

`@property` 派生:

```python
@property
def passed(self) -> bool:
    """PASS / INFO 都算通过——INFO 的语义是'知道一下但不阻断'。"""
    return self.severity in (CheckSeverity.PASS, CheckSeverity.INFO)
```

**单一真相**(`severity`), 派生量(`passed`)永远跟主字段一致——这是数据类设计的"DRY"应用。

注意 `INFO` 也算 passed——这反映了"INFO 是非事件"的语义。后面 `overall_severity` / `exit_code` 会再用这个判定。

### 选择 4: CheckContext 这层包装(而不是直接传 `yaml_path`)

朴素:

```python
def my_check(yaml_path: Path) -> CheckResult:
    ...
```

进阶版有一层包装:

```python
@dataclass
class CheckContext:
    yaml_path: Path
    # 阶段 4 会加 snapshot 字段

def my_check(ctx: CheckContext) -> CheckResult:
    ...
```

为什么多一层?——**这是扩展位**。

阶段 4 引入 `DatasetSnapshot` 后, 所有 check 都要消费 snapshot。如果当时函数签名是 `(yaml_path: Path)`, 我们就要**一次性改所有 check 的签名**(每个 check 一个文件, 8 处改动)——这就违反了开闭。

但如果当时签名是 `(ctx: CheckContext)`, 我们只需要**给 CheckContext 加一个字段**——所有 check 函数签名一行不改, 用 `ctx.snapshot` 直接拿。

**未来的扩展也走这条路**:
- 加历史快照对比? → `ctx.previous_snapshot`
- 加运行时配置覆盖? → `ctx.config_overrides`
- 加并发上下文? → `ctx.executor`

**永远不改 check 函数签名**——这是 D4 设计的"扩展友好性"的物理基础。

> **金句**: **"框架的难度不在'写代码', 在'选好字段'——字段定下来, 代码就只剩抄写; 字段定错了, 代码越长越歪。"**

## 2.3 registry.py 完整代码

`apps/platform/src/odp_platform/data_validation/registry.py`:

```python
# apps/platform/src/odp_platform/data_validation/registry.py
"""data_validation 注册表 + 数据契约 (CheckResult / CheckSeverity / CheckContext)。

跟 D3 的 data_pipeline/registry.py 是【同一模式的两种用法】:
    - D3: 互斥分发 (一次调用一个 converter)
    - D4: 聚合执行 (一次调用全部 check, 收集结果)

公开 API:
    @check(name)                 装饰器, 自动注册一个 check 函数
    CheckResult                  统一返回类型
    CheckSeverity                严重程度 (四级)
    CheckContext                 check 函数的入参合同 (扩展位)
    get_all_checks()             返回全部注册的 check (供 service 调度)
    get_check(name)              按名查询单个 check (供测试)
    list_check_names()           返回注册的 check 名列表
"""
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Callable, Dict, List

logger = logging.getLogger(__name__)


# ============================================================
# 1. CheckSeverity — 严重程度 (四级一次到位)
# ============================================================

class CheckSeverity:
    """Check 结果的严重程度。

    跨级关系 (供 ValidationReport.overall_severity 比较):
        ERROR > WARNING > INFO > PASS

    四级而不是两级 (passed: bool) 的理由——见讲义阶段 2.2 选择 1。
    """
    ERROR   = "ERROR"     # 阻塞级 (CI 必须停, 训练绝不能继续)
    WARNING = "WARNING"   # 关注级 (能继续, 但需要人工 review)
    INFO    = "INFO"      # 告知级 (工程上"知道一下", 不阻断)
    PASS    = "PASS"      # 通过

    _ORDER = {"PASS": 0, "INFO": 1, "WARNING": 2, "ERROR": 3}

    @classmethod
    def rank(cls, level: str) -> int:
        """供聚合层比较 severity 用 — ERROR.rank() > PASS.rank()。"""
        return cls._ORDER.get(level, 0)


# ============================================================
# 2. CheckResult — 单个 check 的统一返回类型
# ============================================================

@dataclass
class CheckResult:
    """单个 check 的完整产出。

    Args:
        name:     check 名 (= 注册时的字符串, 跟 @check("name") 一致)
        severity: ERROR / WARNING / INFO / PASS (见 CheckSeverity)
        summary:  一句话总结, 供终端日志 / 报告头部使用 (给人看)
        details:  结构化详情字典 (给机器看 — JSON 报告 / 监控趋势)
    """
    name:     str
    severity: str
    summary:  str
    details:  Dict[str, Any] = field(default_factory=dict)

    @property
    def passed(self) -> bool:
        """便于聚合层判断是否算"通过"。

        PASS / INFO 都算通过 — INFO 的语义是"知道一下但不阻断"。
        """
        return self.severity in (CheckSeverity.PASS, CheckSeverity.INFO)


# ============================================================
# 3. CheckContext — check 函数的入参合同 (扩展位)
# ============================================================

@dataclass
class CheckContext:
    """Check 函数的入参 — 所有 check 函数签名都是 (ctx: CheckContext) -> CheckResult。

    本阶段先放最简单的字段:
        yaml_path : 数据集 yaml 文件路径

    阶段 4 引入 DatasetSnapshot 时, 这里会加 snapshot 字段 — 所有 check 改成消费
    snapshot, 不再自己扫盘。届时函数签名仍然不变, 只是 ctx 里的可用资源变多了 —
    这就是为什么我们把它做成一个对象, 而不是几个并列的位置参数。

    未来扩展(D4.x / D5+):
        - previous_snapshot: 历史比较
        - config_overrides:  数据集级阈值覆盖
        - executor:          并发上下文
    永远不改 check 函数签名 —— 这是 CheckContext 这层包装的全部意义。
    """
    yaml_path: Path


# ============================================================
# 4. 注册表条目
# ============================================================

@dataclass(frozen=True)
class CheckEntry:
    """注册表里的一条记录。frozen — 注册后不可改, 一锤子买卖。"""
    name: str
    func: Callable[[CheckContext], CheckResult]


# ============================================================
# 5. 模块级注册表 + @check 装饰器
# ============================================================

_REGISTRY: Dict[str, CheckEntry] = {}
_INITIALIZED: bool = False


def check(name: str) -> Callable:
    """注册一个 check 函数到全局注册表。

    用法:
        @check("yaml_schema")
        def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
            ...

    设计:
        - 装饰器不包装函数 — 直接 return func。
          理由: 让被装饰的函数堆栈干净, 单元测试可以直接调用它而不经过装饰器,
          失败时的 traceback 也不会多一层"check_wrapper"的迷惑帧。

        - 重复注册立刻报 ValueError, 不是覆盖。
          理由: 重复注册要么是手误(两个 check 取了同名), 要么是 import 链异常
          导致同一个 check 被加载了两次 — 两种都是 bug, 应该立刻可见。

    Args:
        name: check 的字符串名 (会出现在 JSON 报告 / 日志里, 选稳定的)

    Returns:
        装饰器函数 — 注册副作用发生后, 原函数被原样返回。
    """
    def decorator(func: Callable[[CheckContext], CheckResult]) -> Callable:
        if name in _REGISTRY:
            raise ValueError(
                f"check '{name}' 重复注册 — 第二次出现在 {func.__module__}.{func.__name__}"
            )
        _REGISTRY[name] = CheckEntry(name=name, func=func)
        return func
    return decorator


# ============================================================
# 6. 自动 import — 加新 check 不改框架代码的物理基础
# ============================================================

def _ensure_initialized() -> None:
    """首次调用 get_all_checks() / get_check() 时, 自动 import checks/ 子包下
    所有非下划线开头的模块, 让其中的 @check 装饰器副作用发生。

    设计:
        - 用 pkgutil.iter_modules 自动扫盘, 不维护硬编码 import 列表。
          理由: 框架的 registry.py 维护一个手工 import 列表违反开闭 —
          加新 check 要改两处(checks/ 加文件 + registry.py 加 import)。
          自动 import 5 行解决, 加新 check 只动 checks/。

        - 跳过下划线开头的模块(_placeholder, _internal_helpers 等)。
          理由: 给框架留一个"内部辅助模块不被自动注册"的逃生口。

        - _INITIALIZED 标志防止重复 import (import 已经是 idempotent 的,
          但 pkgutil 扫盘有微小开销, flag 一下省掉)。
    """
    global _INITIALIZED
    if _INITIALIZED:
        return
    _INITIALIZED = True

    import importlib
    import pkgutil
    from odp_platform.data_validation import checks

    for _, name, _ in pkgutil.iter_modules(checks.__path__):
        if name.startswith("_"):
            continue
        importlib.import_module(f"{checks.__name__}.{name}")


# ============================================================
# 7. 查询 API
# ============================================================

def get_all_checks() -> List[CheckEntry]:
    """返回全部注册的 check (按注册顺序)。供 service.run_all_checks 调度用。"""
    _ensure_initialized()
    return list(_REGISTRY.values())


def get_check(name: str) -> CheckEntry:
    """按名查询单个 check。供测试 / CLI 调试用。

    Raises:
        KeyError: 未注册的 check 名 — 立刻报错, 不返回 None 让调用方一脸懵
    """
    _ensure_initialized()
    if name not in _REGISTRY:
        raise KeyError(f"check '{name}' 未注册 — 已注册的: {list(_REGISTRY)}")
    return _REGISTRY[name]


def list_check_names() -> List[str]:
    """返回已注册 check 的名字列表。"""
    _ensure_initialized()
    return list(_REGISTRY.keys())
```

⚠️ **设计点 (装饰器不包装函数)**: 注意 `@check` 的 decorator 直接 `return func`, 不是 `return wrapper(func)`。

```python
# ✓ 好品味
def decorator(func):
    _REGISTRY[name] = CheckEntry(name=name, func=func)
    return func    # ← 原样返回, 不包装

# ✗ 别这么干
def decorator(func):
    _REGISTRY[name] = CheckEntry(name=name, func=func)
    @wraps(func)
    def wrapper(ctx):
        return func(ctx)   # ← 多一层不带任何价值的转发
    return wrapper
```

包装一层有两个副作用:
1. **堆栈多一帧**: 测试失败时 traceback 多一行 "in wrapper, return func(ctx)", 把视线从真正的错误拉走
2. **单元测试不灵活**: 想绕过注册表直接测原函数? 拿到的是 wrapper 不是 func

**装饰器的功能是"注册副作用", 不是"包装行为"**——这点想清楚, 就不会顺手加 `@wraps`。

⚠️ **设计点 (自动 import 跟 Flask / pytest 同款)**: `pkgutil.iter_modules` 这 5 行解决了一件大事——**"加新 check 不改框架代码"的物理基础**。

这套模式不是 D4 发明的, 是 Python 生态的成熟做法:
- **Flask**: 你写的 `@app.route("/users")`, Flask 在 import time 自动登记到路由表
- **pytest**: 你写的 `def test_xxx():`, pytest 在 collect 阶段自动找出来
- **D4**: 你写的 `@check("yaml_schema")`, registry 在 `get_all_checks()` 首次调用时自动找出来

**理解一次, 全行业通用**——这是这 5 行代码真正的教学价值。

```python
# apps/platform/src/odp_platform/data_validation/service.py
"""data_validation 调度层 — run_all_checks。

聚合模式核心承诺: 任何 check 抛异常都不能阻断其他 check。
这条承诺通过 _safe_run_one 里的 try/except Exception 兑现 —— 整个 D4 子系统
里仅此一处用到 Exception 这种宽泛的捕获, 因为这里是"开闭原则下的扩展点"
(谁都能加 check), 调度层无法预知 check 会抛什么。

其他地方应该用具体的 except OSError / except yaml.YAMLError 等。
"""
from __future__ import annotations

import logging
from typing import List

from odp_platform.common.performance_utils import time_it
from odp_platform.data_validation.registry import (
    CheckContext, CheckEntry, CheckResult, CheckSeverity, get_all_checks,
)

logger = logging.getLogger(__name__)


@time_it(name="run_all_checks")
def run_all_checks(ctx: CheckContext) -> List[CheckResult]:
    """跑全部注册的 check, 收集结果。

    聚合模式承诺:
        任何 check 自身抛异常都被本函数接住, 包装成 ERROR 级 CheckResult,
        不阻断其他 check。 这是 D4 跟 D3 (互斥模式) 最大的区别 —— D3 service.convert
        失败立刻抛, D4 service.run_all_checks 失败也继续。

    Returns:
        List[CheckResult] — 按 check 注册顺序, 一条不漏
    """
    entries = get_all_checks()
    logger.info(f"开始执行 {len(entries)} 个 check")

    results: List[CheckResult] = []
    for entry in entries:
        result = _safe_run_one(entry, ctx)
        _log_check_result(result)
        results.append(result)

    _log_summary(results)
    return results


def _safe_run_one(entry: CheckEntry, ctx: CheckContext) -> CheckResult:
    """跑单个 check, 异常包装成 ERROR — 聚合模式承诺的兑现处。

    设计:
        - try/except Exception 是【唯一】合理使用宽泛捕获的地方。
          理由: check 是开闭原则下的扩展点, 谁都能加新 check, 调度层无法预知
          它会抛什么(KeyError / TypeError / 自定义异常 / ...)。 接住一切,
          包装成 ERROR 级 CheckResult, 让其他 check 继续。

        - exc_info=True: 把完整 traceback 写到日志, 方便事后排查 — 这条信息
          不会出现在 CheckResult.summary 里(避免污染用户看到的简短摘要),
          但保留在日志文件里, 调试时 grep 一下就有。
    """
    try:
        return entry.func(ctx)
    except Exception as e:
        logger.exception(f"check '{entry.name}' 抛异常, 已捕获为 ERROR 级结果")
        return CheckResult(
            name=entry.name,
            severity=CheckSeverity.ERROR,
            summary=f"check 内部异常: {type(e).__name__}: {e}",
            details={
                "exception_type": type(e).__name__,
                "exception_msg":  str(e),
            },
        )


def _log_check_result(r: CheckResult) -> None:
    """单个 check 跑完即时打一条日志, 让用户看到进度。

    Severity → log level 映射:
        ERROR   → logger.error
        WARNING → logger.warning
        INFO    → logger.info
        PASS    → logger.debug   (默认不显示, -v 才看)

    为什么 PASS 走 DEBUG:
        PASS 不是"事件", 是"非事件" — 健康数据集每次跑 4 行 [PASS] 刷屏,
        噪声盖过信号。 日志只记 WARN/ERROR/INFO 这种"有变化"的事件,
        PASS 信息走 ValidationReport 的最终摘要 (阶段 8 的 render.py),
        集中展示。
    """
    log_method = {
        CheckSeverity.ERROR:   logger.error,
        CheckSeverity.WARNING: logger.warning,
        CheckSeverity.INFO:    logger.info,
        CheckSeverity.PASS:    logger.debug,
    }.get(r.severity, logger.info)

    log_method(f"[{r.severity:7s}] {r.name}: {r.summary}")


def _log_summary(results: List[CheckResult]) -> None:
    """所有 check 跑完后, 打一行总览。"""
    counts = {}
    for r in results:
        counts[r.severity] = counts.get(r.severity, 0) + 1
    parts = [f"{n} {s}" for s, n in counts.items()]
    logger.info(f"check 执行完毕: {' / '.join(parts)}")
```

⚠️ **设计点 (try/except Exception 的位置)**: 整个 D4 子系统里, 这是**唯一**一处 `try/except Exception` (其他地方都用具体的 `except OSError` / `except yaml.YAMLError` 等)。

为什么唯一? **因为这里是"聚合模式承诺没人能阻断他人"的兑现处**——任何 check 函数自己写错了抛 `KeyError`, 没有这一层, **整个 `run_all_checks` 就被一个 check 顶死**——又退化成阶段 1 的反模式(早返回阻断后面 check)。

其他地方用具体的 except——是因为知道**该哪个异常**、知道**该怎么处理**。这里用 `Exception` 因为**不知道 check 会抛啥**(它是开闭原则下的扩展点, 谁都能加 check)——只能"接住一切, 包装成 ERROR 级 CheckResult"。

⚠️ **设计点 (PASS 走 debug 而不是 info)**: 这是后面阶段 8 富日志输出的伏笔。一个健康数据集每次跑都吐 4 行 `[PASS] yaml_schema / [PASS] pair_existence / ...` ——**信号被噪声淹没**, 用户实际想看的是 "数据集是什么样的" + "出了什么问题"。

我们把 PASS 移到 DEBUG 级(默认不显示), `WARN/ERROR/INFO` 即时走 logger, 完整的"三段式摘要"(数据集摘要 / 各 check 一览 / 失败详情)在阶段 8 的 `render.py` 集中展示。

> **金句**: **"日志只该记'变化'。健康事件不进日志——这是 systemd / k8s 都遵循的规则。"**

## 3.1 任务: 验证 yaml 文件本身

`yaml_schema` 关心:

1. yaml 文件存在
2. 内容能被 yaml 解析(不是损坏的语法)
3. yaml **顶层**是 `dict`(不是 list / scalar) —— **注意**: 这是说 yaml 文件根节点本身, 跟 `names` 字段无关。`names` 字段允许 list[str] 或 dict[int, str] 两种合法写法, 见检查项 5。
4. 包含 `nc` 字段, 正整数
5. 包含 `names` 字段, 是 list[str] 或 dict[int, str](ultralytics 接受两种)
6. `len(names) == nc`

**不**关心:

- train / val / test 路径是否存在(那是 pair_existence 的事)
- 类别 ID 是否越界(那是 label_format 的事)

**关注点分离**这条线在 D4 全程都要守住——单 check 只问自己该问的, 不越界。

# apps/platform/src/odp_platform/data_validation/checks/yaml_schema.py
"""yaml_schema check — 验证数据集 yaml 文件的字段完整性和一致性。

检查项 (任何一项失败都标 ERROR):
    1. yaml 文件存在且可解析
    2. yaml 顶层是 dict (不是 list / scalar) — 仅指文件根节点
    3. 包含 'nc' 字段, 正整数
    4. 包含 'names' 字段, list[str] 或 dict[int,str], 元素非空
    5. len(names) == nc

不检查的项 (后续 check 负责):
    - train / val / test 路径是否存在 → pair_existence
    - 类别 ID 是否越界 → label_format
"""
from __future__ import annotations

from typing import Any, Dict, List, Tuple

import yaml

from odp_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


@check("yaml_schema")
def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
    yaml_path = ctx.yaml_path

    # ---------- 第 1 类错: 文件不存在 ----------
    if not yaml_path.exists():
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 文件不存在: {yaml_path}",
            details={
                "reason":    "file_not_found",
                "yaml_path": str(yaml_path),
            },
        )

    # ---------- 第 2 类错: 解析失败 ----------
    try:
        with open(yaml_path, "r", encoding="utf-8") as f:
            cfg = yaml.safe_load(f)
    except yaml.YAMLError as e:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 解析失败: {e}",
            details={
                "reason":      "parse_error",
                "yaml_path":   str(yaml_path),
                "parse_error": str(e),
            },
        )
    except OSError as e:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 读取失败: {e}",
            details={
                "reason":    "read_error",
                "yaml_path": str(yaml_path),
                "os_error":  str(e),
            },
        )

    # ---------- 第 3 类错: 顶层不是 dict ----------
    if not isinstance(cfg, dict):
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 顶层不是 dict: {type(cfg).__name__}",
            details={
                "reason":     "not_dict",
                "actual_type": type(cfg).__name__,
            },
        )

    # ---------- 第 4-6 类错: 字段一致性 ----------
    problems: List[str] = []

    nc = cfg.get("nc")
    if not isinstance(nc, int) or nc <= 0:
        problems.append(f"nc 缺失或不是正整数: {nc!r}")
        nc = None   # 防止下面继续检查时炸

    names_raw = cfg.get("names")
    names_count, names_problem = _validate_names(names_raw)
    if names_problem:
        problems.append(names_problem)

    if nc is not None and names_count is not None and nc != names_count:
        problems.append(f"nc ({nc}) 跟 names 长度 ({names_count}) 不一致")

    if problems:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 字段不一致: {len(problems)} 处问题",
            details={
                "reason":   "field_inconsistency",
                "problems": problems,
                "nc":       nc,
                "names_count": names_count,
            },
        )

    # ---------- 全部通过 ----------
    return CheckResult(
        name="yaml_schema",
        severity=CheckSeverity.PASS,
        summary=f"yaml 字段一致 (nc={nc}, names_count={names_count})",
        details={
            "nc":          nc,
            "names_count": names_count,
        },
    )


# ============================================================
# 私有辅助
# ============================================================

def _validate_names(names_raw: Any) -> Tuple[Any, str]:
    """验证 names 字段, 返回 (count, problem_msg)。

    names 合法写法 (ultralytics 接受两种):
        list[str]:      [a, b, c]
        dict[int, str]: {0: a, 1: b, 2: c}    ← D3 yaml_writer 写的就是这种

    Returns:
        (names_count, problem_msg)
        - 成功: (len(names), "")
        - 失败: (None, "具体问题描述")
    """
    if isinstance(names_raw, list):
        if not names_raw:
            return None, "names 是空列表"
        if not all(isinstance(n, str) and n for n in names_raw):
            return None, "names 列表里存在非字符串或空字符串元素"
        return len(names_raw), ""

    if isinstance(names_raw, dict):
        if not names_raw:
            return None, "names 是空字典"
        if not all(isinstance(k, int) for k in names_raw.keys()):
            return None, "names 字典的 key 必须全部是 int"
        if not all(isinstance(v, str) and v for v in names_raw.values()):
            return None, "names 字典里存在非字符串或空字符串值"
        return len(names_raw), ""

    return None, f"names 不是合法的 list[str] / dict[int, str]: {type(names_raw).__name__}"
```

⚠️ **设计点 (problems 列表 vs 早返回)**: 注意 yaml_schema 在"字段一致性"这段, 收集多个 problem 进列表, 而不是发现第一个就立刻 return:

```python
problems: List[str] = []
nc = cfg.get("nc")
if not isinstance(nc, int) or nc <= 0:
    problems.append(...)
    nc = None

names_raw = cfg.get("names")
names_count, names_problem = _validate_names(names_raw)
if names_problem:
    problems.append(names_problem)
...
```

这是 D4 跟阶段 1 朴素版的根本区别——**check 内部也遵循"聚合不早返回"**: 用户的 yaml 同时缺 nc 和缺 names, 一次报齐, 不要让用户跑 3 遍才知道 3 个问题。

但**前置错误**(文件不存在 / 解析失败 / 顶层不是 dict)是早返回——因为这些错误之后**没法继续做字段一致性检查**。**"前置 vs 业务"两类错误的边界要分清**——前置必须早返回, 业务尽量收集。

# 阶段 4: 🧱 撞墙② + DatasetSnapshot 一次扫描

第一个真 check `yaml_schema` 写完了——但它**没碰盘**, 只 open 了一个 yaml 文件。下一个 check `pair_existence` 要**扫所有图像**——这是 D4 第一次大规模 IO 出现的地方。

## 4.1 撞墙②的形状: 设想 4 个 check 都自己扫盘

按照阶段 3 的写作范式, 我们准备写剩下 3 个 check:

- `pair_existence` — 遍历每个 split 的图像, 检查对应的 .txt 是否存在
- `label_format` — 遍历每个 .txt, 逐行验证字段数 / 类别 / 坐标
- `split_uniqueness` — 收集每个 split 的图像 stem, 算两两交集

**如果每个 check 自己 open yaml + 自己 glob 扫盘**——会出现两层重复:

**重复层 1: 解析 yaml**

```python
# pair_existence 内部
with open(yaml_path) as f: cfg = yaml.safe_load(f)
data_root = ...

# label_format 内部
with open(yaml_path) as f: cfg = yaml.safe_load(f)     # ← 又来一遍
data_root = ...                                         # ← 又来一遍

# split_uniqueness 内部
with open(yaml_path) as f: cfg = yaml.safe_load(f)     # ← 第三遍
data_root = ...                                         # ← 第三遍
```

3 次 yaml 解析。**算上 yaml_schema 自己那次, 4 次**。yaml 不大不算慢, 但**逻辑重复**:`data_root` 的解析规则(`yaml.path` 字段优先 / 相对路径相对 yaml 同级目录)在 4 个文件里各写一遍——明天改"支持环境变量展开"要改 4 处, 改漏一处就埋雷。

**重复层 2: 扫盘**

```python
# pair_existence: glob train/val/test 的所有图像
for ext in IMAGE_EXTENSIONS:
    for img in split_dir.glob(f"*{ext}"):
        ...

# label_format: 还得 glob 一次找标签 (要先列图像才知道对应的标签)
for ext in IMAGE_EXTENSIONS:
    for img in split_dir.glob(f"*{ext}"):
        ...

# split_uniqueness: 第三次 glob 收集 stem
for ext in IMAGE_EXTENSIONS:
    for img in split_dir.glob(f"*{ext}"):
        ...
```

10 万张图的数据集, 这是 **30 万次文件系统操作**。SSD 上几秒, 机械盘上几十秒——**纯浪费**。

⚠️ **关键观察 (重复 yaml 解析 = 信息漂移风险)**: 扫盘慢只是表象, **逻辑重复**才是核心问题。`data_root` 的解析规则、`images/<split>/foo.jpg → labels/<split>/foo.txt` 的路径推断, **这些规则散在 N 个文件里就会漂移**——某天某个 check 的作者忘了"yaml.path 可以是相对路径", 直接 `Path(cfg['path'])`, **从此那个 check 在 Windows 上跟其他 check 行为不一致**。

这是**比 IO 慢 100 倍可怕**的故障类型——它不会报错, 只会**让两个 check 看到的"数据集"不是同一个**。

## 4.2 解药形状: 共享只读快照

**只扫一次**, 把所有 check 都需要的素材(yaml 数据 / 各 split 的图像列表 / 标签列表 / 简单统计)装进一个共享对象, 然后传给每个 check 消费。

```
build_snapshot(yaml_path, task_type) → DatasetSnapshot {
    yaml_data:        Dict        ← 解析过的 yaml (供 yaml_schema 消费)
    yaml_load_error:  Optional[str] ← yaml 解析失败时的错误信息
    data_root:        Path        ← 解析过的 data_root (yaml.path / yaml 同级目录)
    nc / class_names: ...         ← yaml 解析后归一化的字段
    task_type:        str         ← 'detect' / 'segment'
    images_per_split: {split: [Path, ...]}     ← 每个 split 的图像列表
    labels_per_split: {split: [Path, ...]}     ← 对应的标签路径列表
    stats_per_split:  {split: SplitStats}      ← 顺手算的 3 个数字 per split
}
   ↓ 装进 CheckContext.snapshot
   ↓
service.run_all_checks(ctx)
   ├── yaml_schema(ctx)        ← 读 snapshot.yaml_data / nc / class_names
   ├── pair_existence(ctx)     ← 读 snapshot.images_per_split / labels_per_split
   ├── label_format(ctx)       ← 读 snapshot.labels_per_split / nc / task_type
   └── split_uniqueness(ctx)   ← 读 snapshot.images_per_split (各 split 的 stem)
```

所有 check 都变成**"接受 ctx, 读 ctx.snapshot, 返回 CheckResult"**——**check 内部不再 open / glob**, 只做"判定"。这让 check 文件从平均 100 行降到平均 50-70 行(IO 是冗余主力)。

## 4.3 DatasetSnapshot 三条铁律

snapshot 这个对象贵在**容易被修改**——它要被 4 个 check 不加防备地消费, 任何一个 check 不小心改了它都是调试灾难。所以它的设计有三条不可妥协的铁律:

### 铁律 1: 装事实, 不做判断

snapshot 只**装**数据, 不**做**业务判断。

```python
# ✓ 好品味
@dataclass(frozen=True)
class DatasetSnapshot:
    images_per_split: Dict[str, Tuple[Path, ...]]
    ...

# ✗ 别这么干
class DatasetSnapshot:
    def is_train_size_ok(self) -> bool:
        return len(self.images_per_split["train"]) > 100   # ← 业务判断爬进了数据载体
```

业务判断属于 check 函数, snapshot 只**提供素材**。这样 snapshot 自身永远不会"长出新业务"——它的边界**写完就定**。

### 铁律 2: frozen=True 防止 check 串改

用 `@dataclass(frozen=True)` 让 snapshot 不可变。

理由: snapshot 作为多个 check 的**共享只读视图**, 如果其中一个 check 不小心改了它(比如 append 进 images_per_split["train"]), 后面的 check 看到的就是被污染的数据——**调试灾难**。frozen 让这种错误立刻在写入处抛 `FrozenInstanceError`, 比"后面 check 行为诡异"好排查 10 倍。

注意:`frozen` 不阻止你修改字段**内部的可变对象**(比如 `dict[str, list]` 里那个 list 还是能 append)——所以 snapshot 里所有的容器都用**不可变形式**: `Tuple[Path, ...]` 而不是 `List[Path]`, `dict` 装进去的也是 `Tuple` 不是 `list`。**字面意义上的"装事实"——装完就不动了**。

### 铁律 3: best-effort, 不抛异常

`build_snapshot` 永远**不抛异常**——即使 yaml 解析失败、即使某个 split 目录不存在, 也要返回一个 snapshot 对象(可能 `yaml_load_error` 字段有值、可能 `images_per_split` 是空 dict)。

为什么? **调度层永远不该自己 fail**——

```python
# ✗ 反模式
def build_snapshot(yaml_path):
    with open(yaml_path) as f:    # ← yaml 解析失败这里炸
        cfg = yaml.safe_load(f)
    ...

# 上游调用:
try:
    snap = build_snapshot(...)
except Exception:
    # 然后呢? print("失败了")? 谁来报告这是 yaml_schema 该报的问题?
    ...
```

如果 build_snapshot 抛, 那 **`yaml_schema` 这个 check 永远没机会跑**——但 yaml 解析失败本来就是 `yaml_schema` 该报告的事!这条信息要变成一个 ERROR 级 CheckResult 给用户看, 而不是 build_snapshot 内部一个 traceback。

正解: yaml 解析失败时, 把错误字符串装进 `snapshot.yaml_load_error`, 然后由 `yaml_schema` 消费这个字段报错。**所有"扫描中遇到的非致命问题"都装进 snapshot 让 check 自己拿去判定**。

> **金句**: **"一次性把'我的数据是怎么来的'信息固化进数据对象本身——比任何文档 / 注释 / 命名约定都强。"**

## 4.4 SplitStats: 轻量统计 (3 个数字)

snapshot 顺手算每个 split 的**三个数字**, 装一个小数据类:

```python
@dataclass(frozen=True)
class SplitStats:
    """单个 split 的轻量统计 — 用于 ValidationReport 数据集摘要。"""
    image_count:     int   # 该 split 的图像总数
    annotated_count: int   # 有非空标签的图像数 (= image_count - 无标签 - 空标签)
    total_instances: int   # 全部 bbox 实例数 (label 文件总行数)
```

为什么是这三个数字, 而不是按类别细分 / 不算 bbox 面积:

- **够"看一眼知道数据集啥规模"**: 14823 张图 / 14801 标注 / 52318 实例——直接对比就能看出"标注覆盖率 99.8%"、"平均每张图 3.5 个实例"这种宏观体感
- **不参与任何 check 判定**: SplitStats 只给最终报告展示用(阶段 8 的 render), 任何 check 都不读它来做"严重程度"判断
- **顺手算几乎零成本**: snapshot 内部循环本来就要 `for img in images`, 多一行 `count += 1` 不需要新增 IO

⚠️ **设计点 (顺手做的边界)**: 顺手统计不等于"snapshot 做业务判断"。snapshot 装的是**事实**(每个 split 多少张图、多少标注、多少实例), **业务判定**(标注覆盖率不足是不是问题)仍在 check。这条边界要守住——否则 snapshot 长成什么都干的 god object。

**保持 3 个字段**——不要加"按类别计数 ClassStat"、不要加"bbox 面积 / 宽高比"。如果后续真要做这些细分统计, **加新 check 加新文件**, 不要污染 snapshot。

> **金句**: **"'顺手做'的红利在 I/O 密集型流程里几乎免费——但每次顺手都要问'这个属于装数据还是做判断'。"**

```python
# apps/platform/src/odp_platform/data_validation/snapshot.py
"""DatasetSnapshot — 一次扫描, 多次复用。

设计原则:
    - 装数据, 不做判断 (业务判断属于 check 函数)
    - frozen=True 防止 check 串改造成调试灾难
    - best-effort: yaml 解析失败 / split 目录不存在都不抛, 装进 snapshot
      让 check 自己拿去判定 — 调度层永远不该自己 fail
"""
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Dict, List, Optional, Tuple

import yaml

from odp_platform.common.constants import IMAGE_EXTENSIONS, Task
from odp_platform.common.performance_utils import time_it

logger = logging.getLogger(__name__)


# ============================================================
# SplitStats — 轻量统计 (3 个数字, 供报告展示)
# ============================================================

@dataclass(frozen=True)
class SplitStats:
    """单个 split 的轻量统计 — 用于 ValidationReport 数据集摘要。

    刻意只装 3 个数字 (不分类别 / 不算 bbox 面积):
        够"看一眼知道数据集啥规模", 不越界做分布分析。
        细分统计如果需要, 加独立的 check, 不污染 snapshot。
    """
    image_count:     int    # 该 split 的图像总数
    annotated_count: int    # 有非空标签的图像数
    total_instances: int    # 全部 bbox 实例数 (label 文件总行数)


# ============================================================
# DatasetSnapshot — 不可变数据载体
# ============================================================

@dataclass(frozen=True)
class DatasetSnapshot:
    """对一个数据集做一次完整快照, 供所有 check 共享消费。

    所有路径已经解析为绝对路径; 调用方不需要再考虑相对路径 / cwd 问题。

    Args:
        yaml_path:        数据集 yaml 文件绝对路径
        yaml_data:        原始 yaml 解析结果 (dict, 供 yaml_schema 直接判定)
        yaml_load_error:  yaml 解析过程中的错误信息 (None = 解析成功)
                          注意: 解析失败时 yaml_data 是 {}, 但 snapshot 仍然构造
                          成功 — 让 yaml_schema check 拿着 yaml_load_error 报错,
                          而不是 build_snapshot 自己抛异常
        data_root:        数据集根目录 (yaml 里 'path' 字段, 或 yaml 同级目录兜底)
        nc:               类别数 (yaml.nc; 缺失或非法时 None)
        class_names:      类别名 (按 ID 升序的 tuple; 缺失时空 tuple)
        task_type:        'detect' / 'segment'
        images_per_split: {split_name: (图像 Path, ...)} 绝对路径, 已排序
        labels_per_split: {split_name: (预期标签 Path, ...)} 按 stem 一对一对应
                          注意: 此处只是"按图像名构造期望的 .txt 路径", 不保证
                          文件存在 — pair_existence check 才负责存在性
        stats_per_split:  {split_name: SplitStats} 每个 split 的 3 个数字
        scan_warnings:    扫描过程中的非致命警告 (split 目录不存在等)
    """
    yaml_path:        Path
    yaml_data:        Dict[str, Any]
    yaml_load_error:  Optional[str]
    data_root:        Path
    nc:               Optional[int]
    class_names:      Tuple[str, ...]
    task_type:        str
    images_per_split: Dict[str, Tuple[Path, ...]]
    labels_per_split: Dict[str, Tuple[Path, ...]]
    stats_per_split:  Dict[str, SplitStats]
    scan_warnings:    Tuple[str, ...] = field(default_factory=tuple)

    # ---------- 派生属性 (不存值) ----------

    @property
    def splits(self) -> Tuple[str, ...]:
        """返回快照里实际有图像的 split 名 (按 train/val/test 顺序)。"""
        order = ("train", "val", "test")
        return tuple(s for s in order if s in self.images_per_split)

    @property
    def total_images(self) -> int:
        return sum(len(imgs) for imgs in self.images_per_split.values())


# ============================================================
# 内部辅助 — IO / 解析 (失败不抛)
# ============================================================

def _load_yaml(yaml_path: Path) -> Tuple[Dict[str, Any], Optional[str]]:
    """加载 yaml, 返回 (data, error_str)。失败时返回 ({}, err)。"""
    if not yaml_path.exists():
        return {}, f"yaml 文件不存在: {yaml_path}"
    try:
        with open(yaml_path, "r", encoding="utf-8") as f:
            data = yaml.safe_load(f) or {}
        if not isinstance(data, dict):
            return {}, f"yaml 顶层不是 dict: {type(data).__name__}"
        return data, None
    except yaml.YAMLError as e:
        return {}, f"yaml 解析失败: {e}"
    except OSError as e:
        return {}, f"yaml 读取失败: {e}"


def _resolve_data_root(yaml_path: Path, yaml_data: Dict[str, Any]) -> Path:
    """三种合法写法:
       1. yaml.path 是绝对路径 → 用它
       2. yaml.path 是相对路径 → 相对 yaml 文件所在目录
       3. yaml.path 缺失 → 用 yaml 文件所在目录
    """
    path_str = yaml_data.get("path")
    if not path_str:
        return yaml_path.parent.resolve()
    p = Path(path_str)
    return p.resolve() if p.is_absolute() else (yaml_path.parent / p).resolve()


def _resolve_split_dir(data_root: Path, split_field: Any) -> Optional[Path]:
    """yaml 里的 train/val/test 字段, 只支持字符串形式 (ultralytics 主流写法)。
    其他形式 (list / None) 返回 None, 让 yaml_schema 后续负责报错。
    """
    if not isinstance(split_field, str) or not split_field.strip():
        return None
    p = Path(split_field)
    return p.resolve() if p.is_absolute() else (data_root / p).resolve()


def _list_images(split_dir: Path) -> List[Path]:
    """返回 split_dir 下所有支持扩展名的图像 (绝对路径, 已排序)。"""
    if not split_dir.exists() or not split_dir.is_dir():
        return []
    images: List[Path] = []
    for ext in IMAGE_EXTENSIONS:
        images.extend(split_dir.glob(f"*{ext}"))
        images.extend(split_dir.glob(f"*{ext.upper()}"))   # Windows 大小写
    return sorted(set(images))


def _label_path_for_image(image_path: Path) -> Path:
    """YOLO 默认布局: images/<split>/foo.jpg → labels/<split>/foo.txt

    倒着找最后一个 'images' 替换为 'labels' (路径里可能多次出现 'images',
    取最后一个最稳妥)。
    """
    parts = list(image_path.parts)
    for i in range(len(parts) - 1, -1, -1):
        if parts[i] == "images":
            parts[i] = "labels"
            break
    return Path(*parts[:-1]) / (image_path.stem + ".txt")


def _normalize_names(names_raw: Any) -> Tuple[str, ...]:
    """把 yaml.names 归一化成"按 ID 升序的 tuple"。

    输入合法形式:
        - list[str]:      ['a','b','c']        → ('a','b','c')
        - dict[int,str]:  {0:'a',1:'b',2:'c'}  → ('a','b','c')

    非法或缺失返回空 tuple — 由 yaml_schema 负责报错。
    """
    if isinstance(names_raw, list):
        if all(isinstance(n, str) for n in names_raw):
            return tuple(names_raw)
        return ()
    if isinstance(names_raw, dict):
        if all(isinstance(k, int) for k in names_raw.keys()) \
                and all(isinstance(v, str) for v in names_raw.values()):
            return tuple(v for _, v in sorted(names_raw.items()))
        return ()
    return ()


def _build_split_stats(labels: List[Path]) -> SplitStats:
    """顺手算 3 个数字 — 在 build_snapshot 已经在循环图像了, 多读一次 label 也免不了"""
    image_count     = len(labels)
    annotated_count = 0
    total_instances = 0

    for lbl in labels:
        if not lbl.exists():
            continue
        try:
            content = lbl.read_text(encoding="utf-8")
        except OSError:
            continue
        lines = [l for l in content.splitlines() if l.strip()]
        if not lines:
            continue   # 空 label 不算 annotated
        annotated_count += 1
        total_instances += len(lines)

    return SplitStats(
        image_count=image_count,
        annotated_count=annotated_count,
        total_instances=total_instances,
    )


# ============================================================
# 公开 API: build_snapshot
# ============================================================

@time_it(name="build_snapshot")
def build_snapshot(
    yaml_path: Path,
    task_type: Optional[str] = None,
) -> DatasetSnapshot:
    """构造数据集快照, 一次扫描提供后续 check 全部所需素材。

    错误处理原则: best-effort —— yaml 解析失败 / split 目录不存在都只装进
    snapshot 的 yaml_load_error / scan_warnings, 不抛异常。由对应 check
    负责把这些信号转成 CheckResult。

    Args:
        yaml_path: 数据集 yaml 路径
        task_type: 'detect' / 'segment'。优先级:
                   显式参数 > yaml.task 字段 > Task.DETECT 兜底

    Returns:
        DatasetSnapshot (永远不抛异常)
    """
    yaml_path = yaml_path.resolve()
    warnings: List[str] = []

    yaml_data, yaml_err = _load_yaml(yaml_path)
    if yaml_err:
        warnings.append(yaml_err)

    data_root = _resolve_data_root(yaml_path, yaml_data)

    nc          = yaml_data.get("nc") if isinstance(yaml_data.get("nc"), int) else None
    class_names = _normalize_names(yaml_data.get("names"))

    # task_type 优先级: 显式 > yaml.task > 兜底 detect
    resolved_task = task_type or yaml_data.get("task") or Task.DETECT
    if resolved_task not in (Task.DETECT, Task.SEGMENT):
        warnings.append(f"未知 task_type '{resolved_task}', 回退到 '{Task.DETECT}'")
        resolved_task = Task.DETECT

    images_per_split: Dict[str, Tuple[Path, ...]] = {}
    labels_per_split: Dict[str, Tuple[Path, ...]] = {}
    stats_per_split:  Dict[str, SplitStats]       = {}

    for split in ("train", "val", "test"):
        split_dir = _resolve_split_dir(data_root, yaml_data.get(split))
        if split_dir is None or not split_dir.exists():
            if split in yaml_data:
                warnings.append(f"split '{split}' 目录不可用: {split_dir}")
            continue

        images = _list_images(split_dir)
        if not images:
            warnings.append(f"split '{split}' 目录下无图像: {split_dir}")
            continue

        labels = [_label_path_for_image(img) for img in images]

        images_per_split[split] = tuple(images)
        labels_per_split[split] = tuple(labels)
        stats_per_split[split]  = _build_split_stats(labels)

    snapshot = DatasetSnapshot(
        yaml_path=yaml_path,
        yaml_data=yaml_data,
        yaml_load_error=yaml_err,
        data_root=data_root,
        nc=nc,
        class_names=class_names,
        task_type=resolved_task,
        images_per_split=images_per_split,
        labels_per_split=labels_per_split,
        stats_per_split=stats_per_split,
        scan_warnings=tuple(warnings),
    )
    logger.info(
        f"snapshot 构建完成: {snapshot.total_images} 张图像, "
        f"splits={list(snapshot.splits)}, task={resolved_task}"
    )
    return snapshot
```

snapshot.py 总长 ~220 行——稍稍超 200, 但**没有一个函数超过 50 行**, 各司其职清晰可读。这就是"合理节制"——不死守 200 这个数字, 但单文件可"翻几下能看完"。

⚠️ **设计点 (Tuple 而不是 List)**: 注意 snapshot 里所有"列表"都是 `Tuple[X, ...]`, 不是 `List[X]`。

理由跟铁律 2 一样: `frozen=True` 只阻止重新赋值 snapshot 的字段, **不阻止你 `snapshot.images_per_split['train'].append(x)`** —— 如果用 list 装,这个 append 是合法的, 后续 check 就看到被污染的数据。改用 tuple, 这个 append 直接 `AttributeError`——错误立刻可见。

⚠️ **设计点 (task_type 三级优先级)**:

```python
resolved_task = task_type or yaml_data.get("task") or Task.DETECT
```

显式参数(CLI `--task`)> yaml 文件里写的 `task:` 字段 > 默认 `Task.DETECT`。

为什么是这个顺序? **越靠近用户的越优先**——用户在 CLI 上手敲的 `--task` 一定是"我现在就想用这个"的意图, 比"yaml 里写了"更新鲜; 而 yaml 里写的又比"默认值"更具体。如果反过来"yaml 优先于参数", 用户就没法"用 CLI 临时切 task 来排查问题"——这是常见的"灵活性 vs 严格性"权衡, **临时性 > 持久性 > 默认**是 Unix 工具的惯例(`alias` > `~/.bashrc` > 系统默认)。

    阶段 2 立的字段:
        yaml_path : 数据集 yaml 文件路径

这是 D4 同时也是 **CheckContext 兑现"扩展位"承诺的第一次** —— 阶段 2 我们说"将来扩字段不改 check 签名", 阶段 4 真扩了, 阶段 2 的 yaml_schema 函数签名一行没动, 只是开始从 ctx 里取 snapshot 用而已。

## 4.7 yaml_schema 重构: 消费 snapshot, 不自己 open yaml

snapshot 既然已经把 yaml 解析过了, yaml_schema check 应该**直接消费 snapshot**, 不再自己 open/parse yaml。改 `checks/yaml_schema.py`:

```python
# apps/platform/src/odp_platform/data_validation/checks/yaml_schema.py
"""yaml_schema check — 验证数据集 yaml 文件的字段完整性和一致性。

阶段 4 重构: 不再自己 open / parse yaml, 改成消费 snapshot.yaml_data /
yaml_load_error / nc / class_names。代码从 ~150 行缩到 ~80 行 —— 收益来自
"只判断不做 IO"。
"""
from __future__ import annotations

from typing import List

from odp_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


@check("yaml_schema")
def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    # ---------- 前置错: yaml 加载阶段就出问题了 ----------
    if snap.yaml_load_error is not None:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=snap.yaml_load_error,
            details={
                "reason":          "yaml_load_error",
                "yaml_path":       str(snap.yaml_path),
                "yaml_load_error": snap.yaml_load_error,
            },
        )

    # ---------- 业务错: 字段一致性 (收集所有, 一次报齐) ----------
    problems: List[str] = []

    nc = snap.nc
    if nc is None or nc <= 0:
        problems.append(f"nc 缺失或不是正整数: {snap.yaml_data.get('nc')!r}")

    names = snap.class_names
    if not names:
        # _normalize_names 返回空 tuple 表示"非法或缺失"
        raw = snap.yaml_data.get("names")
        problems.append(f"names 缺失或不是合法的 list[str] / dict[int,str]: {type(raw).__name__}")

    if nc is not None and nc > 0 and names and len(names) != nc:
        problems.append(f"nc ({nc}) 跟 names 长度 ({len(names)}) 不一致")

    if problems:
        return CheckResult(
            name="yaml_schema",
            severity=CheckSeverity.ERROR,
            summary=f"yaml 字段不一致: {len(problems)} 处问题",
            details={
                "reason":      "field_inconsistency",
                "problems":    problems,
                "nc":          nc,
                "names_count": len(names) if names else 0,
            },
        )

    # ---------- 全部通过 ----------
    return CheckResult(
        name="yaml_schema",
        severity=CheckSeverity.PASS,
        summary=f"yaml 字段一致 (nc={nc}, names_count={len(names)})",
        details={
            "nc":          nc,
            "names_count": len(names),
        },
    )
```

代码从 ~150 行缩到 ~80 行——直接看出**"不做 IO, 只做判断"的红利**。

# 阶段 5: pair_existence + 按比例分级 severity

第二个真 check: **图像和标签必须成对存在**——这是 YOLO 训练对数据集的硬要求。但跟 yaml_schema 全部 ERROR 不同, pair_existence 要**按缺失比例分级 severity**——这是 D4 第一次出现"同一个 check 的失败有多种紧张度"。

## 5.1 任务: 检查图像-标签成对

数据集里, 每个 `images/train/foo.jpg` 都应该有对应的 `labels/train/foo.txt` (允许空文件, 表示无目标图像; 但**不能不存在**)。

但**缺多少算严重**?

- 缺 0.1% (1000 张图缺 1 张) → 标注遗漏, 工程上能容忍
- 缺 5% → 该警告, 影响训练精度
- 缺 60% → 流程级问题, 训不出来

**一刀切 ERROR 太严**, **一刀切 WARNING 太松**——按比例分级最合理。这就是 D4 第一次出现的**业务 check 内部分级 severity**。

D4 阶段 0 立的两个阈值正好用上:

```python
PAIR_MISSING_ERROR_RATIO: float = 0.5   # >= 50%: ERROR
PAIR_MISSING_WARN_RATIO:  float = 0.05  # 5% ~ 50%: WARNING
                                         # 0% ~ 5%:  INFO
                                         # 0%:        PASS
```

## 5.2 checks/pair_existence.py

```python
# apps/platform/src/odp_platform/data_validation/checks/pair_existence.py
"""pair_existence check — 验证每张图都有对应的 .txt 标签文件。

Severity 按缺失比例分级:
    missing_ratio == 0.0           → PASS
    missing_ratio <  WARN_RATIO    → INFO    (个别遗漏, 可容忍)
    missing_ratio <  ERROR_RATIO   → WARNING (显著遗漏, 影响精度)
    missing_ratio >= ERROR_RATIO   → ERROR   (流程级问题, 不允许训练)

注: 空 .txt 文件算"存在"且合法 (YOLO 接受空标签表示无目标图像)。
    这里只判定"文件是否存在", 不判定"是否为空"。
"""
from __future__ import annotations

from typing import Any, Dict, List

from odp_platform.common.constants import (
    PAIR_MISSING_ERROR_RATIO,
    PAIR_MISSING_WARN_RATIO,
)
from odp_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


# 详情里展示的"前 N 个缺失文件"上限 — 避免 details 爆炸
DETAILS_PREVIEW_LIMIT = 20


@check("pair_existence")
def validate_pair_existence(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if not snap.images_per_split:
        return CheckResult(
            name="pair_existence",
            severity=CheckSeverity.INFO,
            summary="无任何 split 可检查 (snapshot 为空)",
            details={"reason": "empty_snapshot"},
        )

    # 收集每个 split 的孤儿图像 (有图无标签)
    orphan_per_split: Dict[str, List[str]] = {}
    total_images  = 0
    total_missing = 0

    for split, images in snap.images_per_split.items():
        labels = snap.labels_per_split.get(split, ())
        missing_in_split: List[str] = []
        # build_snapshot 保证两者长度一致 + 顺序一致 (按图像名构造标签路径)
        for img, lbl in zip(images, labels):
            total_images += 1
            if not lbl.exists():
                total_missing += 1
                missing_in_split.append(str(img))
        if missing_in_split:
            orphan_per_split[split] = missing_in_split

    missing_ratio = total_missing / max(total_images, 1)

    # ---------- 按比例分级 severity (从最严到最松写) ----------
    if total_missing == 0:
        severity = CheckSeverity.PASS
        summary  = f"全部 {total_images} 张图像都有对应标签"
    elif missing_ratio >= PAIR_MISSING_ERROR_RATIO:
        severity = CheckSeverity.ERROR
        summary = (
            f"缺标签比例 {missing_ratio:.1%} ≥ {PAIR_MISSING_ERROR_RATIO:.0%} "
            f"({total_missing}/{total_images} 张图无标签)"
        )
    elif missing_ratio >= PAIR_MISSING_WARN_RATIO:
        severity = CheckSeverity.WARNING
        summary = (
            f"缺标签比例 {missing_ratio:.1%} ≥ {PAIR_MISSING_WARN_RATIO:.0%} "
            f"({total_missing}/{total_images} 张图无标签)"
        )
    else:
        severity = CheckSeverity.INFO
        summary = (
            f"少量标签缺失 ({total_missing}/{total_images} = {missing_ratio:.2%})"
        )

    # ---------- 构造 details (含 preview 截断防止报告爆炸) ----------
    details: Dict[str, Any] = {
        "total_images":  total_images,
        "total_missing": total_missing,
        "missing_ratio": round(missing_ratio, 4),
        "thresholds": {
            "error_at": PAIR_MISSING_ERROR_RATIO,
            "warn_at":  PAIR_MISSING_WARN_RATIO,
        },
        "missing_per_split": {
            split: len(orphans) for split, orphans in orphan_per_split.items()
        },
    }

    # 详细清单截断 — 前 N 条供日志展开, 完整列表通过 JSON 报告查看
    if orphan_per_split:
        details["missing_examples"] = {
            split: orphans[:DETAILS_PREVIEW_LIMIT]
            for split, orphans in orphan_per_split.items()
        }

    return CheckResult(
        name="pair_existence",
        severity=severity,
        summary=summary,
        details=details,
    )
```

## 5.3 写作范式细节

这个 check 比 yaml_schema 多了几处写法值得点出来——后面的 check 都会按这套范式来。

### 范式 1: 按比例分级的 if/elif 链, 从最严到最松写

```python
if total_missing == 0:
    severity = PASS
elif missing_ratio >= PAIR_MISSING_ERROR_RATIO:
    severity = ERROR
elif missing_ratio >= PAIR_MISSING_WARN_RATIO:
    severity = WARNING
else:
    severity = INFO
```

**从最严到最松写**, 而不是从松到严——因为 if/elif 是短路的, 先判最严的情况, 后面默认走 INFO。**反着写**(从松到严)就要 `not` + 复合条件, 可读性差很多。

### 范式 2: DETAILS_PREVIEW_LIMIT 防爆

如果 1 万张图全缺标签, `details["missing_examples"]["train"]` 就是 1 万条路径——日志里塞不下, JSON 报告也丑得没法看。

```python
DETAILS_PREVIEW_LIMIT = 20

details["missing_examples"] = {
    split: orphans[:DETAILS_PREVIEW_LIMIT]   # 只保留前 20 条
    for split, orphans in orphan_per_split.items()
}
```

`missing_per_split` 字段给出**总数**, `missing_examples` 给出**前 20 条示例**——总数永远准, 示例够定位即可。完整清单不是不能给, 但**不在 details 里塞**——展示层(阶段 8)的事, details 是数据契约。

### 范式 3: details 字段尽量"机器可读 + 结构化"

```python
details = {
    "total_images":  total_images,
    "total_missing": total_missing,
    "missing_ratio": round(missing_ratio, 4),
    "thresholds": {...},
    "missing_per_split": {...},
    "missing_examples":  {...},
}
```

每个字段都是**单独可查询的语义单元**, 不是"一句字符串"。监控趋势图想画"缺失比例随时间的变化", 直接 `report["pair_existence"]["details"]["missing_ratio"]`——不用 regex 解析。

# 阶段 6: label_format + task_type 走 snapshot 字段

第三个真 check: **每行 .txt 的格式合法**。这个 check 跟前两个不一样——它**需要区分 task_type**(detect 5 字段 / segment 1+2N 字段), 所以会演示"check 怎么消费 snapshot 里的 task_type"。

## 6.1 任务: 验证 label 行格式

**detect 任务** 每行格式: `class_id cx cy w h` (5 个字段)
- `class_id`: 整数, 范围 `[0, nc)`
- `cx, cy, w, h`: 浮点, 范围 `[0, 1]` (归一化坐标)

**segment 任务** 每行格式: `class_id x1 y1 x2 y2 ... xN yN` (1 + 2N 个字段)
- `class_id`: 整数, 范围 `[0, nc)`
- `(x1,y1), ..., (xN,yN)`: 多边形顶点, 浮点, 范围 `[0, 1]`
- N >= 3 (至少三角形)

**两种 task 一个 check 里都处理**——因为本质上判定的是"一行 .txt 字段是否合法", task_type 只是分支条件。

## 6.2 错误分级: 不分级, 任何错就 ERROR

label_format 是**结构性**问题——1 行格式错就意味着 YOLO 训练在加载时会从那行 raise。**不像 pair_existence "少量缺失可容忍"**, label 格式不存在"小问题"——一行错就是一个不能训的样本。

所以 label_format 的设计是:

- 全过 → PASS
- 任何错 → ERROR

但 details 里**收集所有错的类型 + 截断前 N 条样例**——跟 pair_existence 一样, 详细字段给消费者, 截断给日志。

## 6.3 checks/label_format.py

```python
# apps/platform/src/odp_platform/data_validation/checks/label_format.py
"""label_format check — 验证每行 .txt 的格式合法性。

支持两种 task (从 snapshot.task_type 读):
    detect:  每行 'cls cx cy w h'                 (5 字段)
    segment: 每行 'cls x1 y1 x2 y2 ... xN yN'    (1+2N 字段, N>=3)

判定:
    - 全通过 → PASS
    - 任何一行错 → ERROR (不分级 — label 格式是结构性问题)

details 收集:
    - error_kinds:    错误类型计数 (字段数 / 类别越界 / 坐标越界 / 多边形点数不足)
    - errors_preview: 前 N 条具体错误样例 (label 文件名 + 行号 + 错误类型)
"""
from __future__ import annotations

from collections import Counter
from typing import Any, Dict, List, Optional

from odp_platform.common.constants import Task
from odp_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


ERRORS_PREVIEW_LIMIT = 20

# 错误类型常量 (机器可读, 用于 details["error_kinds"] 计数)
KIND_FIELD_COUNT_MISMATCH = "field_count_mismatch"
KIND_PARSE_ERROR          = "parse_error"
KIND_CLASS_ID_OUT_OF_RANGE = "class_id_out_of_range"
KIND_COORD_OUT_OF_RANGE   = "coord_out_of_range"
KIND_POLYGON_TOO_FEW      = "polygon_too_few_points"


@check("label_format")
def validate_label_format(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if snap.nc is None or snap.nc <= 0:
        # 没有合法 nc, 没法验证 class_id 越界 — 留给 yaml_schema 报错, 我们 INFO 跳过
        return CheckResult(
            name="label_format",
            severity=CheckSeverity.INFO,
            summary="缺少合法 nc, 跳过 label_format (yaml_schema 应已报告)",
            details={"reason": "nc_unavailable"},
        )

    task_type = snap.task_type
    errors: List[Dict[str, Any]] = []
    error_kinds: Counter = Counter()
    total_lines = 0

    for split, labels in snap.labels_per_split.items():
        for lbl in labels:
            if not lbl.exists():
                continue   # pair_existence 报
            try:
                content = lbl.read_text(encoding="utf-8")
            except OSError:
                continue   # IO 错也跳过 — best effort
            for line_no, line in enumerate(content.splitlines(), 1):
                line = line.strip()
                if not line:
                    continue   # 空行允许
                total_lines += 1
                err = _validate_one_line(line, task_type, snap.nc)
                if err is not None:
                    kind, detail = err
                    error_kinds[kind] += 1
                    if len(errors) < ERRORS_PREVIEW_LIMIT:
                        errors.append({
                            "label":   str(lbl),
                            "line_no": line_no,
                            "kind":    kind,
                            "detail":  detail,
                        })

    if not error_kinds:
        return CheckResult(
            name="label_format",
            severity=CheckSeverity.PASS,
            summary=f"全部 {total_lines} 行标签格式正确 (task={task_type})",
            details={
                "task_type":   task_type,
                "total_lines": total_lines,
            },
        )

    total_errors = sum(error_kinds.values())
    return CheckResult(
        name="label_format",
        severity=CheckSeverity.ERROR,
        summary=f"{total_errors}/{total_lines} 行标签格式错误 (task={task_type})",
        details={
            "task_type":       task_type,
            "total_lines":     total_lines,
            "total_errors":    total_errors,
            "error_kinds":     dict(error_kinds),
            "errors_preview":  errors,
        },
    )


# ============================================================
# 私有: 验证单行
# ============================================================

def _validate_one_line(line: str, task_type: str, nc: int):
    """验证单行 .txt, 返回 (kind, detail) 或 None (通过)。"""
    parts = line.split()

    # ---------- detect ----------
    if task_type == Task.DETECT:
        if len(parts) != 5:
            return KIND_FIELD_COUNT_MISMATCH, f"detect 任务要求 5 字段, 实际 {len(parts)}"
        try:
            cls_id = int(parts[0])
            coords = [float(x) for x in parts[1:5]]
        except ValueError as e:
            return KIND_PARSE_ERROR, f"字段类型错: {e}"
        if not (0 <= cls_id < nc):
            return KIND_CLASS_ID_OUT_OF_RANGE, f"cls_id={cls_id} 不在 [0,{nc}) 内"
        if not all(0.0 <= c <= 1.0 for c in coords):
            bad = [round(c, 4) for c in coords if not (0.0 <= c <= 1.0)]
            return KIND_COORD_OUT_OF_RANGE, f"坐标越界 [0,1]: {bad}"
        return None

    # ---------- segment ----------
    if task_type == Task.SEGMENT:
        # 至少: cls + 3 对 (x,y) = 1 + 6 = 7 字段
        if len(parts) < 7 or (len(parts) - 1) % 2 != 0:
            if len(parts) < 7:
                return KIND_POLYGON_TOO_FEW, f"segment 任务多边形至少 3 个点 (7 字段), 实际 {len(parts)}"
            return KIND_FIELD_COUNT_MISMATCH, f"segment 任务字段数应为 1+2N, 实际 {len(parts)} (N 非整)"
        try:
            cls_id = int(parts[0])
            coords = [float(x) for x in parts[1:]]
        except ValueError as e:
            return KIND_PARSE_ERROR, f"字段类型错: {e}"
        if not (0 <= cls_id < nc):
            return KIND_CLASS_ID_OUT_OF_RANGE, f"cls_id={cls_id} 不在 [0,{nc}) 内"
        if not all(0.0 <= c <= 1.0 for c in coords):
            bad_count = sum(1 for c in coords if not (0.0 <= c <= 1.0))
            return KIND_COORD_OUT_OF_RANGE, f"{bad_count}/{len(coords)} 个坐标越界 [0,1]"
        return None

    # 未知 task_type — build_snapshot 已经兜底过了, 这里再防一层
    return KIND_PARSE_ERROR, f"未知 task_type: {task_type}"
```

## 6.4 设计点

⚠️ **设计点 (task_type 走 snapshot 字段)**: 注意 check 没有自己解析 yaml.task / 没有接受 task_type 参数——**只 `snap.task_type` 一行取出来用**。

这就是阶段 4 的 build_snapshot 里那段三级优先级(显式参数 > yaml.task > 默认)的兑现处——所有"task_type 这个状态是怎么来的"复杂度集中在 build_snapshot 一处, **check 完全不操心**, 它信任 ctx.snapshot.task_type 是"已解析好的最终值"。

如果以后增加第三种 task(姿态估计 pose / 旋转框 obb), 改 `_validate_one_line` 加一个 `if task_type == Task.POSE:` 分支即可——**判定逻辑增长**, 但 snapshot / 调度层 / 其他 check 一行不动。这是"判定下沉到 check"设计的开闭红利。

⚠️ **设计点 (错误类型用常量, 不是字符串字面量)**:

```python
KIND_FIELD_COUNT_MISMATCH = "field_count_mismatch"
KIND_PARSE_ERROR          = "parse_error"
...

error_kinds[KIND_FIELD_COUNT_MISMATCH] += 1
```

为什么不直接 `error_kinds["field_count_mismatch"] += 1`?

- 字符串字面量散在 5 个地方, 改一个名字要 grep 全文
- 拼写错(`field_count_mistmatch`)IDE 不会警告——常量名拼错 IDE 立刻飘红
- 消费方(JSON 报告 / 监控)读 `error_kinds` 字段时, 这些 key 是**对外 API**——集中定义就是对 API 的版本管理

**5 行常量声明 = 5 个潜在 bug 提前死**。

⚠️ **设计点 (errors_preview 截断在收集时而不是构造 details 时)**:

```python
if len(errors) < ERRORS_PREVIEW_LIMIT:
    errors.append({...})    # ← 收集时就截断
```

而不是:

```python
errors.append({...})        # 全收集
...
details["errors_preview"] = errors[:ERRORS_PREVIEW_LIMIT]   # ← 用时再截
```

理由: 1 万张图全错的极端情况下, **不截断收集**会在内存里攒 1 万个 dict——白浪费。`error_kinds` 计数仍然准(每错一次 `+1`), 只是不再收集第 21 个之后的样本——**计数 vs 样本是两个独立的语义**, 一个准一个截断, 各司其职。

## 7.1 任务: 用 stem 判重, 不 hash 文件内容

"重复"怎么定义?

**方案 A: 按文件内容 hash 判重**——把每张图 MD5 一遍, 对比 hash 是否重复。
**方案 B: 按图像 stem (文件名不含扩展名) 判重**——`train/foo.jpg` 跟 `val/foo.png` 视为重复。

D4 选 **B**。理由:

1. **YOLO 训练时按文件名找 label**——`foo.jpg` 对应 `foo.txt`。如果 train 和 val 都有 `foo.txt`, **训练时哪个 label 生效是未定义行为**, 即使图像内容不同, 训练也会出错。判重的本质是"按消费方的视角", train 和 val 同名就是**事实上的冲突**。
2. **方案 A 太重**——10 万张图 MD5 慢且没必要; 真要查"内容相同的图", 那是另一个 check (`image_dedup`) 的事。
3. **业界惯例**——ultralytics 自己的数据集校验工具也是按 stem 判重。

⚠️ **设计点 (判重的边界 = 消费方的视角)**:

> **金句**: **"判重的本质是'按消费方的视角'——不是按'最严格'。"**

如果你脑子里想的是"两张完全相同的图不应该出现在 train 和 val", 答案是 hash。但消费方(YOLO 训练流程)**根本看不到内容相同**——它看到的是文件名相同, 然后 label 路径冲突。**按消费方的视角判重, 是 split_uniqueness 这个 check 的语义本源。**

## 7.2 severity: 硬性 ERROR, 不分级

split_uniqueness 是 D4 里**唯一一个**硬性 ERROR 的业务 check(yaml_schema 算前置 check, 不计入)。

为什么不像 pair_existence 那样按比例分级? — **数据泄露的破坏性跟数量无关**。哪怕只有 1 张图重复, 都可能让测试集评估失真——评估失真又会进一步污染"模型是否值得发布"的决策。这是**质变型错误**, 不是量变。

```python
任何 split 间有重复 → ERROR
没有重复 → PASS
```

简单粗暴, 故意的。

## 7.3 checks/split_uniqueness.py

```python
# apps/platform/src/odp_platform/data_validation/checks/split_uniqueness.py
"""split_uniqueness check — 验证 train / val / test 之间无图像名重复。

判重方式: 按图像 stem (文件名不含扩展名)。理由见讲义阶段 7.1。

Severity: 任何重复 → ERROR。数据泄露的破坏性跟数量无关 — 1 张图重复
就足以让评估指标失真, 是质变型错误, 不按比例分级。
"""
from __future__ import annotations

from itertools import combinations
from typing import Any, Dict, List

from odp_platform.data_validation.registry import (
    check, CheckContext, CheckResult, CheckSeverity,
)


OVERLAP_PREVIEW_LIMIT = 20


@check("split_uniqueness")
def validate_split_uniqueness(ctx: CheckContext) -> CheckResult:
    snap = ctx.snapshot

    if len(snap.images_per_split) < 2:
        # 只有一个或零个 split, 没法判重 — 不算错, 但也没什么可报告的
        return CheckResult(
            name="split_uniqueness",
            severity=CheckSeverity.PASS,
            summary=f"少于 2 个 split, 跳过判重 (当前 splits: {list(snap.splits)})",
            details={"reason": "fewer_than_2_splits"},
        )

    # 每个 split 收集 stem 集合
    stems_by_split: Dict[str, set] = {
        split: {img.stem for img in images}
        for split, images in snap.images_per_split.items()
    }

    # 两两求交集
    overlaps: List[Dict[str, Any]] = []
    for s1, s2 in combinations(stems_by_split.keys(), 2):
        common = stems_by_split[s1] & stems_by_split[s2]
        if common:
            stems_sorted = sorted(common)
            overlaps.append({
                "split_a": s1,
                "split_b": s2,
                "count":   len(common),
                "preview": stems_sorted[:OVERLAP_PREVIEW_LIMIT],
            })

    if not overlaps:
        return CheckResult(
            name="split_uniqueness",
            severity=CheckSeverity.PASS,
            summary=(
                f"{len(snap.splits)} 个 split "
                f"({' / '.join(snap.splits)}) 之间无图像名重复"
            ),
            details={
                "splits": list(snap.splits),
            },
        )

    total_dup = sum(o["count"] for o in overlaps)
    pairs_str = ", ".join(f"{o['split_a']}↔{o['split_b']}({o['count']})" for o in overlaps)
    return CheckResult(
        name="split_uniqueness",
        severity=CheckSeverity.ERROR,
        summary=f"split 间有 {total_dup} 张图像名重复 — 数据泄露! [{pairs_str}]",
        details={
            "reason":      "splits_overlap",
            "splits":      list(snap.splits),
            "total_duplicates": total_dup,
            "overlaps":    overlaps,
        },
    )
```

## 8.1 撞墙③ 的形状: 调度跑完了, 展示完全没做

D4 现在的"展示"分布在两处:

1. **service 里的 `_log_check_result`**: 每个 check 跑完即时打一行 — 信号即时, 但**单行**
2. **service 里的 `_log_summary`**: 跑完打 `1 PASS / 2 WARNING / 1 ERROR` — 计数, 但**没数据集体感**

中间缺一大块: **"这是个什么样的数据集 + 各 check 详细情况"** —— 这部分信息我们**已经收集到了**(都在 snapshot 和 CheckResult.details 里), 但**没人把它组织成可读的输出**。

**症状:**

- 健康数据集跑完 4 行 `[PASS]`, 看不到数据集多大、多少类、各 split 分布——**用户不知道自己在验证什么**
- 失败时 details 里有信息(`error_kinds` / `missing_per_split` / `overlaps` 等), 但只有写代码才能访问——**用户得自己 print 才能看到**
- JSON 报告完全没写——下次跑没法对比, 监控系统接不进来

## 8.2 撞墙③ 的本质

**调度层的职责是"跑 check 收集结果", 不是"组织展示"**——这两件事混在一起会出现三个问题:

1. **加新输出格式很贵**: 想加 HTML 报告? 现在的 `print` / `logger.info` 都是命令式的, 改一遍是重写
2. **测试不可能**: 想测"摘要应该包含类别名"? 现在的输出是 `logger.info` 副作用, 没法单元测试
3. **JSON 报告写起来重复劳动**: 同一份数据要写两份代码——一份给 logger, 一份给 JSON

**解药**: **数据 / 展示物理分离**。

```
ValidationReport (纯数据)
   ├── results: List[CheckResult]
   ├── snapshot: DatasetSnapshot
   ├── overall_severity (派生)
   ├── exit_code (派生)
   ├── failed_results (派生)
   └── to_dict() → JSON 序列化
        ↓
        ↓ 喂给 renderer
        ↓
render_to_logger(report, logger)    ← 阶段 8 我们写这个
render_to_html(report)              ← 未来 (D4.x) HTML 报告
render_to_markdown(report)          ← 未来 (D4.x) Markdown 报告
```

**ValidationReport 是 single source of truth**, renderer 是消费方——这跟 CheckResult.summary/details 的 split 同款思路:**人看的 / 机器看的, 分开存**。

⚠️ **设计点 (YAGNI: 不引入 ReportSection 中间抽象)**: 一种更"完备"的设计是引入 `ReportSection(kind="kv"|"table"|...)` 中间表示层, 让所有 renderer 消费同一份 sections。

但 **D4 主线不做这一步**——理由是:

1. **这是 D4 自创的小语言**, 学员看到 `kind="checklist"` 第一反应是"这又是什么?"——多一层翻译成本
2. **当前只有一种 renderer (logger)**——抽象需要至少 2 个具体场景才能验证抽象方向对不对
3. **数据 / 展示的物理边界已经在**(`report.py` 纯数据 / `render.py` 纯展示), 真要做 HTML 时直接加 `render_to_html.py`, 不会因为"没有 ReportSection"而抓瞎

**复杂度排时间表**——这一站只解决"撞墙③ 展示干瘪", 不顺手解决"以后要做多 renderer"。等 HTML 真要做的那一刻(讲义 TODO 留位), 学员吃透了主干再引入 sections 中间层, 心智负担小得多。

## 8.3 report.py 完整代码 (纯数据)

`apps/platform/src/odp_platform/data_validation/report.py`:

```python
# apps/platform/src/odp_platform/data_validation/report.py
"""ValidationReport — 一次验证的完整产出 (纯数据)。

设计原则: 只装数据 + 派生属性 + to_dict。
所有"怎么展示给人看"的逻辑归 render.py — 这是 D4 撞墙③ 的解药。

派生属性 (永不存值):
    - overall_severity:   全部结果里最严重的级别
    - counts_by_severity: 各级别计数
    - exit_code:          0/1/2 — Unix 退出码语义
    - failed_results:     非 PASS 的 results 子集
    - report_path:        run_dir/report.json (run_dir 未设则 None)
"""
from __future__ import annotations

from dataclasses import dataclass
from pathlib import Path
from typing import Any, Dict, List, Optional

from odp_platform.data_validation.registry import CheckResult, CheckSeverity
from odp_platform.data_validation.snapshot import DatasetSnapshot


@dataclass
class ValidationReport:
    """一次验证的完整产出。

    Args:
        run_id:            形如 '20260516_184523' 的时间戳 ID
        yaml_path:         验证的 yaml 文件路径
        snapshot:          一次扫描产物 (含 stats_per_split / class_names 等)
        results:           各 check 的 CheckResult 列表 (按注册顺序)
        duration_seconds:  整体耗时
        started_at_iso:    起始时间 ISO 8601 字符串 (UTC)
        run_dir:           本次运行的产出目录 (None = 不写盘模式)
    """
    run_id:           str
    yaml_path:        Path
    snapshot:         DatasetSnapshot
    results:          List[CheckResult]
    duration_seconds: float
    started_at_iso:   str
    run_dir:          Optional[Path] = None

    # ---------- 派生属性 (从 results 算, 不存值) ----------

    @property
    def overall_severity(self) -> str:
        """全部结果里最严重的 severity。

        无结果时返回 PASS — 'nothing went wrong' 的合理默认。
        """
        if not self.results:
            return CheckSeverity.PASS
        return max(
            self.results,
            key=lambda r: CheckSeverity.rank(r.severity),
        ).severity

    @property
    def counts_by_severity(self) -> Dict[str, int]:
        """{ERROR: 0, WARNING: 1, INFO: 2, PASS: 1} 之类。"""
        counts: Dict[str, int] = {}
        for r in self.results:
            counts[r.severity] = counts.get(r.severity, 0) + 1
        return counts

    @property
    def exit_code(self) -> int:
        """Unix 退出码:
            0 — PASS / 只有 INFO
            1 — 有 WARNING
            2 — 有 ERROR
        """
        sev = self.overall_severity
        if sev == CheckSeverity.ERROR:
            return 2
        if sev == CheckSeverity.WARNING:
            return 1
        return 0

    @property
    def failed_results(self) -> List[CheckResult]:
        """非 PASS / 非 INFO 的结果 (即 WARNING + ERROR)。"""
        return [r for r in self.results if not r.passed]

    @property
    def report_path(self) -> Optional[Path]:
        """JSON 报告路径 — run_dir 未设时返回 None。"""
        return (self.run_dir / "report.json") if self.run_dir else None

    # ---------- JSON 序列化 ----------

    def to_dict(self) -> Dict[str, Any]:
        """供 JSON 报告序列化。结构稳定 — 监控 / 趋势分析可以读这份。"""
        return {
            "run_id":           self.run_id,
            "yaml_path":        str(self.yaml_path),
            "task_type":        self.snapshot.task_type,
            "started_at":       self.started_at_iso,
            "duration_seconds": round(self.duration_seconds, 3),
            "overall_severity": self.overall_severity,
            "exit_code":        self.exit_code,
            "counts":           self.counts_by_severity,
            "dataset_summary": {
                "nc":          self.snapshot.nc,
                "class_names": list(self.snapshot.class_names),
                "stats_per_split": {
                    split: {
                        "image_count":     stat.image_count,
                        "annotated_count": stat.annotated_count,
                        "total_instances": stat.total_instances,
                    }
                    for split, stat in self.snapshot.stats_per_split.items()
                },
            },
            "results": [
                {
                    "name":     r.name,
                    "severity": r.severity,
                    "summary":  r.summary,
                    "details":  r.details,
                }
                for r in self.results
            ],
        }
```

report.py 落在 ~120 行——纯数据 + 派生属性, 没有任何展示逻辑。

⚠️ **设计点 (派生属性永不存值)**: 注意 `overall_severity` / `counts_by_severity` / `exit_code` / `failed_results` 全是 `@property`, 不是 dataclass 字段。

```python
# ✓ 好品味 (派生从 results 算)
@property
def overall_severity(self) -> str:
    return max(self.results, key=lambda r: CheckSeverity.rank(r.severity)).severity

# ✗ 反模式 (跟 results 冗余)
@dataclass
class ValidationReport:
    results:          List[CheckResult]
    overall_severity: str    # ← 必然跟 results 不一致
```

如果两个都存, 一次 results.append 后 overall_severity 没更新, 数据立刻自相矛盾。**派生属性永远派生**——这是 `CheckResult.passed` 立的同款规矩, 在这里再次应用。

## 8.4 render.py 完整代码 (纯展示)

`apps/platform/src/odp_platform/data_validation/render.py`:

```python
# apps/platform/src/odp_platform/data_validation/render.py
"""render.py — D4 报告的展示层 (纯展示, 不动数据)。

唯一公开 API: render_to_logger(report, logger, report_path=None)

输出结构 (三段式):
    1. 报告头 (run_id / yaml_path / task / 耗时)
    2. 数据集摘要 (类别 / 各 split 的 3 个数字)
    3. 检查项一览 (每个 check 一行)
    4. 失败详情 (仅当有非 PASS check 时出现, 按 check 类型展开 details)
    5. 报告尾 (JSON 报告路径)

未来扩展:
    - render_to_html(report) → str         给非工程师看的报告 (D4.x)
    - render_to_markdown(report) → str     给 CI / Slack 摘要 (D4.x)
    所有 renderer 消费同一份 ValidationReport, 互不影响。
"""
from __future__ import annotations

import logging
from pathlib import Path
from typing import Optional

from odp_platform.data_validation.registry import CheckResult
from odp_platform.data_validation.report import ValidationReport
from odp_platform.data_validation.snapshot import DatasetSnapshot


H1_LINE = "=" * 72
H2_LINE = "-" * 72


# ============================================================
# 公开 API
# ============================================================

def render_to_logger(
    report:      ValidationReport,
    logger:      logging.Logger,
    report_path: Optional[Path] = None,
) -> None:
    """把 ValidationReport 渲染成三段式日志输出。

    无副作用之外的事 — 不写盘, 不 raise, 只调用 logger.info 把内容打出来。
    """
    _render_header(report, logger)
    _render_dataset_summary(report.snapshot, logger)
    _render_check_overview(report.results, logger)

    if report.failed_results:
        _render_failure_details(report.failed_results, logger)

    _render_footer(report_path, logger)


# ============================================================
# 段 1: 报告头
# ============================================================

def _render_header(report: ValidationReport, logger: logging.Logger) -> None:
    logger.info(H1_LINE)
    logger.info("                       YOLO 数据集验证报告")
    logger.info(H1_LINE)
    logger.info(f"  run_id   {report.run_id}")
    logger.info(f"  yaml     {report.yaml_path}")
    logger.info(
        f"  task     {report.snapshot.task_type:8s}  "
        f"耗时  {report.duration_seconds:.2f}s  "
        f"severity  {report.overall_severity}"
    )


# ============================================================
# 段 2: 数据集摘要
# ============================================================

def _render_dataset_summary(snapshot: DatasetSnapshot, logger: logging.Logger) -> None:
    logger.info(H2_LINE)
    logger.info("  ▸ 数据集摘要")

    if snapshot.class_names:
        names_str = ", ".join(snapshot.class_names)
        logger.info(f"    类别:  {names_str}  (nc={snapshot.nc})")
    else:
        logger.info("    类别:  (未取到 — yaml_schema 应已报错)")

    if not snapshot.stats_per_split:
        logger.info("    (无任何 split 可统计)")
        return

    for split, stat in snapshot.stats_per_split.items():
        logger.info(
            f"    {split:6s}:  "
            f"{stat.image_count:>6,} 张  /  "
            f"{stat.annotated_count:>6,} 标注  /  "
            f"{stat.total_instances:>6,} 实例"
        )


# ============================================================
# 段 3: 检查项一览
# ============================================================

def _render_check_overview(results: list, logger: logging.Logger) -> None:
    logger.info(H2_LINE)
    logger.info("  ▸ 检查项一览")
    for r in results:
        logger.info(f"    [{r.severity:7s}]  {r.name:18s}  {r.summary}")


# ============================================================
# 段 4: 失败详情 (well-known keys 模式)
# ============================================================

def _render_failure_details(failed: list, logger: logging.Logger) -> None:
    logger.info(H2_LINE)
    logger.info("  ▸ 失败详情")
    for r in failed:
        _render_one_check_details(r, logger)


def _render_one_check_details(r: CheckResult, logger: logging.Logger) -> None:
    """按 check name 选择性展开 details 里的 well-known 字段。

    展示层局部 if/elif —— 这里的增长很慢 (只在新 check 想自定义展开时
    才动这里), 跟"调度层的 if/elif"性质完全不同。 见讲义阶段 8.5 设计点。
    """
    logger.info("")
    logger.info(f"    >> {r.name}  [{r.severity}]")
    det = r.details

    if r.name == "yaml_schema" and "problems" in det:
        for p in det["problems"]:
            logger.info(f"        - {p}")

    elif r.name == "pair_existence":
        mps = det.get("missing_per_split", {})
        if mps:
            parts = ", ".join(f"{s}={n}" for s, n in mps.items())
            logger.info(f"        各 split 缺失:  {parts}")
        ex = det.get("missing_examples", {})
        for split, paths in ex.items():
            logger.info(f"        示例 ({split}, 前 {min(5, len(paths))} 条):")
            for p in paths[:5]:
                logger.info(f"          {p}")

    elif r.name == "label_format":
        kinds = det.get("error_kinds", {})
        if kinds:
            parts = ", ".join(f"{k}={v}" for k, v in kinds.items())
            logger.info(f"        错误类型:  {parts}")
        for e in det.get("errors_preview", [])[:5]:
            logger.info(
                f"        - {Path(e['label']).name}:{e['line_no']}  "
                f"{e['kind']}  {e['detail']}"
            )

    elif r.name == "split_uniqueness" and det.get("overlaps"):
        for o in det["overlaps"]:
            logger.info(
                f"        {o['split_a']} ↔ {o['split_b']}:  "
                f"{o['count']} 张重复"
            )
            for stem in o["preview"][:5]:
                logger.info(f"          {stem}")

    # 通用 fallback: 任何未匹配 well-known 分支的 check, 至少展示 reason
    else:
        if "reason" in det:
            logger.info(f"        reason: {det['reason']}")


# ============================================================
# 段 5: 报告尾
# ============================================================

def _render_footer(report_path: Optional[Path], logger: logging.Logger) -> None:
    logger.info(H2_LINE)
    if report_path is not None:
        logger.info(f"  详细报告:  {report_path}")
    logger.info(H1_LINE)
```

render.py 落在 ~170 行——每个 `_render_*` 子函数都在 30 行内, 主入口 `render_to_logger` 把 5 个段串起来一目了然。

## 8.5 设计点细讲

⚠️ **设计点 (well-known keys 模式: if/elif 在展示层是药)**:

`_render_one_check_details` 里:

```python
if r.name == "yaml_schema" and "problems" in det:
    for p in det["problems"]:
        logger.info(...)
elif r.name == "pair_existence":
    ...
elif r.name == "label_format":
    ...
```

看起来是个**长 if/elif**(违反我们在阶段 1 撞墙①的教训), 但它**对**——它不是"调度", 是"按 check 类型选择性展开 details 里的 well-known 字段"。

跟"调度"的本质区别:

| | 调度层 if/elif (撞墙①) | 展示层 if/elif (本处) |
|---|---|---|
| **增长触发** | 加任何新 check 都增长 | 只在新 check 想自定义展开时才增长 |
| **不动它的代价** | 新 check 接入不了 (调度散落) | 新 check 接入了, 只是详情走通用 fallback |
| **数量上限** | 等于实现的数量 (无上限) | 等于"想自定义展示的 check 数量"(增长很慢) |

**展示层的 if/elif 不强制增长**——加新 check 时如果不写 elif, 会走 `else` 的通用 fallback(至少把 `reason` 字段展示出来)。这就是它跟调度散落根本性不同的地方。

如果以后展示层复杂度爆炸, 可以把"按 check name 派发到展开函数"做成 `_DETAIL_RENDERERS = {"yaml_schema": _render_yaml_schema_details, ...}` 这样的 dispatch dict——但 4 个 check 用 if/elif 就够了, **过早抽象的成本超过收益**。

> **金句**: **"if/elif 在调度层是病, 在展示层是药——区别在'谁会增长'。调度的 elif 数量等于实现的数量, 必然增长; 展示的 elif 数量等于'想自定义展示的 check 类型'数量, 增长很慢, 而且不动它的代价只是'用通用 fallback'而已。"**

⚠️ **设计点 (PASS 不再单独打日志, 集中到摘要里)**:

回想阶段 2 的 `_log_check_result` 里, PASS 走 DEBUG 不输出——当时我们说"避免 4 行 [PASS] 噪声盖过信号"。

阶段 8 的 `_render_check_overview` 把所有 check 都列出来——**PASS 也展示**, 因为这里是**集中报告**, 不是即时事件流。

这就是 D4 全程坚持的两个不同语义:

- **即时日志**(service `_log_check_result`): 只记"变化"事件 (ERROR / WARNING / INFO), PASS 不进
- **最终报告**(render `_render_check_overview`): 全列, PASS 也算"我跑过了"的存在性证明

两种语义在不同的输出层各自合理——日志看进度, 报告看全貌。

## 9.1 validate_dataset() — 端到端函数

`apps/platform/src/odp_platform/data_validation/service.py` 末尾追加(跟 `run_all_checks` 放一起, 都是调度层职责):

```python
# apps/platform/src/odp_platform/data_validation/service.py
# ... 前面是阶段 2 写的 run_all_checks / _safe_run_one / _log_check_result / _log_summary ...

import json
import time
from datetime import datetime, timezone

from odp_platform.common.paths import validation_run_dir
from odp_platform.common.system_utils import log_device_info
from odp_platform.data_validation.report import ValidationReport
from odp_platform.data_validation.snapshot import build_snapshot


def validate_dataset(
    yaml_path:    Path,
    task_type:    Optional[str] = None,
    run_id:       Optional[str] = None,
    run_dir:      Optional[Path] = None,
    write_report: bool = True,
) -> ValidationReport:
    """端到端验证: 构造 snapshot → 跑 check → 包装 report → 可选写盘。

    Args:
        yaml_path:    数据集 yaml 文件路径
        task_type:    'detect' / 'segment' / None (None → 读 yaml.task, 再不行 detect)
        run_id:       手动指定运行 ID; None 表示自动用时间戳
        run_dir:      手动指定运行目录; None 表示用 validation_run_dir(run_id)
        write_report: 是否写 JSON 报告到 run_dir/report.json

    Returns:
        ValidationReport (run_dir 字段已填, 调用方可以拿 .report_path 取 JSON 位置)
    """
    # ---- 解析 run_id / run_dir ----
    resolved_run_id  = run_id  or datetime.now().strftime("%Y%m%d_%H%M%S")
    resolved_run_dir = run_dir or (validation_run_dir(resolved_run_id) if write_report else None)

    if write_report and resolved_run_dir is not None:
        resolved_run_dir.mkdir(parents=True, exist_ok=True)

    # ---- 跑核心流程 ----
    t0 = time.perf_counter()
    started_iso = datetime.now(timezone.utc).isoformat()

    log_device_info(logger)   # 端到端入口打一次设备信息, 让"慢"可归因
    snapshot = build_snapshot(yaml_path=yaml_path, task_type=task_type)
    ctx      = CheckContext(yaml_path=yaml_path, snapshot=snapshot)
    results  = run_all_checks(ctx)

    duration = time.perf_counter() - t0

    # ---- 包装 ValidationReport ----
    report = ValidationReport(
        run_id=resolved_run_id,
        yaml_path=yaml_path,
        snapshot=snapshot,
        results=results,
        duration_seconds=duration,
        started_at_iso=started_iso,
        run_dir=resolved_run_dir,
    )

    # ---- 写 JSON 报告 ----
    if write_report and resolved_run_dir is not None:
        report_path = resolved_run_dir / "report.json"
        report_path.write_text(
            json.dumps(report.to_dict(), indent=2, ensure_ascii=False),
            encoding="utf-8",
        )

    return report
```

⚠️ **设计点 (函数 vs 类)**: 为什么是 `validate_dataset()` 函数而不是 `DatasetValidator` 类?

判定标准: **类的存在感, 应该来自"持有状态"或"多个相关方法"**。这里:

- 函数内部的"状态"(run_id / run_dir / snapshot / results / duration)全部进了**返回值** `ValidationReport`——调用方拿了 report 之后, 这个函数内部就没东西可保留了
- 没有第二个相关公共方法——只有一个"端到端跑一次"

把它做成类要么强行让 `report` 字段挂在 `self.report` 上(然后 `validator.run()` 之后 `validator.report` 取——为什么不直接 `report = validator.run()`?), 要么写一个空空的 `Validator(yaml_path).run()` 包装——**两步都是 OOP 仪式感, 没增加任何能力**。

> **金句**: **"只有一个公共方法的类应该是一个函数。给类找'状态'当借口, 不如把状态明确还给返回值。"**

⚠️ **设计点 (`@time_it` 装在哪一层 / `log_device_info` 打几次)**:

- `@time_it` 装在**框架函数**(`build_snapshot` / `run_all_checks`), **不**装在各个 check。理由: (a) 单个 check 通常毫秒级, 量化无意义; (b) check 是**实现层**, 装饰器会污染插件; (c) 框架层量化能直接回答"为什么慢"——要么慢在扫盘 (snapshot), 要么慢在聚合 (run_all_checks), 二选一。
- `validate_dataset` 自己**没装** `@time_it`, 因为它已经手动用 `time.perf_counter()` 测了 `duration_seconds` 进 `ValidationReport`——重复加装饰器会让"端到端总耗时"在日志里出现两次。
- `log_device_info(logger)` 在 `validate_dataset` 入口**只调一次**, 不要往 `build_snapshot` 里塞——设备信息是**会话级元数据**, 不是流程级事件。这里直接拿模块的 `__name__` logger 即可 (handler 都是 CLI 入口挂的, D2 `get_logger` 幂等, 业务模块走冒泡)。

> **金句**: **"`@time_it` 装框架不装实现——单个 check 几毫秒, 量化它纯属噪音; 框架层量化才能告诉你'慢在哪一层'。"**

更新公开 API:

```python
# __init__.py

__all__ = [
    ...,
    "validate_dataset",     # ← 阶段 9 新增
]
```

## 9.2 CLI: odp-validate

`apps/platform/src/odp_platform/cli/validate_data.py`:

```python
# apps/platform/src/odp_platform/cli/validate_data.py
"""odp-validate CLI — D4 子系统的端到端入口。

用法:
    odp-validate --dataset NAME [--task detect|segment]
    odp-validate --yaml /path/to/yaml [--task detect|segment]

退出码:
    0  PASS or only INFO
    1  WARNING present
    2  ERROR present
    3  Ctrl-C or unexpected exception (CLI bug, not data issue)
"""
from __future__ import annotations

import argparse
import logging
import sys
from pathlib import Path

from odp_platform.common.logging_utils import get_logger
from odp_platform.common.paths import LOGGING_DIR, dataset_yaml_path
from odp_platform.data_validation import (
    render_to_logger,
    validate_dataset,
)


def _build_parser() -> argparse.ArgumentParser:
    parser = argparse.ArgumentParser(
        prog="odp-validate",
        description="YOLO 数据集质量验证 (data_validation 子系统的 CLI 入口)",
    )

    # 数据集来源 — 互斥, 必须有且只能有一个
    source = parser.add_mutually_exclusive_group(required=True)
    source.add_argument(
        "--dataset",
        help="数据集名 (= configs/datasets/<name>.yaml 的 stem; 日常使用)",
    )
    source.add_argument(
        "--yaml",
        type=Path,
        help="直接指定 yaml 完整路径 (调试 / 临时数据集用)",
    )

    parser.add_argument(
        "--task",
        choices=("detect", "segment"),
        default=None,
        help="任务类型 (None = 读 yaml.task 字段, 仍读不到则 detect)",
    )
    parser.add_argument(
        "--no-report",
        action="store_true",
        help="不写 JSON 报告 (只跑验证, 看日志即可)",
    )
    parser.add_argument(
        "-v", "--verbose",
        action="store_true",
        help="DEBUG 级日志输出 (控制台和文件同步开 DEBUG)",
    )

    return parser


def main() -> int:
    args = _build_parser().parse_args()

    log_level = logging.DEBUG if args.verbose else logging.INFO
    logger = get_logger(
        base_path=LOGGING_DIR,
        log_type="validate",
        log_level=log_level,
    )

    try:
        # ---- 解析 yaml_path ----
        if args.dataset:
            yaml_path = dataset_yaml_path(args.dataset)
        else:
            yaml_path = args.yaml.resolve()

        # ---- 跑端到端 ----
        report = validate_dataset(
            yaml_path=yaml_path,
            task_type=args.task,
            write_report=not args.no_report,
        )

        # ---- 渲染日志 ----
        render_to_logger(report, logger, report_path=report.report_path)

        return report.exit_code

    except KeyboardInterrupt:
        logger.warning("用户中断 (Ctrl-C)")
        return 3
    except Exception:
        logger.exception("未预期异常 — CLI bug, 不是数据问题")
        return 3


if __name__ == "__main__":
    sys.exit(main())
```

⚠️ **设计点 (CLI 故意写薄)**: 这个文件 ~100 行, 但**绝大部分是 argparse 参数声明 + try/except 兜底**——argparse 自我文档化承担了 `--help` 输出和参数校验。真正的"业务"只有 `main()` 中间那 **8 行**:解析 yaml_path → 调 validate_dataset → render_to_logger → return exit_code。

CLI 这层皮**永远不该写业务**——任何"if pair_existence 怎么样, 就特殊处理" 进 CLI 就是错的, 该回到 check 或 service 里。**CLI = argparse + logger + return**, 三件事。

⚠️ **设计点 (`--dataset` vs `--yaml` 互斥)**: `add_mutually_exclusive_group(required=True)` 保证两者必须有且只能有一个。

为什么要支持两种:
- `--dataset rsod` 是**日常使用**(知道数据集名就好, 路径走 paths.py 解析)
- `--yaml /tmp/some.yaml` 是**调试 / 临时数据集** (不在 configs/datasets/ 下的 yaml)

两种语义是**用户感知层面的真实区分**, 不是为了灵活性堆 flag。

⚠️ **设计点 (logger.exception 捕获未预期异常)**:

主流程外面套 `try/except KeyboardInterrupt + except Exception`——CLI 入口该做的最后兜底:

- **KeyboardInterrupt**: 用户主动中断, 不算 bug, 友好结束, 退出码 3
- **Exception**: 任何没被前面流程处理的异常, 走 `logger.exception` 把 traceback 完整记录(包括日志文件), 然后退出码 3

裸 traceback 吐到 stderr 看着不专业 + 不方便事后排查 + 日志文件里没东西。**让所有报错统一进 logger**, 这是 D2 立 logging_utils 的本意之一。

⚠️ **设计点 (复用工具的纪律 = SSoT 包含 API)**:

D4 CLI 这一行 `get_logger(base_path=..., log_type=..., log_level=...)` 看着平淡, 但它在守一条容易破的纪律——**复用别人立的工具, 要么按它的签名用, 要么回去改它的签名**。

反例 (讲义早期版本就栽过的坑): 想要"控制台开 DEBUG, 文件保持 INFO"的更细粒度, 于是在 CLI 里写 `console_level=logging.DEBUG`——D2 的 `get_logger` **根本没有这个参数**, 直接 `TypeError`。

正确做法两条路:

1. **接受现状**: D2 `get_logger` 让 console 和 file 共用一个 `log_level`, 这是 D2 的设计选择, D4 就按这个粒度用。`--verbose` 的真实语义就是"控制台和文件双双 DEBUG"——文档老实写, 不要装。
2. **升级工具**: 真有充分理由要拆 console / file 两个 level (比如 DEBUG 日志暴量、文件存档不该被淹), 回到 **`apps/platform/src/odp_platform/common/logging_utils.py`** 改 `get_logger` 的签名让两个 handler 接收独立 level, **并把所有调用方一起升级** (D2 自身、D3 transform_data、D4 validate_data)。

什么都不该做的, 是**在调用方编一个不存在的参数**——这不仅会立刻报错, 还把"D4 假装 D2 提供了某个能力"这种错觉固化进文档。

> **金句**: **"复用工具的纪律: 要么按它的签名用, 要么回去改它的签名——别在调用方编一个不存在的参数。SSoT 不只在数据上有效, 在 API 上也有效。"**

## 10.2 ADR-004: 决策记录

把 D4 的关键决策固化成一份**架构决策记录** (Architecture Decision Record)。这种文档不是给"现在的人"看的, 是给**一年后想 refactor D4 的人**看的——回答"当初为什么这么做"。

`apps/platform/docs/adr/ADR-004-data-validation.md`:

```markdown
# ADR-004: data_validation 子系统架构

**状态**: Accepted
**日期**: 2026-05-16
**作用范围**: `apps/platform/src/odp_platform/data_validation/`

## 上下文

D3 产出的 YOLO yaml 直接进 D5 训练前, 需要质检——验证图像/标签成对、
yaml 字段一致、各 split 无图像名重复、label 格式合法等。

设计这套质检子系统时, 我们面对几个核心选择:

- 调度方式: 一坨函数 / if-elif / 注册表?
- check 间数据共享: 各自扫盘 / 共享快照?
- 失败语义: 早返回 / 收集所有?
- 修复能力: 检测 + 修复 / 仅检测?
- 数据与展示: 耦合 / 分离?

本 ADR 记录最终选择 + 理由。

## 决策

### 1. 调度: 注册表聚合执行

- 每个 check 是 `checks/` 下的独立文件 + `@check("name")` 装饰器自动注册
- 调度层 `service.run_all_checks(ctx)` 跑全部 check, 收集 `List[CheckResult]`
- **任何 check 抛异常都被 service 接住, 包装成 ERROR 级 CheckResult, 不阻断其他 check**

跟 D3 的 data_pipeline.registry 同源, 但调度模式相反:
- D3: 互斥分发 (一次调用选 1 个 converter)
- D4: 聚合执行 (一次调用跑全部 check)

聚合模式的灵魂在 `_safe_run_one` 的 `try/except Exception` — 整个子系统
唯一一处宽泛异常捕获, 因为这是开闭原则下的扩展点, 调度层无法预知 check
会抛什么。

### 2. CheckResult 4 字段 + Severity 4 级

字段: `name / severity / summary / details`, 派生 `passed: bool`。

severity 四级 (PASS/INFO/WARNING/ERROR) 而不是两级 (passed: bool):
- 业务上"失败"分多种紧张度 — pair_existence 缺 0.1% 跟缺 60% 不该同级
- 跟 syslog / k8s events 的成熟习惯对齐, 接监控系统时不需要再翻译

### 3. DatasetSnapshot: 一次扫描多次复用

`build_snapshot(yaml_path, task_type)` 一次扫盘, 装进 frozen dataclass,
通过 `CheckContext.snapshot` 喂给所有 check。

三条铁律:
- **装事实, 不做判断**: snapshot 装数据, check 做判定
- **frozen=True + Tuple 容器**: 不可变, 防 check 串改
- **best-effort, 不抛异常**: yaml 解析失败装进 `yaml_load_error` 字段,
  由对应 check 报告 — 调度层永远不该自己 fail

附带"顺手统计" 3 个数字 per split (image_count / annotated_count /
total_instances) — I/O 密集型流程的免费收益, 但保持 snapshot 装事实的
边界, 不算"业务判断"。

### 4. 自动 import: 加新 check 不改框架代码

`_ensure_initialized` 用 `pkgutil.iter_modules` 自动扫 `checks/` 子包
下所有非下划线模块, 在 import time 触发 `@check` 装饰器副作用。

加新 check = 加新文件, **registry.py 一行不动**。同 pattern 同 Flask 路由
/ pytest 测试函数的自动发现。

### 5. ValidationReport(纯数据) + render.py(纯展示)分离

- `report.py`: ValidationReport dataclass + 派生属性 (overall_severity /
  exit_code / failed_results) + to_dict()
- `render.py`: `render_to_logger(report, logger)` 三段式输出
  (头 / 数据集摘要 / check 一览 / 失败详情 / 尾)

未来加 HTML / Markdown renderer = 新增 `render_to_xxx.py`, 数据层
一行不动。

**已知边界**: 未引入 `ReportSection` 中间抽象层。理由 — YAGNI: 当前只有
一种 renderer, 抽象层需要至少 2 个具体场景才能验证方向。 加 HTML 报告时
再做 refactor 引入。

### 6. 质检与修复在 SRP 层面分开

D4 **只检测, 不修复**。`CheckResult` 无 `fixable` 字段, CLI 无
`--fix` / `--apply` 选项。

理由:
- "质检工具的诚实在于它什么都不动 — 数据是不是被改了, 用户绝不该靠
  看日志推断。"
- 修复属于 data_pipeline 的下一次产线运行 (改完原始标注重跑 odp-transform),
  或独立的 odp-clean 工具
- 把"破坏性操作"作为质检副产品是常见的反模式 (eslint --fix 类工具不在
  CI 里默认开)

### 7. 不引入 DatasetValidator 类

`validate_dataset()` 是函数, 不是类。 内部状态 (run_id / duration /
snapshot / results) 全部进返回值 `ValidationReport`, 调用方一行拿:

```python
report = validate_dataset(yaml_path, task_type='detect')
```

只有一个公共方法的类是 OOP 仪式感, 没增加任何能力。

## 后果

**正面**:
- 加新 check 零摩擦 (加文件即可)
- 加新 renderer 零摩擦 (加文件即可)
- 单文件全部 <= 220 行, 教学场景可消化
- 退出码语义稳定, CI 集成简单

**负面 / 已知边界**:
- well-known keys 模式在 render.py 里有 if/elif (4 个 check 一个 elif),
  增长虽慢但确实存在。 阈值在 ~8 个 check, 之后再考虑 dispatch dict
- 无 SAMPLE 模式 — 超大数据集 (>50 万图) 验证慢, 留给 D4.x 加并发
- 无配置化阈值 — 不同数据集场景 (医学影像 vs 通用检测) 容忍度不同,
  PAIR_MISSING_ERROR/WARN_RATIO 硬编码, 留给 D4.x 在数据集级 yaml
  支持覆盖

## 拒绝的方案

- **互斥分发模式**: 不适合"全部跑一遍"的语义
- **每个 check 自己扫盘**: 重复 IO + 数据漂移风险, 解药是 snapshot
- **fixable 双层语义** (装饰器 + 实例): SRP 越界, 砍
- **DatasetValidator 类**: OOP 仪式感, 砍
- **ReportSection 中间抽象层**: 教学成本 + YAGNI, 等真做 HTML 再引入
- **砍掉 INFO 这一级**: pair_existence "少量遗漏可容忍"无处表达
```

## D4 学到的"工业级模式"

| 模式 | 用在 | 你的工具箱 |
|---|---|---|
| 注册表 (聚合模式) | data_validation.registry | 任何需要"插件全跑一遍"的系统 |
| 装饰器自动注册 | `@check` | 跟 D3 同款, 复用度极高 |
| 框架 vs 实现物理分离 | `checks/` 子包 | 任何插件化系统的标准结构 |
| pkgutil 自动 import | `_ensure_initialized` | Flask / pytest 同款, 通用模式 |
| 严重程度四级常量 | CheckSeverity | 任何"质量分级"需求 (跟 syslog / k8s 对齐) |
| 派生属性永不存值 | `passed` / `overall_severity` / `exit_code` | 数据类设计的 DRY |
| 装饰器不包装函数 | `@check` 直接 return func | 测试友好 + 堆栈干净 |
| 调度层异常兜底 | `_safe_run_one` 里的 try/except | 聚合模式的灵魂 |
| 共享数据载体 (frozen + Tuple) | DatasetSnapshot | 任何"一次扫描多次消费" |
| 顺手统计 (在主循环里) | `_build_split_stats` | I/O 密集流程的免费收益 |
| best-effort 工厂 | build_snapshot 不抛异常 | 让上游错误在下游报错处理 |
| 按比例分级 severity | pair_existence | 量化失败的工程化表达 |
| well-known keys 模式 | `_render_one_check_details` | 展示层局部 case-by-case |
| 数据 / 展示物理分离 | report.py + render.py | 准备多 renderer 但不预制 |
| CheckContext 扩展位 | (yaml_path + snapshot, 留位) | 永不改 check 函数签名 |
| Unix 退出码 0/1/2 | exit_code 派生属性 | 任何 CLI 的标配 |
| 函数 > 类 (只有一个公共方法时) | validate_dataset() | 反 OOP 仪式感 |
| logger.exception 兜底 | CLI main() | 最后一道"裸 traceback 不进 stderr" |
| 灵魂测试 | test_aggregator_catches_check_exceptions | 测设计哲学, 不是测细节 |
| 真文件 tmp_path (非 mock) | test_e2e | 复杂数据流的测试不该绕开复杂度 |
| `@time_it` 装框架层不装实现层 | build_snapshot / run_all_checks | 任何 I/O 密集 + 插件化系统 |
| 设备信息打在端到端入口 | validate_dataset 起手 log_device_info | "慢"的归因—— 机器 vs 数据 |

**22 个工业级模式**——D4 一天的收获放到任何 Python 项目都能直接复用。注册表 + 共享数据载体 + 数据展示分离 + 自动 import, 这四个组合是数据工具的"基本骨架"。

## "工程素养金句"汇总

D4 散落在各章节的金句, 集中起来一遍:

1. **"目录的'不存在'本身也是信息——别用预先 mkdir 抹掉它。"** — 阶段 0.2
2. **"SSoT 的纪律不是写出来的, 是【克制】出来的——每次想新立常量前先问'是不是已经有了'。"** — 阶段 0.3
3. **"一个子系统的工程素养, 在它的第一次 commit 就能看出来——是直接开干, 还是先把规矩立好。"** — 阶段 0.4
4. **"聚合模式的本质不是'调用所有'——是'承诺没人能阻断所有'。后者比前者贵, 但缺了它聚合就不成立。"** — 阶段 1.6
5. **"代码不止靠'写得对'传达意图——'有没有进 git'本身也是意图的一部分。"** — 阶段 1.8
6. **"框架的难度不在'写代码', 在'选好字段'——字段定下来, 代码就只剩抄写; 字段定错了, 代码越长越歪。"** — 阶段 2.2
7. **"日志只该记'变化'。健康事件不进日志——这是 systemd / k8s 都遵循的规则。"** — 阶段 2.4
8. **"一次性把'我的数据是怎么来的'信息固化进数据对象本身——比任何文档 / 注释 / 命名约定都强。"** — 阶段 4.3
9. **"'顺手做'的红利在 I/O 密集型流程里几乎免费——但每次顺手都要问'这个属于装数据还是做判断'。"** — 阶段 4.4
10. **"判重的本质是'按消费方的视角'——不是按'最严格'。"** — 阶段 7.1
11. **"if/elif 在调度层是病, 在展示层是药——区别在'谁会增长'。"** — 阶段 8.5
12. **"只有一个公共方法的类应该是一个函数。给类找'状态'当借口, 不如把状态明确还给返回值。"** — 阶段 9.1
13. **"灵魂测试 = 测设计哲学, 不是测代码细节——它在告诉未来的 refactor 者: 改什么都行, 别改这条。"** — 阶段 10.1
14. **"真文件 tmp_path > mock —— 复杂数据流的测试不该绕开复杂度。"** — 阶段 10.1
15. **"产线和质检, 是数据工程的左右脚——D3 教你迈出去, D4 教你站得稳。"** — D4 开篇
16. **"`@time_it` 装框架不装实现——单个 check 几毫秒, 量化它纯属噪音; 框架层量化才能告诉你'慢在哪一层'。"** — 阶段 9.1
17. **"复用工具的纪律: 要么按它的签名用, 要么回去改它的签名——别在调用方编一个不存在的参数。SSoT 不只在数据上有效, 在 API 上也有效。"** — 阶段 9.2

**17 句金句**——贴在工位上, 每天回头看一眼, 比看 10 本设计模式书都顶用。

## 练习 1: 跟着流程跑一遍 (必做)

不要看现成代码——按本讲义 10 个阶段的顺序, **从空仓库开始**写一遍。

合格标准:

- 12 次 commit, message 都对得上里程碑
- `odp-validate --dataset <你的数据集> --task detect` 能跑通
- 故意删几个标签后, exit code 应该是 1 (WARNING) 或 2 (ERROR), 取决于缺多少
- 故意把 train 一张图复制到 val, `split_uniqueness` 报 ERROR + exit code 2

**最少 5 小时**, 大概率 6-8 小时。 这是 D4 唯一**绕不开**的练习——没自己写过的人对"撞墙②为什么必然出现"理解不深。

## 练习 2: 加一种新 check (必做, 检验开闭原则)

加一个 **image_integrity check**: 验证每张图能被 PIL 正常解码、尺寸 > 32×32、不超过 8192×8192。

要求:

- 一个新文件 `checks/image_integrity.py`, 用 `@check("image_integrity")` 注册
- 复用 `snapshot.images_per_split`, 不自己 glob
- 按损坏比例分级 severity: PASS / INFO / WARNING / ERROR (跟 pair_existence 同款分级)
- 写到 `render.py` 的 `_render_one_check_details` 里加一个 elif 分支(可选, 不写也能跑, 走通用 fallback)

**关键验证**: 加完这个 check 后, **框架代码一行不动**——
- `registry.py` 一行不动 (自动 import 抓到新文件)
- `service.py` 一行不动
- `report.py` 一行不动
- 其他 check 一行不动

这就是开闭原则的具体体验, **比读 10 篇 SOLID 教程都顶用**。

## 练习 3: 实现 D4.1 配置化阈值 (中等难度)

当前 `PAIR_MISSING_ERROR_RATIO = 0.5` 写死在 constants.py——但不同数据集场景容忍度不同。

实现一个**数据集级**阈值覆盖机制:

- 数据集 yaml 里加 `odp_validation:` 块, 用户可以写 `pair_missing_error_ratio: 0.3` 覆盖全局
- snapshot 从 yaml 读这些值, 装到一个新的 `config_overrides` 字段
- 各 check 优先用 snapshot 里的覆盖值, 没覆盖才用全局

**思考题**: CheckContext 在 D4 设计时就预留了"扩展位"——这次实现实际改了几个 check 函数的签名? 如果改的话, 是不是说明扩展位设计失败?

(答案: 不应改任何 check 函数签名。 覆盖值要么走 `ctx.snapshot.config_overrides`, 要么扩 CheckContext 加 `config_overrides` 字段。这是检验阶段 2 那个"扩展位"设计是否真扎实的好题目。)

## 练习 4: 加一种新 renderer (中等难度, 检验数据/展示分离)

实现 `render_to_markdown(report) -> str`, 输出 GitHub-flavored Markdown:

- 头部用 `#` 一级标题
- 数据集摘要用 markdown 表格
- 检查项一览用 checkbox 列表 (`- [x]` 表示 PASS, `- [ ]` 表示失败)
- 失败详情每个 check 用 `<details>` 折叠块

要求:

- 新文件 `render_markdown.py`, 公开 `render_to_markdown` 函数
- 消费同一份 `ValidationReport`
- **`report.py` 一行不动**——所有需要的信息都在 ValidationReport 里
- CLI 加一个 `--output {logger,markdown}` 选项, markdown 模式写到 `run_dir/report.md`

**关键体验**: 写完这个 renderer 之后, **回看是不是真的没改 report.py**——这是数据/展示分离设计是否成功的试金石。

## 练习 5: 灵魂测试改造 (高难度, 思考题)

阶段 10 立的 `test_aggregator_catches_check_exceptions` 是一条灵魂测试。但它有一个隐藏假设——**测的是 service.run_all_checks 兜异常**。

如果未来某天有人把 `_safe_run_one` 里的 `try/except Exception` 改成 `try/except (KeyError, ValueError)` — 假装"具体化"地兜底, 但实际上漏了别的异常——**当前这条测试还能抓到吗?**

写一段测试代码, 让它对"`except Exception` → `except (KeyError, ValueError)`"这种 silently-wrong 的退化也能报警。

(提示: 用一个抛 `RuntimeError` 而不是 `KeyError` 的 fake check。)

**写 200 字的设计权衡**: 灵魂测试的边界在哪? 测得太具体 → 实际是测代码细节; 测得太抽象 → 容易漏掉退化。 这是一门艺术, 不是科学。

# 给你的最后一段话

D4 的难, 不在写 4 个 check——那部分跟着模板, 一两个小时就能搬完。 D4 的难在**理解"为什么这 4 个 check 不能写成一个 validate() 函数"** —— 也就是**注册表模式的灵魂**:

- 不是"分发", 是"聚合"
- 不是"任意接口", 是"统一返回"
- 不是"框架装实现", 是"框架承诺没人能阻断他人"

学完 D3 + D4, 你已经用同一种 idiom 解了两道不同的题——**互斥分发**(D3) 和**聚合执行**(D4)。这就是工业级模式的复用价值。

下一站 D5 训练子系统, 又是新的题, 但你的工具箱里已经有了:

- 注册表 (插件化的基本骨架)
- 共享数据载体 frozen dataclass (避免 IO 浪费)
- 数据 / 展示分离 (准备多 renderer 但不预制)
- pkgutil 自动 import (开闭原则的真正物理基础)
- 严重程度四级 (跨工具 / 跨监控的语义同步)
- well-known keys 模式 (展示层的局部 case-by-case)
- CheckContext 扩展位 (永不改 check 函数签名)

D5 会发现:**训练子系统的 80% 设计, 都是从这箱工具里找一个**。

工程师和程序员的差别——**程序员让代码跑起来, 工程师让代码【可以解释自己为什么是这样】**。

D4 教完, **不只是教你写了一个 validator, 而是教你写一个能在 ADR 里讲清楚每个决策为什么这么做的 validator**。 这两者的差别, 三年后会变成你工资条上的数字。
