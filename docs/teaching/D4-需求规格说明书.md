# ODPlatform D4: `data_validation` 子系统 — 需求规格说明书

> **文档性质**: 开发任务说明书 (PRD + Technical Spec)
> **目标读者**: 学生开发者
> **预计工时**: 6–8 课时
> **前置条件**: 完成 D3 (`data_pipeline` 子系统 + `odp-transform` CLI)

---

## 1. 项目背景

D3 已经把"原始数据集 + 一行命令"变成了"ultralytics 能直接吃的训练 yaml"。但产出的 yaml 描述的数据集**质量怎么样**——图像和标签是不是成对、yaml 字段是不是合法、训练/验证集之间有没有数据泄露——D3 不管。

**D4 的任务**: 实现一个独立子系统 `data_validation`,在数据被喂给 D5 训练之前**质检一次**——给一份"这数据能不能训"的诊断报告 + 一个 Unix 退出码。

> **设计原则**: 质检子系统**只检测,不修复**。发现问题归 `data_validation`,修复问题归 `data_pipeline` 重跑产线。两件事混在一起会让"我刚跑了一次 validate,数据是不是变了?"都说不清楚。

## 2. 功能需求

### 2.1 用户视角:一行命令

```bash
odp-validate --dataset safety_helmet --task detect
```

→ 加载 `apps/platform/configs/datasets/safety_helmet.yaml`(D3 产出),跑 4 个独立 check,输出:

- **日志**:三段式可读输出(数据集摘要 + 各 check 一览 + 失败详情)
- **JSON 报告**:`runs/data_validation/<run_id>/report.json`(供监控/趋势分析消费)
- **退出码**:`0` / `1` / `2`(供 CI 集成)

### 2.2 四个互相独立的 check

| # | check 名 | 在问什么 | 严重程度策略 |
|---|---|---|---|
| 1 | `yaml_schema` | yaml 文件本身的字段完整性、一致性 | 任何错 → ERROR |
| 2 | `pair_existence` | 每张图都有对应 .txt 标签吗 | **按缺失比例分级** |
| 3 | `label_format` | 每行 .txt 格式合法吗 | 任何错 → ERROR |
| 4 | `split_uniqueness` | train/val/test 之间有图像名重复(数据泄露)吗 | 任何错 → ERROR |

「互相独立」是核心约束——**任何一个 check 失败/抛异常都不得阻断其他 check**。用户一次跑应该看到所有问题,而不是修一个跑一次。

### 2.3 各 check 的检查规则与严重程度细节

#### 2.3.1 `yaml_schema`

检查项(任何一项不通过都算 ERROR):

1. yaml 文件存在且可解析(无 YAMLError)
2. yaml 顶层是 `dict`(不是 list / scalar)
3. 含 `nc` 字段,正整数
4. 含 `names` 字段,合法形式为 `list[str]` 或 `dict[int, str]`,元素非空
5. `len(names) == nc`

**不**检查的项(由其他 check 负责):train/val/test 路径是否存在、类别 ID 是否越界。

**前置 vs 业务错误**:文件不存在、解析失败、顶层不是 dict——这三类**前置错误**早返回;字段一致性问题(缺 nc、缺 names、长度不匹配)**收集进同一个 problems 列表一次报齐**,不要让用户跑 3 遍才知道 3 个问题。

#### 2.3.2 `pair_existence` ⭐ 唯一分级 severity 的业务 check

每张图都应有对应的 `.txt` 标签(空文件合法,代表无目标图像;但不能不存在)。

按缺失比例(`missing_ratio = total_missing / total_images`)分级:

```
missing_ratio == 0.0                     → PASS
missing_ratio <  PAIR_MISSING_WARN_RATIO → INFO
missing_ratio <  PAIR_MISSING_ERROR_RATIO → WARNING
missing_ratio >= PAIR_MISSING_ERROR_RATIO → ERROR
```

阈值常量(放进 `common/constants.py`):

```python
PAIR_MISSING_ERROR_RATIO: float = 0.5    # ≥50% 缺失 → 流程崩了
PAIR_MISSING_WARN_RATIO:  float = 0.05   # ≥5% 缺失 → 影响精度
```

`details` 必须含:`total_images`、`total_missing`、`missing_ratio`、`thresholds`(把使用的两个阈值写进去,便于复盘)、`missing_per_split`(各 split 缺失计数)、`missing_examples`(各 split 前 N 条缺失文件,N 由模块级常量 `DETAILS_PREVIEW_LIMIT = 20` 控制,防 JSON 爆炸)。

#### 2.3.3 `label_format`

支持两种 task(从 `snapshot.task_type` 读,**不要**自己解析 yaml.task):

- **detect**:每行 `cls cx cy w h` — 5 字段,`cls ∈ [0, nc)`,坐标 `∈ [0, 1]`
- **segment**:每行 `cls x1 y1 x2 y2 ... xN yN` — `1 + 2N` 字段,N ≥ 3,`cls ∈ [0, nc)`,坐标 `∈ [0, 1]`

错误分级:**不分级,任何错 → ERROR**(结构性问题,一行错就是一个不能训的样本)。

`details` 必须含:`task_type`、`total_lines`、`total_errors`、`error_kinds`(各错误类型计数,用机器可读的常量字符串作 key)、`errors_preview`(前 N 条具体错误样例,带 label 文件名 + 行号 + 错误类型)。

错误类型常量(在 check 文件内部定义):`field_count_mismatch`、`parse_error`、`class_id_out_of_range`、`coord_out_of_range`、`polygon_too_few_points`。

> 边界:`snapshot.nc` 是 None / 非法时,此 check 输出 INFO 跳过(`yaml_schema` 应已报错,不重复)。

#### 2.3.4 `split_uniqueness`

判重方式:**按图像 stem**(文件名不含扩展名),不按文件内容 hash。

理由:YOLO 按 stem 找 label(`foo.jpg` → `foo.txt`),`train/foo.jpg` 跟 `val/foo.png` 在训练时**事实上**会冲突,跟内容是否相同无关。

严重程度:**不分级,任何重复 → ERROR**(数据泄露是质变型错误,1 张重复就足以让评估失真)。

`details` 必须含:`splits`、`total_duplicates`、`overlaps`(每对 split 的重复列表 + 前 N 条 stem 预览)。

### 2.4 退出码

| 退出码 | 含义 |
|---|---|
| `0` | PASS,或仅 INFO 级问题 |
| `1` | 有 WARNING(可继续,但需 review) |
| `2` | 有 ERROR(不能训) |
| `3` | CLI bug / KeyboardInterrupt(非数据问题) |

退出码从 `overall_severity` 派生,**不存值**。

## 3. 接口契约(必须严格遵守)

### 3.1 数据契约

#### `CheckSeverity`

四级字符串常量类(**不要用** `enum.Enum`,字符串值跟 JSON / 日志 / 命令行参数对接最自然):

```python
class CheckSeverity:
    ERROR   = "ERROR"
    WARNING = "WARNING"
    INFO    = "INFO"
    PASS    = "PASS"

    # 必须提供 rank 方法,便于 ValidationReport 算 overall_severity
    @classmethod
    def rank(cls, level: str) -> int: ...
```

排序:`ERROR > WARNING > INFO > PASS`。

#### `CheckResult`

```python
@dataclass
class CheckResult:
    name:     str                  # check 名 (= @check("name") 的字符串)
    severity: str                  # 见 CheckSeverity
    summary:  str                  # 一句话总结 (给人看)
    details:  Dict[str, Any]       # 结构化详情 (给机器看)

    @property
    def passed(self) -> bool:      # ★ 派生,不存值
        return self.severity in (CheckSeverity.PASS, CheckSeverity.INFO)
```

**字段语义边界**:
- `summary` 一句话,短,给人看,放进日志单行
- `details` 结构化字典,给机器看,可被 JSON 报告 / 监控趋势消费
- `INFO` 也算 `passed`(语义:知道一下,不阻断)
- 严禁同时存 `severity` 和 `passed` 两个字段(数据冗余必然不一致)

#### `CheckContext`

```python
@dataclass
class CheckContext:
    yaml_path: Path             # 阶段 2 先放这一个字段
    snapshot:  DatasetSnapshot  # 阶段 4 加上这个字段
```

**这一层包装是扩展位**:check 函数签名永远是 `(ctx: CheckContext) -> CheckResult`,以后扩展能力都改 `CheckContext` 加字段,**不改 check 函数签名**。

#### `DatasetSnapshot`(必须 `frozen=True`)

```python
@dataclass(frozen=True)
class DatasetSnapshot:
    yaml_path:        Path
    yaml_data:        Dict[str, Any]               # 解析过的 yaml
    yaml_load_error:  Optional[str]                # 解析失败时的错误信息
    data_root:        Path
    nc:               Optional[int]
    class_names:      Tuple[str, ...]              # 必须是 Tuple
    task_type:        str                          # 'detect' / 'segment'
    images_per_split: Dict[str, Tuple[Path, ...]]  # 内层必须是 Tuple
    labels_per_split: Dict[str, Tuple[Path, ...]]  # 内层必须是 Tuple
    stats_per_split:  Dict[str, SplitStats]
    scan_warnings:    Tuple[str, ...]

    @property
    def splits(self)       -> Tuple[str, ...]: ...  # 按 train/val/test 顺序
    @property
    def total_images(self) -> int: ...
```

附属类:

```python
@dataclass(frozen=True)
class SplitStats:
    image_count:     int
    annotated_count: int   # 有非空 label 的图像数
    total_instances: int   # label 文件总行数
```

**三条铁律(任一破坏均不合格)**:

1. **装事实,不做判断** — snapshot 只装数据,不带业务方法(`is_train_size_ok()` 这种是反模式)
2. **frozen=True + 内部容器全是 Tuple** — 防 check 串改;Tuple 而非 List,因为 frozen 不阻止你 `snapshot.images_per_split['train'].append(...)`,Tuple 才能让这个 append 立刻抛 AttributeError
3. **best-effort,不抛异常** — `build_snapshot` 永远返回 snapshot 对象,yaml 解析失败装进 `yaml_load_error` 让 `yaml_schema` check 拿去报告,**调度层永远不该自己 fail**

#### `ValidationReport`

```python
@dataclass
class ValidationReport:
    run_id:           str
    yaml_path:        Path
    snapshot:         DatasetSnapshot
    results:          List[CheckResult]
    duration_seconds: float
    started_at_iso:   str
    run_dir:          Optional[Path] = None

    # ★ 全部派生属性,不存值
    @property
    def overall_severity(self)   -> str: ...
    @property
    def counts_by_severity(self) -> Dict[str, int]: ...
    @property
    def exit_code(self)          -> int: ...
    @property
    def failed_results(self)     -> List[CheckResult]: ...  # 非 PASS 非 INFO
    @property
    def report_path(self)        -> Optional[Path]: ...

    def to_dict(self) -> Dict[str, Any]: ...  # JSON 序列化
```

### 3.2 行为契约

#### `@check(name)` 装饰器

```python
@check("yaml_schema")
def validate_yaml_schema(ctx: CheckContext) -> CheckResult:
    ...
```

**实现约束**:
- 装饰器直接 `return func`,**不要** `@wraps` 包装(包装会让 traceback 多一帧,单元测试也想绕过注册表直接调原函数)
- 同名重复注册**立刻 raise ValueError**,不要覆盖(重复注册要么手误要么 import 链 bug,两种都该立刻可见)

#### `service.run_all_checks(ctx)` 调度

```python
def run_all_checks(ctx: CheckContext) -> List[CheckResult]:
    ...
```

**核心承诺(必须用代码兑现)**: 任何 check 抛异常都被本函数接住,包装成 ERROR 级 CheckResult,**不阻断其他 check**。

这意味着 `_safe_run_one` 内部必须有 `try/except Exception`(且是整个子系统**唯一**一处宽泛 Exception 捕获,因为这里是开闭原则下的扩展点)。其他地方应该用具体的 `except OSError` / `except yaml.YAMLError`。

#### `validate_dataset()` 端到端

```python
def validate_dataset(
    yaml_path:    Path,
    task_type:    Optional[str] = None,
    run_id:       Optional[str] = None,
    run_dir:      Optional[Path] = None,
    write_report: bool = True,
) -> ValidationReport:
    ...
```

负责把 6 件事串起来:解析 run_id → mkdir 运行目录 → 记录起始时间 → `build_snapshot` → 构造 `CheckContext` → `run_all_checks` → 包装 `ValidationReport` → 写 JSON 报告(可选)。

**强制要求**:必须是**函数**,不允许做成 `DatasetValidator` 类。理由:函数内部状态(run_id / duration / snapshot / results)全部进了返回值 `ValidationReport`,没有第二个相关公共方法——做成类是 OOP 仪式感,没增加任何能力。

### 3.3 CLI 契约

```
odp-validate --dataset <name>           # 日常: 走 dataset_yaml_path(name)
odp-validate --yaml /path/to/some.yaml  # 调试: 直接给 yaml 路径
[--task detect|segment]                  # 不传则按 yaml.task → detect 兜底
[--no-report]                            # 不写 JSON 报告
[-v|--verbose]                           # DEBUG 级日志
```

**实现约束**:
- `--dataset` 和 `--yaml` **互斥且必填**(`add_mutually_exclusive_group(required=True)`)
- CLI 文件**禁止写业务**:`main()` 主体应只有「parse 参数 → 调 `validate_dataset` → 调 `render_to_logger` → return `exit_code`」这几行
- 必须有 `try/except KeyboardInterrupt + except Exception` 兜底,任何未预期异常走 `logger.exception` 进日志,**不准让裸 traceback 喷到 stderr**

## 4. 设计模式要求(必须使用)

### 4.1 注册表 + 装饰器 + 自动发现(沿用 D3,核心模式)

跟 D3 `data_pipeline.registry` 同源,但**调度模式相反**:

| 维度 | D3 | D4 |
|---|---|---|
| 调度模式 | **互斥分发**(一次选 1 个 converter) | **聚合执行**(一次跑全部 check) |
| 失败语义 | 立刻抛异常 | 装进 CheckResult 继续跑下一个 |
| 调度层承诺 | "我从注册表挑一个跑" | "我从注册表把每一个都跑一遍,**没人能阻断他人**" |

**自动发现**:用 `pkgutil.iter_modules` 扫 `checks/` 子包,在 import time 触发 `@check` 装饰器副作用。新增 check = 加新文件,**`registry.py` 一行不动**。下划线前缀的文件(`_helpers.py`)被自动跳过,留作内部辅助。

参考:Flask `@app.route` 自动登记路由表;pytest 自动收集 `def test_xxx`;SQLAlchemy 方言。

### 4.2 数据 / 展示分离(D4 新模式)

```
ValidationReport (纯数据)        →  render_to_logger     (阶段 8 实现)
                                 →  render_to_html       (未来 D4.x)
                                 →  render_to_markdown   (未来 D4.x)
```

`report.py` 只装数据 + 派生属性 + `to_dict()`;`render.py` 只把 ValidationReport 翻译成日志输出。两个文件**物理隔离**,以后加 HTML / Markdown renderer = 新增文件,数据层一行不动。

> **YAGNI 注意**: 当前**不要**引入 `ReportSection` 中间抽象层(`kind="kv"|"table"|...`)。等真做第二个 renderer 时再说。

### 4.3 派生属性(`@property`)永不存值

任何能从已存字段算出来的量都用 `@property`,**不要**存进 dataclass 字段。

适用范围:`CheckResult.passed`、`ValidationReport.overall_severity` / `counts_by_severity` / `exit_code` / `failed_results` / `report_path`、`DatasetSnapshot.splits` / `total_images`。

反例:同时存 `severity` 和 `passed`,results.append 后 overall_severity 不更新——数据立刻自相矛盾。

### 4.4 不可变共享视图(`frozen=True` + Tuple)

`DatasetSnapshot` 和 `SplitStats` 必须 `@dataclass(frozen=True)`,且所有容器字段用 `Tuple` 而非 `List`。

`frozen=True` 阻止重新赋值字段,但**不阻止**你修改字段内部的可变对象(`d['train'].append(x)` 合法)。Tuple 才能让这种串改在写入处立刻 AttributeError。

### 4.5 依赖注入 via `CheckContext`

所有 check 函数签名都是 `(ctx: CheckContext) -> CheckResult`。新增能力(snapshot、未来的 previous_snapshot、config_overrides 等)只改 `CheckContext` 加字段,**永远不改 check 函数签名**。

### 4.6 「装饰器不包装」原则

```python
# ✅ 正确
def decorator(func):
    _REGISTRY[name] = CheckEntry(name=name, func=func)
    return func    # 原样返回

# ❌ 错误
def decorator(func):
    _REGISTRY[name] = CheckEntry(name=name, func=func)
    @wraps(func)
    def wrapper(ctx): return func(ctx)
    return wrapper
```

装饰器的功能是"注册副作用",不是"包装行为"。包装一层会让 traceback 多一帧 + 单元测试拿不到原函数。

### 4.7 工具复用 SSoT

**严禁**重新定义 D3 / D2 已有的常量、路径辅助函数。具体:

- ✅ 直接 `from odp_platform.common.constants import Task, IMAGE_EXTENSIONS`
- ❌ 不要立一个 `class TaskType: DETECTION = "detection"`(跟 `Task.DETECT = "detect"` 字符串都不一样,立刻产生两份"任务名"真相)

复用 `common/logging_utils.get_logger`、`common/performance_utils.@time_it`、`common/system_utils.log_device_info`、`common/paths.dataset_yaml_path`。

## 6. 非功能性需求

### 6.1 单文件体量

每个 `.py` 文件控制在 ~220 行内(`snapshot.py` 接近上限可接受,因为 IO 辅助函数多)。每个内部函数 ≤ 50 行,展示层的 `_render_*` 子函数 ≤ 30 行。

### 6.2 日志策略

- `PASS` 级 check 单行**走 DEBUG**(默认不显示,`-v` 才看)。理由:健康数据集每次跑刷 4 行 `[PASS]` 是噪声盖过信号。
- `WARN/ERROR/INFO` 即时走 `logger.warning/error/info`
- 完整三段式摘要由 `render_to_logger` 在跑完后**集中展示**
- 端到端入口 `validate_dataset` 调一次 `log_device_info(logger)`(设备信息是会话级元数据,只该打一次)

### 6.3 量化 (`@time_it`)

装在**框架函数**(`build_snapshot` / `run_all_checks`),**不**装在各个 check。理由:单个 check 通常毫秒级,量化无意义;框架层量化能直接回答"为什么慢"。

`validate_dataset` 本身**不要**装 `@time_it`(已用 `time.perf_counter()` 测了 `duration_seconds` 进 ValidationReport,重复装会让总耗时出现两次)。

### 6.4 跨平台

`_list_images` 必须同时尝试小写和大写扩展名(Windows 大小写不敏感、Linux 大小写敏感),且对结果 `sorted(set(...))` 去重。

### 6.5 测试要求

至少三个测试文件,覆盖三个不同关注点:

| 文件 | 关注点 |
|---|---|
| `test_registry.py` | **框架**: 装饰器、聚合执行、异常兜底承诺、自动 import |
| `test_checks_yaml_schema.py` | **单 check 样例**: 各类错误能产出对的 severity + details |
| `test_e2e.py` | **端到端 smoke**: CLI → JSON 报告 → 退出码 |

> 不追求覆盖率。三个文件够把三种关注点立起来,加新 check 用样例文件复制粘贴就行。

## 8. 设计取舍预告(实现前必读)

下面这些选择是**已经定下的设计决策**,实现时按这个来,不要自己重新发明:

| 抉择点 | 选择 | 不选什么 |
|---|---|---|
| 注册表数据结构 | 模块级 `dict[str, CheckEntry]` | 单例类、配置文件 |
| 装饰器实现 | 直接 `return func` | `@wraps + wrapper` |
| 自动发现机制 | `pkgutil.iter_modules` 扫 `checks/` | 手工列 import、entry_points |
| `CheckSeverity` 表示 | 字符串常量类 | `enum.Enum` |
| `passed` 字段 | `@property` 派生 | dataclass 字段 |
| snapshot 不变性 | `frozen=True` + `Tuple` 容器 | 普通 dataclass + `List` |
| snapshot 失败语义 | best-effort 不抛 | 让 build_snapshot 抛 |
| pair_existence 严重度 | 按比例分级 | 一刀切 ERROR / WARN |
| split_uniqueness 严重度 | 任何重复就 ERROR | 按重复比例分级 |
| label_format 错误类型 | 模块级字符串常量 | 字面量散落各处 |
| 端到端入口 | `validate_dataset()` 函数 | `DatasetValidator` 类 |
| 渲染抽象层 | 直接写 `render_to_logger` | `ReportSection` 中间表示 |

## 9. 提示与建议

### 9.1 推荐开发顺序

1. **先把骨架立起来**:`registry.py` + `service.py` + 一个 `_placeholder` check,跑通整套机制再写真 check
2. **从最简单的 check 开始**:`yaml_schema` 不依赖 snapshot,先写它能让你熟悉 `@check` + `CheckResult` 的用法
3. **再写 snapshot**:这时候你已经知道 check 怎么用 ctx 了,设计 snapshot 字段就有抓手
4. **其他三个 check 平行**:有了 snapshot 后 pair / label / split 三个 check 互相独立,顺序无所谓
5. **最后串端到端**:report + render + service.validate_dataset + CLI 一起写,因为它们互相引用

### 9.2 容易踩的坑(实现时多看一眼)

- **`@check` 装饰器写完后,验证「重复注册立刻 raise」**——用同名注册两个函数,应该 `ValueError`,不是覆盖
- **`run_all_checks` 的异常兜底**——故意让一个 check `raise KeyError("xxx")`,其他 check 应该全跑完,该 check 应该产出 ERROR 级 CheckResult 含 `details["exception_type"]`
- **`snapshot.images_per_split["train"].append(p)`——必须 `AttributeError`(因为是 Tuple),不是无声成功
- **`CheckSeverity.rank("PASS") < rank("INFO") < rank("WARNING") < rank("ERROR")`——单元测试覆盖这个
- **`yaml_schema` 在「字段一致性」段必须收集多个 problems**,而不是发现第一个就 return(但「文件不存在 / 解析失败 / 顶层不是 dict」三类前置错误必须早返回)
- **`pair_existence` 分级 if/elif 链从最严到最松写**(`total_missing == 0 → PASS, else missing_ratio >= ERROR_RATIO → ERROR, ...`),避开反着写要用 `not` + 复合条件的可读性问题
- **CLI 文件不准超 100 行,不准写业务**

### 9.3 时间预算

| 阶段 | 预算 |
|---|---|
| paths/constants 增量 + 框架 (registry/service) | 1.5h |
| yaml_schema check | 0.5h |
| DatasetSnapshot | 1.5h |
| pair_existence + label_format + split_uniqueness | 1.5h |
| ValidationReport + render | 1h |
